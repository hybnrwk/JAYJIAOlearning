

## 关系数据库综合详解

关系数据库（Relational Database）是当今世界范围内应用最广泛的数据库类型。从银行的交易系统、企业的ERP、SaaS应用到网站的用户管理，其背后几乎都是关系数据库管理系统（RDBMS）在支撑。

本次讲解将围绕你提出的几个核心概念展开，并扩展相关的重要知识点，以构建一个完整的知识体系。

### 一、 关系模型概述 (The Relational Model)

关系模型是整个关系数据库的理论基石。它由埃德加·弗兰克·科德（Edgar F. Codd）于1970年在一篇名为《大型共享数据库的关系模型》（"A Relational Model of Data for Large Shared Data Banks"）的论文中首次提出。Codd的贡献是革命性的，他将严格的数学（集合论和一阶谓词逻辑）引入了数据管理领域。

关系模型的核心思想是**用二维表（Table）的形式来组织和表示数据及其联系**。

一个完整的关系模型由三个部分组成：

1. **数据结构（Data Structure）：** 即数据是如何组织的。在关系模型中，数据被组织成“关系”（Relation），也就是我们常说的“表”（Table）。
    
2. **数据操作（Data Operations）：** 即可以对数据执行哪些操作。关系模型定义了一套标准的操作语言，理论上是“关系代数”（Relational Algebra）和“关系演算”（Relational Calculus），实践中就是SQL（Structured Query Language）。
    
3. **数据完整性（Data Integrity）：** 即如何确保数据的正确性和一致性。模型定义了三类完整性约束：实体完整性、参照完整性和域完整性。
    

下面我们详细分解这三个部分。

#### 1. 核心数据结构

- **域（Domain）：** 域是**一组具有相同数据类型的值的集合**。例如，`int`（整数集合）、`char(20)`（长度为20的字符串集合）、`Date`（日期集合）。域定义了某个属性（列）的**合法取值范围**。
    
- **属性（Attribute）：** 即我们常说的“列”（Column）。每个属性都有一个名字和一个域。例如，属性“学生姓名”的域可能是`varchar(50)`。
    
- **元组（Tuple）：** 即我们常说的“行”（Row）。一个元组是属性的集合，其中每个属性都从其对应的域中取一个值。例如，一行学生数据 `(1001, '张三', '计算机科学')` 就是一个元组。
    
- **关系（Relation）：** 即我们常说的“表”（Table）。关系是**元组的集合**。这个定义非常重要：
    
    - **集合性**：一个关系（表）中**不允许出现完全相同的元组（行）**。
        
    - **顺序无关**：元组（行）之间没有固定的顺序，属性（列）之间也没有固定的顺序。
        
- **候选键（Candidate Key）：** 关系中，能够**唯一标识一个元组**的一个属性或属性组合。例如，“学生表”中的“学号”可以唯一标识一个学生。
    
- **主键（Primary Key）：** 当一个关系有多个候选键时，我们**选择其中一个**作为主键。主键是数据库设计者的选择。
    
- **外键（Foreign Key）：** 一个关系（表A）中的一个属性（或属性组），它**引用**了另一个关系（表B）的**主键**。外键是建立两个关系（表）之间联系的桥梁。
    

#### 2. 核心数据操作

关系模型定义了一套丰富的操作，用于从一个或多个关系中查询数据，其理论基础是关系代数。

- **选择 (Select, $\sigma$)：** 从关系中**筛选**出满足特定条件的**元组（行）**。
    
    - _SQL示例：_ `SELECT * FROM Student WHERE Major = 'CS';`
        
- **投影 (Project, $\pi$)：** 从关系中**选择**出特定的**属性（列）**。
    
    - _SQL示例：_ `SELECT StudentID, Name FROM Student;`
        
- **并 (Union, $\cup$)：** 合并两个具有相同结构的关系的元组，并**去除重复元组**。
    
- **差 (Difference, $-$)：** 从第一个关系中移除所有也存在于第二个关系中的元组。
    
- **笛卡尔积 (Cartesian Product, $\times$)：** 将两个关系的元组进行所有可能的组合。这是一个非常基础但开销巨大的操作。
    
- **连接 (Join, $\bowtie$)：** 连接是关系模型中最重要、最强大的操作。它通常是先执行一个笛卡尔积，然后再根据连接条件执行一个“选择”操作。
    
    - _自然连接（Natural Join）：_ 在两个关系中寻找同名且值相等的属性进行连接。
        
    - _Theta连接（$\theta$-Join）：_ 使用任意比较（如 $>, <, =$）进行连接。
        
    - _等值连接（Equi-Join）：_ 仅使用 $=$ 进行连接。
        

SQL语言（如`SELECT`, `INSERT`, `UPDATE`, `DELETE`）就是关系代数和关系演算的高度工程化和商业化的实现。

#### 3. 核心数据完整性

- **实体完整性 (Entity Integrity)：** 规定了**主键（Primary Key）的值不能为空（NOT NULL）且必须唯一**。这确保了表中的每一行数据都是可唯一标识的。
    
- **参照完整性 (Referential Integrity)：** 规定了**外键（Foreign Key）**的值：
    
    1. 要么必须等于它所引用的那个表（父表）中某个元组的主键值。
        
    2. 要么必须为空（NULL），表示“未知”或“不适用”。
        
        这确保了关系之间的引用是有效的，不会出现“悬挂引用”（Dangling Reference）。
        
- **域完整性 (Domain Integrity)：** 规定了每个属性（列）的值都必须来自其定义的域。这是最基本（也最容易被忽略）的完整性，通过 `DATA TYPE`（如 `INT`, `DATE`）和 `CHECK` 约束来实现。

### 二、 关系和关系变量 (Relations and Relation Variables)

这是一个在学术上非常重要，但在实践中经常被混淆的概念。理解它们的区别，有助于你理解“视图”和“事务”的本质。

- **关系变量 (Relation Variable / Relvar)：**
    
    - 这是一个**“模式”或“定义”**。它是一个“容器”的声明。
        
    - 当你执行 `CREATE TABLE Student (...)` 语句时，你实际上是在**定义一个关系变量**。
        
    - 这个定义包含了关系的名字（如 `Student`）、它的属性（列名，如 `StudentID`, `Name`）、每个属性的域（数据类型，如 `INT`, `VARCHAR`）以及约束（如 `PRIMARY KEY`, `FOREIGN KEY`）。
        
    - 关系变量是**持久的**，它存储在数据字典中，除非你 `DROP TABLE`，否则它一直存在。
        
- **关系 (Relation)：**
    
    - 这是一个**“实例”或“值”**。它是关系变量在**某一特定时刻所包含的数据**。
        
    - 它是一个元组的集合（一个具体的值）。
        
    - 当你执行 `SELECT * FROM Student;` 时，你看到的结果集就是 `Student` 这个关系变量在**当前时刻**的“关系”（值）。
        
    - 这个“关系”（值）是**瞬时的**。下一秒，如果有人 `INSERT` 了一条新数据，`Student` 关系变量就会持有一个_不同_的“关系”（值）。
        

**总结：** 关系变量（Relvar）是“表结构”，关系（Relation）是“表在某个时间点的数据”。

> 类比：
> 
> 在编程语言中，一个变量（Variable），比如 int x;，对应于一个关系变量（Relvar）。
> 
> 而这个变量在某个时刻的值（Value），比如 x = 10;，对应于一个关系（Relation）。

### 三、 关系的含义 (Meaning of a Relation)

在关系模型中，一个关系（表）不仅仅是数据的随意堆砌，它具有严格的**逻辑含义**。

这要追溯到关系模型的数学基础——**一阶谓词逻辑**。

- 一个**关系变量（表结构）**可以被看作是一个**谓词（Predicate）**。
    
- 一个**元组（行）**可以被看作是一个**命题（Proposition）**。
    

**举例说明：**

假设我们定义了关系变量 `Student(StudentID, Name, Major)`。

- **谓词**就是：“...是一个学生，其学号是...，姓名是...，专业是...”
    
- 这个谓词有三个占位符（`StudentID`, `Name`, `Major`）。
    

当我们将一个元组 `(1001, '张三', '计算机科学')` **插入**到 `Student` 表中时，我们的**真正含义**是：

> 我们向数据库**断言（Assert）**一个命题：“存在一个学生，其学号是1001，姓名是张三，专业是计算机科学” **这个命题为真（TRUE）**。

当你**查询** `SELECT * FROM Student WHERE StudentID = 1001;` 时，你是在问数据库：

> “请告诉我关于学号为1001的学生的所有为真的命题。”

数据库返回了 `(1001, '张三', '计算机科学')`，它是在说：“是的，这个命题为真。”

如果数据库返回空集，它是在说：“根据我所知，没有关于学号1001的命题为真。”（这被称为**封闭世界假说 CWA** - Closed World Assumption，即数据库中没有明确说明为真的，就被认为是假的）。

**为什么这个含义很重要？**

1. **无重复行：** 在谓词逻辑中，断言同一个命题两次，和断言一次是完全等价的。因此，关系（表）中不允许出现重复的元组（行）。
    
2. **无顺序：** 命题的集合是无序的。因此，表中的行是无序的，列也是无序的（尽管SQL实现中为了方便指定了列序）。
    
3. **NULL的含义：** Codd最初的模型中没有NULL。`NULL` 的引入带来了巨大的理论复杂性，因为它破坏了经典二值逻辑（True/False），引入了三值逻辑（True/False/Unknown）。`NULL` 的标准含义是“**未知**”（Unknown）或“**不适用**”（Not Applicable）。

### 四、 基本关系变量和视图 (Base Relation Variables and Views)

基于我们对“关系变量”的理解，我们可以将其分为两类：

#### 1. 基本关系变量 (Base Relation Variable)

- 这就是我们常说的**基表（Base Table）**。
    
- 当你执行 `CREATE TABLE ...` 时，你创建的就是一个基本关系变量。
    
- 它的“关系”（值，即数据）是**物理存储**在数据库中的（可能在磁盘上）。
    
- 对它的`INSERT`, `UPDATE`, `DELETE` 操作是直接修改存储的数据。
    

#### 2. 视图 (View)

- 视图是一个**虚拟关系变量（Virtual Relation Variable）**。
    
- 当你执行 `CREATE VIEW ... AS SELECT ...` 时，你创建的就是一个视图。
    
- 视图**不独立存储数据**。它是一个“窗口”，通过它可以查看或（有限地）操作基表中的数据。
    
- 视图的“关系”（值）是在**查询它的时候动态计算**出来的。
    

**视图的工作机制：查询修改（Query Modification）**

当你对一个视图执行查询时，RDBMS并不会先去计算视图的完整结果集。相反，它会执行一个称为“查询修改”的过程：

1. **用户查询：** `SELECT Name FROM CS_Student_View WHERE StudentID > 1000;`
    
2. **视图定义：** `CREATE VIEW CS_Student_View AS (SELECT * FROM Student WHERE Major = 'CS');`
    
3. RDBMS重写查询： 系统会将视图定义（一个SELECT）与用户查询（另一个SELECT）合并成一个直接作用于基表的新查询：
    
    SELECT Name FROM (SELECT * FROM Student WHERE Major = 'CS') AS CS_Student_View WHERE StudentID > 1000;
    
4. 优化器介入： 优化器会将上述查询进一步简化为：
    
    SELECT Name FROM Student WHERE Major = 'CS' AND StudentID > 1000;
    

**视图的优点：**

1. **逻辑数据独立性：** 这是视图**最重要**的作用。如果基表的结构发生变化（比如增加了一个新列），只要这个变化不影响视图的定义，那么依赖该视图的应用程序就**不需要修改**。它在应用程序和物理表之间提供了一个抽象层。
    
2. **安全性（Security）：** 可以只向特定用户暴露表的某些列或某些行，隐藏敏感数据。
    
3. **简化复杂查询：** 可以将一个非常复杂的、多表连接的查询封装成一个视图，用户只需简单地 `SELECT *` 这个视图即可。
    

**可更新的视图（Updatable Views）：**

这是一个复杂的问题。

- 如果一个视图是“简单”的（通常指一对一映射到基表的行，不含`GROUP BY`, `DISTINCT`, 聚合函数等），那么对视图的`UPDATE`, `INSERT`, `DELETE` 才是可能的。
    
- 如果视图涉及多表`JOIN`或聚合，那么更新操作是“**歧义的**”（Ambiguous）。例如，更新一个`AVG(Salary)`的视图？数据库不知道该去更新基表中的哪一行数据。因此，大多数RDBMS会禁止此类更新。

### 五、 数据字典 (Data Dictionary) / 系统目录 (System Catalog)

如果说关系数据库存储的是“数据”（Data），那么数据字典存储的就是“**关于数据的数据**”（Metadata）。

**数据字典是一个数据库的核心和灵魂。** 它本身通常就是**一组关系表**（所以RDBMS可以用自己的查询语言去查询它）。它由RDBMS自动维护，普通用户通常只有只读权限。

**数据字典中存储了什么？**

- **关系（表）信息：**
    
    - 表名、表的所有者、创建时间。
        
    - 表的物理存储位置。
        
    - 表中的行数、占用的磁盘空间等统计信息（**对优化器至关重要**）。
        
- **属性（列）信息：**
    
    - 列名、所属的表。
        
    - 数据类型（Domain）。
        
    - 是否为NULL。
        
    - 默认值。
        
- **约束（Constraint）信息：**
    
    - 主键约束、外键约束（以及它引用的表和列）、Check约束。
        
- **视图（View）信息：**
    
    - 视图名。
        
    - 定义视图的`SELECT`语句（查询文本）。
        
- **索引（Index）信息：**
    
    - 索引名、所属的表和列。
        
    - 索引类型（如 B-Tree, Hash）。
        
    - 索引的统计信息（如唯一值的数量）。
        
- **用户和权限（Security）信息：**
    
    - 用户名、加密的密码。
        
    - 用户对哪些表/视图/列拥有`SELECT`, `INSERT`, `UPDATE`等权限。
        
- **事务和锁（Transaction）信息：**
    
    - （动态部分）当前活动的事务、它们持有的锁。
        

**数据字典的作用：**

- **SQL解析与验证：** 当你执行一条SQL时，RDBMS会查询数据字典，检查表名、列名是否存在？数据类型是否匹配？你是否有权限执行此操作？
    
- **查询优化：** （详见下一节）优化器会查询数据字典中的统计信息（表有多大？索引的选择性如何？）来决定一个查询的最佳执行计划。
    
- **完整性约束：WHENEVER**：当你 `INSERT` 或 `UPDATE` 数据时，RDBMS会查询数据字典中的约束（主键、外键、Check），并强制执行它们。

### 六、 优化 (Optimization)

这是关系数据库（RDBMS）最复杂、最核心的技术之一，也是RDBMS（如 Oracle, PostgreSQL, SQL Server）区别于简单数据存储（如

SQLite, MySQL/MyISAM）的“护城河”。

**核心问题：** SQL是**声明性（Declarative）**语言，你只告诉数据库“**你想要什么**”（What），而没有告诉它“**如何获取**”（How）。

例如：`SELECT * FROM A JOIN B ON A.id = B.id WHERE A.value > 100;`

RDBMS的**查询优化器（Query Optimizer）**的工作就是找到最高效的“How”（即**执行计划, Execution Plan**）。

这个过程通常分为两个阶段：

#### 1. 逻辑优化（Logical Optimization）

也称为基于规则的优化（Rule-Based Optimization, RBO）。

在这个阶段，优化器使用一组等价变换规则（基于关系代数）来重写（Rewrite）原始的SQL查询，使其在逻辑上等价，但可能更易于执行。

- **核心思想：** 尽早地减少中间结果集的大小。
    
- **常见规则：**
    
    - **谓词下推（Predicate Pushdown）：** 尽早地执行`SELECT`（$\sigma$）操作。
        
        - _原始：_ `(A JOIN B ON A.id = B.id) WHERE A.value > 100`
            
        - _优化：_ `( (A WHERE A.value > 100) JOIN B ON A.id = B.id )`
            
        - _好处：_ 在`JOIN`之前就过滤掉A表中的大量数据，极大地减少`JOIN`的计算量。
            
    - **投影下推（Projection Pushdown）：** 尽早地执行`PROJECT`（$\pi$）操作。
        
        - _原始：_ `SELECT Name FROM (A JOIN B ON A.id = B.id)`
            
        - _优化：_ `SELECT Name FROM ( (SELECT id, Name FROM A) JOIN (SELECT id FROM B) ON A.id = B.id )`
            
        - _好处：_ 在`JOIN`之前丢弃不需要的列，减少I/O和内存使用。
            
    - **连接顺序交换：** `(A JOIN B) JOIN C` $\Leftrightarrow$ `A JOIN (B JOIN C)`（满足结合律和交换律）。
        
    - **子查询展开（Subquery Unflattening）：** 尽可能将子查询（尤其是相关的子查询）转换成`JOIN`操作，因为`JOIN`的算法实现通常比嵌套循环执行子查询要高效得多。
        

#### 2. 物理优化（Physical Optimization）

也称为基于成本的优化（Cost-Based Optimization, CBO）。这是现代优化器的核心。

在逻辑优化之后，我们有了一个逻辑执行树。现在，我们需要为树中的每一个操作选择一个具体的物理实现算法。

- **核心问题：** 对于同一个逻辑操作，有多种物理实现。
    
    - _逻辑操作：_ 获取 `A WHERE A.value > 100` 的数据。
        
    - _物理选择1：_ **全表扫描（Full Table Scan）**。
        
    - _物理选择2：_ 如果 `A.value` 上有索引，使用**索引扫描（Index Scan）**。
        
    - _逻辑操作：_ `A JOIN B`。
        
    - _物理选择1：_ **嵌套循环连接（Nested Loop Join, NLJ）**。
        
    - _物理选择2：_ **排序合并连接（Sort-Merge Join, SMJ）**。
        
    - _物理选择3：_ **哈希连接（Hash Join, HJ）**。
        
- **CBO的工作：**
    
    1. **生成可能的执行计划：** 优化器会探索不同的物理算法组合和`JOIN`顺序。
        
    2. **估算每个计划的成本（Cost）：** 这是最关键的一步。优化器会查询**数据字典**中的**统计信息**（表有多少行？`A.value` 的数据分布直方图是怎样的？`A.value > 100` 能过滤掉多少数据？）。
        
    3. **成本模型：** 成本通常被估算为 `CPU 成本 + I/O 成本`。
        
    4. **选择成本最低的计划：** 优化器会选择那个估算成本最低的执行计划来执行。
        

**关键的物理实现：索引（Indexes）**

索引是提高查询性能**最重要**的物理结构。它是一种**以空间换时间**的策略。

- **B+ 树索引（B+ Tree Index）：**
    
    - 这是最常见的索引类型，几乎所有RDBMS都默认使用它。
        
    - 它是一种自平衡的树形结构，特别适合处理**范围查询**（如 `WHERE id > 100`）和**等值查询**（`WHERE id = 10`）。
        
    - 它的叶子节点包含所有索引键的有序列表，并且通常是双向链表，便于范围扫描。
        
    - 查询、插入、删除的时间复杂度都是 $O(\log N)$。
        

**关键的物理实现：连接算法（Join Algorithms）**

1. **嵌套循环连接 (Nested Loop Join)：**
    
    - _算法：_ `for rowA in TableA: for rowB in TableB: if rowA.id == rowB.id: output(rowA, rowB)`
        
    - _成本：_ $O(N \times M)$。非常高。
        
    - _优化（Index NLJ）：_ 如果B表的`id`列上有索引，则内层循环变成 $O(\log M)$ 的索引查找。总成本降为 $O(N \log M)$。
        
    - _适用：_ 当一个表（驱动表，外层）非常小，且另一个表（内层）在连接键上有索引时，非常高效。
        
2. **排序合并连接 (Sort-Merge Join)：**
    
    - _算法：_
        
        1. 按连接键对A表排序。
            
        2. 按连接键对B表排序。
            
        3. 像拉拉链一样，同时扫描两个已排序的表，合并匹配的行。
            
    - _成本：_ $O(N \log N + M \log M)$（排序成本）+ $O(N+M)$（合并成本）。
        
    - _适用：_ 当两个表都很大，且都没有索引，或者数据本身就是有序的时。
        
3. **哈希连接 (Hash Join)：**
    
    - _算法：_
        
        1. 选择较小的表（如A），对其连接键计算哈希值，在内存中建立一个**哈希表**。
            
        2. 扫描较大的表B，对B的每一行，计算连接键的哈希值，去哈希表中“探测（Probe）”匹配的行。
            
    - _成本：_ $O(N + M)$。
        
    - _适用：_ 当至少有一个表可以（或大部分可以）放入内存时，这是**最快**的等值连接算法。

### 七、 事务 (Transactions)

如果说优化是为了“快”，那么事务就是为了“**准**”和“**稳**”。

事务（Transaction）是数据库并发控制和故障恢复的**基本单位**。它是一个**逻辑工作单元（Logical Unit of Work）**，包含一个或多个SQL操作。

**核心思想：** 事务必须是“**全或无**”（All-or-Nothing）。

- _银行转账示例：_
    
    1. `UPDATE Account SET Balance = Balance - 100 WHERE Name = 'A';`
        
    2. `UPDATE Account SET Balance = Balance + 100 WHERE Name = 'B';`
        
- 这两个操作必须**要么全部成功**，**要么全部失败**。如果执行了1之后系统崩溃，而2未执行，那么数据库就处于不一致状态（钱凭空消失了）。
    

#### 1. ACID 属性

事务必须具备四个特性，合称 ACID：

- **A - 原子性 (Atomicity)：**
    
    - _含义：_ 事务中的所有操作是一个不可分割的原子单元，要么全部执行，要么全部不执行（回滚，Rollback）。
        
    - _实现机制：_ **日志（Logging）** 和 **恢复（Recovery）** 系统。
        
- **C - 一致性 (Consistency)：**
    
    - _含义：_ 事务必须使数据库从一个**一致性状态**转变到另一个**一致性状态**。
        
    - _注意：_ 一致性是**最容易被误解**的。数据库_本身_（通过约束）只能保证_数据层面_的一致性（如主键不重、外键有效）。
        
    - _业务层面_的一致性（如转账前后总金额不变）是**应用程序的责任**。
        
    - 原子性、隔离性、持久性是_为了保障_一致性而存在的。
        
- **I - 隔离性 (Isolation)：**
    
    - _含义：_ 多个事务**并发**执行时，一个事务的执行不应被其他事务干扰。每个事务都感觉好像“只有它自己在访问数据库”。
        
    - _实现机制：_ **并发控制（Concurrency Control）**，主要是**锁（Locking）**或**多版本并发控制（MVCC）**。
        
- **D - 持久性 (Durability)：**
    
    - _含义：_ 一旦事务被**提交（Commit）**，它对数据库的修改就是**永久性**的。即使系统随后崩溃，已提交的数据也不应丢失。
        
    - _实现机制：_ **日志（Logging）** 和 **恢复（Recovery）** 系统。
        

#### 2. 并发控制（Concurrency Control）

隔离性（Isolation）是ACID中最难实现的。如果完全隔离（串行化），性能会极差。因此，RDBMS定义了不同的**隔离级别（Isolation Levels）**作为性能和一致性之间的妥协。

**并发带来的问题：**

1. **脏读 (Dirty Read)：** 事务A读取了事务B**已修改但尚未提交**的数据。如果B回滚，A读到的就是“脏”数据。
    
2. **不可重复读 (Non-Repeatable Read)：** 事务A在_同一个事务_中两次读取同一数据，但中间事务B修改并提交了该数据，导致A两次读取结果不同。
    
3. **幻读 (Phantom Read)：** 事务A在_同一个事务_中两次执行_范围查询_（如 `SELECT COUNT(*)`），但中间事务B插入或删除了符合该范围的数据并提交，导致A两次查询的结果集（行数）不同（像出现了幻影）。
    

**四种标准隔离级别：**

|**隔离级别**|**脏读**|**不可重复读**|**幻读**|
|---|---|---|---|
|**读未提交** (Read Uncommitted)|可能|可能|可能|
|**读已提交** (Read Committed)|不允许|可能|可能|
|**可重复读** (Repeatable Read)|不允许|不允许|可能|
|**可串行化** (Serializable)|不允许|不允许|不允许|

- `Read Committed` 是大多数数据库（如 Oracle, SQL Server, PostgreSQL）的默认级别。
    
- `Repeatable Read` 是 MySQL (InnoDB) 的默认级别。
    

**并发控制的实现机制：**

- **基于锁的并发控制 (Locking)：**
    
    - **共享锁 (S-Lock / 读锁)：** 其他事务可以读，但不能写。
        
    - **排他锁 (X-Lock / 写锁)：** 其他事务既不能读，也不能写。
        
    - **两阶段锁协议 (2PL)：**
        
        1. _增长阶段：_ 事务只能获取锁，不能释放锁。
            
        2. _缩减阶段：_ 事务只能释放锁，不能获取锁。
            
    - 2PL是保证**可串行化**的经典方法，但它也带来了**死锁（Deadlock）**的问题（A等B的锁，B等A的锁）。RDBMS必须有死锁检测和回滚机制。
        
- **多版本并发控制 (MVCC - Multi-Version Concurrency Control)：**
    
    - 这是更现代、性能更好的机制（被 Oracle, PostgreSQL, MySQL/InnoDB 使用）。
        
    - **核心思想：** **写操作不会覆盖旧数据，而是创建数据的一个新版本**。
        
    - 每个事务在启动时会获得一个“快照”（Snapshot）时间戳。
        
    - **读操作：** 事务只读取在它启动之前就已经提交的那个数据版本。
        
    - **写操作：** 创建一个新版本，并标记上自己的事务ID。
        
    - **好处：** **“读”永远不会阻塞“写”，“写”也永远不会阻塞“读”**。极大地提高了并发性能。
        
    - 在MVCC下，`Read Committed` 和 `Repeatable Read` 的实现变得非常高效（只需读取不同时间点的快照）。
        

#### 3. 故障恢复（Recovery）

恢复系统用于保证**原子性**和**持久性**。

- **核心组件：日志（The Log）**
    
    - 日志是一个**仅追加（Append-Only）**的磁盘文件。
        
    - **预写日志协议 (WAL - Write-Ahead Logging)：**
        
        1. 在**数据（Data）**被写入磁盘**之前**，
            
        2. 描述该数据修改的**日志记录（Log Record）**必须**首先**被写入磁盘。
            
    - _为什么？_ 日志是顺序I/O，数据是随机I/O。写日志远快于写数据。只要日志安全落地，即使数据还没写（还在内存Buffer中）系统就崩溃了，我们也能通过日志“重做（REDO）”这个修改。
        
- **恢复操作：**
    
    - **REDO (重做)：** 用于保证**持久性**。当系统崩溃重启时，扫描日志，将所有**已提交**事务的修改（但在崩溃时可能还在内存中）**重新应用**到数据库中。
        
    - **UNDO (撤销)：** 用于保证**原子性**。当系统崩溃重启时，扫描日志，将所有**未提交**（Aborted 或 Active）事务的修改**回滚**（倒着做一遍）。

### 总结

关系数据库是一个建立在严格数学（集合论与谓词逻辑）基础之上的，高度工程化的、复杂而可靠的系统。

- **关系模型**（表、键、完整性）提供了简单、统一的数据视图。
    
- **关系变量**（模式）与**关系**（实例）的分离，是理解视图和事务的基础。
    
- **数据字典**是RDBMS的“大脑”，存储所有元数据。
    
- **优化器**（逻辑优化、物理优化、索引、连接算法）是RDBMS的“心脏”，负责将声明性的SQL转变为高效的执行计划。
    
- **事务管理器**（ACID、并发控制、恢复）是RDBMS的“灵魂”，负责在并发和故障环境中保证数据的绝对正确和可靠。
    

这几大模块协同工作，才造就了关系数据库在过去50年里不可动摇的地位。
