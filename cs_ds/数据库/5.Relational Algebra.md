## 关系代数综合详解

关系代数（Relational Algebra）是在关系模型中，由E.F. Codd（埃德加·弗兰克·科德）定义的，用于**操作关系（表）的形式化语言**。

它是一种**过程性（Procedural）语言。这意味着，当你使用关系代数时，你不仅是在描述“你想要什么”（What），更是在明确地指定“如何一步步得到它”**（How）。这与SQL（一种声明性语言）形成了鲜明对比。


---

### 一、 关系代数的核心：关系封闭性 (Relational Closure)

这是理解关系代数**威力**的**关键所在**。

**关系封闭性（Relational Closure）** 指的是：**所有关系代数操作的输入是一个或多个关系（Relation），其输出也必定是一个关系（Relation）。**

这个特性看起来很简单，但它却是构建复杂查询的基础。

- **类比（算术）：** 算术运算（+、-、*、/）具有“数值封闭性”。`2 + 3` 的结果是 `5`，它仍然是一个数。正因为如此，你可以写出 `(2 + 3) * 4` 这样的嵌套表达式，因为 `(2 + 3)` 的结果 `5` 可以立即作为 `*` 运算的输入。
    
- **类比（关系代数）：** 关系代数也是如此。
    
    1. 一个 `WHERE` 操作（选择）作用于一个表（关系），输出是一个新的表（关系）。
        
    2. 一个 `JOIN` 操作（连接）作用于两个表（关系），输出也是一个新表（关系）。
        

封闭性的含义：

由于任何操作的输出（一个关系）都可以立即成为下一个操作的输入，我们可以将操作符像“乐高积木”一样任意嵌套和组合，构建出极其复杂的查询表达式。

例如，你提供的语法本身就是“关系封闭性”的绝佳展示：

(((P WHERE COLOR = 'Red') JOIN SP) {S#} JOIN S) {SNAME}

这个表达式的执行顺序（从内到外）完美地体现了封闭性：

1. `R1 := (P WHERE COLOR = 'Red')`：执行选择，其结果 `R1` 是一个**新的、临时的关系**。
    
2. `R2 := (R1 JOIN SP)`：`R1` 作为输入，与 `SP` 执行连接，其结果 `R2` 是**又一个新的关系**。
    
3. `R3 := (R2 {S#})`：`R2` 作为输入，执行投影，其结果 `R3` 是**又一个新的关系**。
    
4. `R4 := (R3 JOIN S)`：`R3` 作为输入，与 `S` 执行连接...
    
5. `Result := (R4 {SNAME})`：...最后投影，得到**最终的结果关系**。
    

没有关系封闭性，关系代数将只是一组互不相关的一次性操作，而无法构建复杂的查询。

---

### 二、 关系代数的作用 (The Purpose of Relational Algebra)

关系代数绝不仅仅是“数据库理论课上的数学符号”，它在现代数据库系统中扮演着三个至关重要的角色。

#### 1. 关系模型的理论基础

关系代数是关系模型“数据操作”部分的数学化、形式化定义。它为“我们能对表做什么”提供了严格的、无歧义的定义。

#### 2. “关系完备性”的度量基准

Codd使用关系代数来定义一个**度量标准**，称为**关系完备性 (Relational Completeness)**。

- **定义：** 一种查询语言，如果它能表达（或等价于）关系代数中_所有_的基本操作（我们稍后会讲到的5个或8个操作符），那么这种语言就是“关系完备的”。
    
- **意义：** SQL、QUEL、QBE等真实世界的查询语言，都必须被证明是“关系完备的”。这确保了它们具有足够的表达能力来执行任何基本的数据查询和组合。
    
- **注意：** 现代SQL的能力**远超**基本的关系代数（例如，它能进行聚合、排序、递归等），但“关系完备”是它作为查询语言的“准入门槛”。
    

#### 3. 查询优化 (Query Optimization) 的核心

这是关系代数在现代RDBMS中**最重要、最实际**的作用。

如前所述，SQL是**声明性**的（你只说“要什么”），而关系代数是**过程性**的（你指定“怎么做”）。

当你向数据库提交一条SQL时，RDBMS的内部执行流程如下：

1. **解析 (Parsing)：** SQL被解析成一个“解析树”。
    
2. **转换 (Translation)：** 解析树被转换成一个**逻辑查询计划**，这个计划通常就是一棵**关系代数表达式树**。
    
3. **优化 (Optimization)：** 这是关键！**查询优化器（Query Optimizer）会介入。优化器知道关系代数的一系列等价变换规则**（就像数学中的 $a(b+c) = ab+ac$）。
    
4. **物理执行：** 优化器选择一个成本最低的物理计划并执行。
    

**优化的核心：代数变换**

假设我们要查询：“选修了'CS101'课程的'CS'专业学生的姓名”。

- **SQL:**
    
    SQL
    
    ```
    SELECT S.Name
    FROM Student S
    JOIN Enroll E ON S.StudentID = E.StudentID
    WHERE S.Major = 'CS' AND E.CourseID = 'CS101';
    ```
    
- **RDBMS可能生成的“天真”的（Naive）关系代数计划：**
    
    1. `R1 := (Student TIMES Enroll)` (执行笛卡尔积)
        
    2. `R2 := (R1 WHERE Student.StudentID = Enroll.StudentID AND Student.Major = 'CS' AND Enroll.CourseID = 'CS101')` (在一个巨大的表上执行所有选择)
        
    3. `Result := (R2 {Student.Name})` (最后投影)
        
    
    - 这个计划是**灾难性**的，因为它第一步就创建了一个 $N \times M$ 行的巨大中间表。
        
- **RDBMS通过等价变换，生成的“优化”的（Optimized）关系代数计划：**
    
    - **规则：** “尽早执行选择（WHERE）和投影（{...}）” （即“谓词下推”和“投影下推”）。
        
    - **变换过程：**
        
        1. 原始计划：`(((Student TIMES Enroll) WHERE Student.StudentID = Enroll.StudentID AND Student.Major = 'CS' AND Enroll.CourseID = 'CS101') {Student.Name})`
            
        2. 变换（连接替换）： TIMES 和 WHERE 合并为 JOIN ON
            
            ((Student JOIN Enroll ON Student.StudentID = Enroll.StudentID) WHERE Student.Major = 'CS' AND Enroll.CourseID = 'CS101') {Student.Name}
            
        3. 变换（谓词下推）： 将 WHERE 操作符“推”到 JOIN 的下面，使其尽可能靠近数据源。
            
            (((Student WHERE Major = 'CS') AS S JOIN (Enroll WHERE CourseID = 'CS101') AS E ON S.StudentID = E.StudentID)) {S.Name}
            
- **优化后的计划语义：**
    
    1. `R1 := (Student WHERE Major = 'CS')`：**先**从`Student`表中筛出'CS'专业的学生（结果集很小）。
        
    2. `R2 := (Enroll WHERE CourseID = 'CS101')`：**先**从`Enroll`表中筛出'CS101'课程的注册（结果集很小）。
        
    3. `R3 := (R1 AS S JOIN R2 AS E ON S.StudentID = E.StudentID)`：**只**连接这两个很小的结果集。
        
    4. `Result := (R3 {S.Name})`：最后投影。
        

这两个代数表达式是**逻辑等价**的，但它们的执行成本（I/O、CPU）相差成千上万倍。

**因此，关系代数是查询优化器进行逻辑优化的“草稿纸”和“语言”。**

### 三、 基本关系代数操作符 (Basic Operators)

关系代数有一组“基本”操作符。理论上，我们可以将其缩减为一个**最小完备集（5个）**，其他操作都可以由这5个派生出来：

1. **选择 (Selection):** `(R WHERE cond)`
    
2. **投影 (Projection):** `(R {attrs})`
    
3. **并 (Union):** `(R UNION S)`
    
4. **差 (Difference):** `(R MINUS S)`
    
5. **笛卡尔积 (Product):** `(R TIMES S)`
    

我们将详细讲解所有这些以及重要的派生操作符。

示例数据：

为了方便说明，我们使用以下两个关系（表）：

- Student (学生表)
    
    | StudentID | Name | Major |
    
    | :--- | :--- | :--- |
    
    | 101 | 张三 | CS |
    
    | 102 | 李四 | Math |
    
    | 103 | 王五 | CS |
    
- Enroll (选课表)
    
    | StudentID | CourseID | Grade |
    
    | :--- | :--- | :--- |
    
    | 101 | CS101 | A |
    
    | 101 | Math101 | B |
    
    | 102 | Math101 | A |
    
    | 103 | CS101 | C |
    

---

#### 1. 集合理论操作符 (Set-Theoretic Operators)

这组操作符直接来源于数学中的集合论。它们要求操作的两个关系 R 和 S 必须是**并相容的 (Union-Compatible)**。

> 什么是“并相容”？
> 
> 两个关系 R 和 S 是并相容的，当且仅当：
> 
> 1. 它们具有**相同数量的属性（列）**。
>     
> 2. 它们对应属性的域（Domain）必须相同或兼容（例如，不能一个是 INT，一个是 String）。
>     
>     注意： 属性名（列名）可以不同（尽管我们的语法 UNION 通常要求它们相同或使用 AS 重命名）。
>     

##### 1.1 并 (Union)

- **语法：** `(R UNION S)`
    
    - (标准符号: $R \cup S$)
        
- **语义：** 返回一个**新的关系**，包含所有**存在于 R 中，或者存在于 S 中**的元组（行）。
    
- **关键特性：** 它会自动**去除重复的元组**，因为关系（Relation）是一个“集合”（Set），集合中不允许有重复元素。
    
- **约束：** R 和 S 必须是并相容的。
    
- 示例：
    
    R1 := ((Student WHERE Major = 'CS') {StudentID, Name})
    
    $\rightarrow \{ (101, \text{'张三'}), (103, \text{'王五'}) \}$
    
    R2 := ((Student WHERE Major = 'Math') {StudentID, Name})
    
    $\rightarrow \{ (102, \text{'李四'}) \}$
    
    Result := (R1 UNION R2)
    
    $\rightarrow \{ (101, \text{'张三'}), (103, \text{'王五'}), (102, \text{'李四'}) \}$
    

##### 1.2 差 (Difference)

- **语法：** `(R MINUS S)`
    
    - (标准符号: $R - S$)
        
- **语义：** 返回一个**新的关系**，包含所有**存在于 R 中，但不存在于 S 中**的元组。
    
- **约束：** R 和 S 必须是并相容的。
    
- 示例：
    
    （为了让例子有意义，我们假设学生104也存在于Student表，但他没有选课）
    
    Student (新)
    
    | StudentID | Name | Major |
    
    | :--- | :--- | :--- |
    
    | ... | ... | ... |
    
    | 104 | 赵六 | EE |
    
    R1 := (Student {StudentID}) $\rightarrow \{ (101), (102), (103), (104) \}$
    
    R2 := (Enroll {StudentID}) $\rightarrow \{ (101), (102), (103) \}$
    
    Result := (R1 MINUS R2) $\rightarrow \{ (104) \}$
    
    （这个查询的含义是：“找出所有没有选修任何课程的学生ID”）
    

##### 1.3 交 (Intersection)

- **语法：** `(R INTERSECT S)`
    
    - (标准符号: $R \cap S$)
        
- **语义：** 返回一个**新的关系**，包含所有**同时存在于 R 和 S 中**的元组。
    
- **约束：** R 和 S 必须是并相容的。
    
- 派生性： 这是一个派生操作符。它不是“最小集”的一部分，因为它可以由“差”运算推导出来：
    
    R INTERSECT S $\equiv$ R MINUS (R MINUS S)
    
- 示例：
    
    R1 := ((Enroll WHERE CourseID = 'CS101') {StudentID}) $\rightarrow \{ (101), (103) \}$
    
    R2 := ((Enroll WHERE Grade = 'A') {StudentID}) $\rightarrow \{ (101), (102) \}$
    
    Result := (R1 INTERSECT R2) $\rightarrow \{ (101) \}$
    
    （含义：“找出既选了CS101又拿了A的学生ID”）

#### 2. 专门的关系操作符 (Relational-Specific Operators)

这是Codd对集合论的重大扩展，是关系代数的核心。

##### 2.1 笛卡尔积 (Cartesian Product)

- **语法：** `(R TIMES S)`
    
    - (标准符号: $R \times S$)
        
- **语义：** 也称为“交叉积”（Cross-Product）。它将 R 中的**每一个元组**与 S 中的**每一个元组**进行**拼接（Concatenate）**。
    
- **结果模式（Schema）：**
    
    - 如果 R 有 $k_1$ 个属性，S 有 $k_2$ 个属性，则 `(R TIMES S)` 的结果将有 $k_1 + k_2$ 个属性。
        
    - 如果 R 有 $n_1$ 行，S 有 $n_2$ 行，则 `(R TIMES S)` 的结果将有 $n_1 \times n_2$ 行。
        
- **属性命名：** 如果 R 和 S 有同名的属性（例如，两者都有 `StudentID`），则在结果中必须使用**重命名（AS）**来区分，例如 `Student.StudentID` 和 `Enroll.StudentID`。
    
- 示例： (Student TIMES Enroll)
    
    （为了简化，我们只看 R1 = \{ (101, \text{'张三'}) \} 和 R2 = \{ (101, \text{'CS101', 'A'}), (101, \text{'Math101', 'B'}) \}）
    
    (R1 TIMES R2) 的结果将是：
    
    | S.StudentID | S.Name | E.StudentID | E.CourseID | E.Grade |
    
    | :--- | :--- | :--- | :--- | :--- |
    
    | 101 | 张三 | 101 | CS101 | A |
    
    | 101 | 张三 | 101 | Math101 | B |
    
- **重要性：** 笛卡尔积本身很少单独使用，因为它会产生大量（通常是无意义的）数据。但是，它是**所有连接（Join）操作的理论基础**。
    

##### 2.2 选择 (Selection)

- **语法：** `(R WHERE condition)`
    
    - (标准符号: $\sigma_{\text{condition}} (R)$)
        
- **语义：** 这是一个**行过滤器（Row Filter）**。它从关系 R 中“选择”出所有满足 `condition`（条件）的**元组（行）**。
    
- **结果模式：** 结果关系的模式（列定义）与 R **完全相同**。
    
- **条件（Condition）：**
    
    - 可以是属性的比较，例如 `Major = 'CS'`。
        
    - 可以是属性与常量，或属性与属性的比较。
        
    - 可以使用逻辑运算符组合：`AND`, `OR`, `NOT`。
        
- 示例：
    
    (Student WHERE Major = 'CS')
    
    结果：
    
    | StudentID | Name | Major |
    
    | :--- | :--- | :--- |
    
    | 101 | 张三 | CS |
    
    | 103 | 王五 | CS |
    
    (Enroll WHERE StudentID = 101 AND Grade = 'A')
    
    结果：
    
    | StudentID | CourseID | Grade |
    
    | :--- | :--- | :--- |
    
    | 101 | CS101 | A |
    

##### 2.3 投影 (Projection)

- **语法：** `(R {attribute_list})`
    
    - (标准符号: $\pi_{\text{attribute\_list}} (R)$)
        
- **语义：** 这是一个**列选择器（Column Selector）**。它从关系 R 中选出指定的**属性（列）**，并丢弃其他所有列。
    
- **关键特性：** 投影操作也会**自动去除重复的行**，以保证结果仍然是一个“集合”。
    
- **结果模式：** 结果关系的模式只包含 `attribute_list` 中指定的列。
    
- 示例：
    
    (Student {Name, Major})
    
    结果：
    
    | Name | Major |
    
    | :--- | :--- |
    
    | 张三 | CS |
    
    | 李四 | Math |
    
    | 王五 | CS |
    
    （注意：没有重复行，所以3行都在）
    
    (Student {Major})
    
    结果：
    
    | Major |
    
    | :--- |
    
    | CS |
    
    | Math |
    
    （注意：原始数据有2个'CS'和1个'Math'，但由于投影会去重，'CS'只出现一次）

### 四、 附加与派生的关系操作符 (Additional Operators)

这组操作符虽然可以由基本操作（5个）派生出来，但它们在实际使用中极其频繁，因此被视为关系代数的核心组成部分。

#### 4.1 连接 (Join)

连接是关系数据库中**最强大、最重要**的操作。它用于将**来自不同关系的相关信息组合在一起**。

##### 4.1.1 $\theta$-连接 (Theta Join)

- **语法：** `(R1 JOIN R2 ON condition)`
    
- **语义：** $\theta$-连接是**笛卡尔积**和**选择**的组合。
    
- **定义：** `(R1 JOIN R2 ON \theta)` $\equiv$ `((R1 TIMES R2) WHERE \theta)`
    
- **$\theta$（Theta）** 是一个任意的比较条件，例如 `R.A > S.B`。
    
- 示例： (Student JOIN Enroll ON Student.StudentID = Enroll.StudentID)
    
    这个操作首先计算 (Student TIMES Enroll)（一个 $3 \times 4 = 12$ 行的表），然后筛选出其中 Student.StudentID = Enroll.StudentID 的行。
    
    结果： (注意：存在冗余列 StudentID)
    
    | S.StudentID | Name | Major | E.StudentID | CourseID | Grade |
    
    | :--- | :--- | :--- | :--- | :--- | :--- |
    
    | 101 | 张三 | CS | 101 | CS101 | A |
    
    | 101 | 张三 | CS | 101 | Math101 | B |
    
    | 102 | 李四 | Math | 102 | Math101 | A |
    
    | 103 | 王五 | CS | 103 | CS101 | C |
    

##### 4.1.2 等值连接 (Equijoin)

- **语义：** 这是 $\theta$-连接的一种特殊情况，其中 $\theta$ **仅仅是“相等”（=）** 比较。
    
- **重要性：** 上面的例子就是一个等值连接。这是实践中 99% 的连接类型。
    

##### 4.1.3 自然连接 (Natural Join)

- **语法：** `(R JOIN S)` (不带 `ON` 子句)
    
    - (标准符号: $R \bowtie S$)
        
- **语义：** 这是最常用、最“智能”的连接方式，它在等值连接的基础上做了两项改进：
    
    1. **自动匹配：** 它自动在 R 和 S 中寻找所有**同名**的属性（列）。
        
    2. **执行等值连接：** 在所有这些同名属性上执行等值连接。
        
    3. **消除冗余：** 在结果中，只保留**一份**同名属性。
        
- **示例：** `(Student JOIN Enroll)`
    
    1. RDBMS发现 `Student` 和 `Enroll` 共享一个同名属性：`StudentID`。
        
    2. 它自动执行等值连接：`Student.StudentID = Enroll.StudentID`。
        
    3. 它合并冗余的 StudentID 列。
        
        结果：（这是最“干净”的连接结果）
        
        | StudentID | Name | Major | CourseID | Grade |
        
        | :--- | :--- | :--- | :--- | :--- |
        
        | 101 | 张三 | CS | CS101 | A |
        
        | 101 | 张三 | CS | Math101 | B |
        
        | 102 | 李四 | Math | Math101 | A |
        
        | 103 | 王五 | CS | CS101 | C |
        

#### 4.2 除 (Division)

##### 一、 核心意义：为何需要除法？

关系代数中，除法是唯一一个专门用于处理**全称量词（"For All"，即“对所有的...”）** 需求的

操作符。

它的存在意义是解决一个在 `AND`（与）和 `OR`（或）之外的常见逻辑难题：

- `AND` 逻辑（例如 `P#='P2' AND P#='P4'`）通常作用于**同一行**数据内部。一行数据无法同时满足两个互斥的条件。
    
- 除法解决的是**跨行（Inter-row）的“与”逻辑**。它要寻找的是那些“与一个集合中的**每一个**元素都存在关联”的实体。
    

因此，除法的通法意义在于，它提供了一种标准化的方法，来**识别出那些在某个关联关系中“全勤”或“圆满完成”一个条件清单的候选者**。

---

##### 二、 通用方法：除法运算的三个逻辑角色

无论语法如何（是 $R \div S$ 还是 `A DIVIDEBY B PER C`），一个完整的除法运算在逻辑上**必须**涉及三个组件：

1. **A - 候选域 (Candidate Domain)**
    
    - **角色：** “被考察的对象清单”。
        
    - **意义：** 它定义了我们最终答案的**可能范围**。结果必须是这个域的一个子集。
        
2. **B - 条件集 (Requirement Set / The Checklist)**
    
    - **角色：** “必须被**全部**满足的硬性标准”。
        
    - **意义：** 这是“全称量词”中“所有”所指向的那个集合。它定义了“圆满”的标准。
        
3. **C - 关联证据 (Relationship / The Evidence)**
    
    - **角色：** “事实依据表”。
        
    - **意义：** 它提供了候选者（A）与条件（B）之间的**事实关联**。运算的判断必须_依据_这个关系表。
        

---

##### 三、 运算本质：基于“集合包含”的过滤

除法的**通用算法**可以被描述为一个基于**集合包含关系 (Set Containment)** 的**过滤过程**：

1. **遍历 (Iteration):** 运算器会遍历“候选域 (A)”中的**每一个**候选者 `a`。
    
2. **映射与提取 (Mapping & Extraction):** 对于当前的候选者 `a`，运算器会去“关联证据 (C)”中，找出所有与 `a` 相关的条件，构成一个“**`a`的实际达成集**”（`achieved_set(a)`）。
    
3. **核心测试 (The Core Test):** 运算器将 `a` 的“实际达成集”与“条件集 (B)”进行比较。
    
4. **判断 (Judgment):**
    
    - 如果 `achieved_set(a)` **完全包含（contains）**了“条件集 (B)”（即 `B` 是 `achieved_set(a)` 的子集），
        
    - **则：** 该候选者 `a` 通过测试，被放入最终结果集。
        
    - **否则：** 候选者 `a` 被淘汰。
        

##### 总结

从通法意义上讲，关系除法是一种**高级的、基于集合的过滤器**。

它不关心单个的关联，而是关心**关联的完备性**。它的核心逻辑不是简单的 `A=B`，而是 `Set(A) ⊇ Set(B)`（A的实际达成集 是否 包含了 B的条件要求集）。

`A DIVIDEBY B PER C` 这种语法之所以有意义，是因为它明确地将这个高级过滤器的三个关键输入（候选人A、条件B、证据C）都显式地分离了出来，完美地揭示了除法运算的本质。

### 总结

关系代数是关系数据库的“汇编语言”。它是一种功能强大、数学上严谨的过程性语言，为关系模型提供了坚实的理论操作基础。

使用你所建议的 `(R WHERE ...)` 和 `(R {...})` 这样的线性语法，使得代数表达式的嵌套结构和书写（尤其是在推导复杂操作如“除法”时）变得更加清晰和易于管理。

- **关系封闭性**是其核心特性，允许构建无限复杂的嵌套查询。
    
- 它在实践中的**最大作用**是作为SQL查询优化的**中间语言和理论工具**，优化器通过代数等价变换来寻找最高效的执行计划。
    
- **基本操作（5个）**（`WHERE`, `{...}`, `UNION`, `MINUS`, `TIMES`）构成了理论核心。
    
- **派生操作（3个）**（`JOIN`, `INTERSECT`, `DIVIDEBY`）极大地增强了语言的表达力，特别是“连接”和“除法”。
    

    

深入理解关系代数，就是深入理解了RDBMS的“思考方式”。
