下面按一条主线来讲：

> 数据库解决什么问题 → 关系数据库怎么组织数据 → 表怎么设计 → SQL 怎么查、增、删、改 → 约束、事务、索引、范式这些底层概念为什么重要 → 常见易错点和面试常问点。

# 一、数据库到底是什么

数据库不是简单的 Excel 表格。更准确地说：

**数据库是长期存储、有组织管理、可高效查询、可保证一致性的数据集合。**

比如你做一个音乐平台 MusicHub，里面可能有：

- 用户表：用户 ID、用户名、注册时间；
    
- 歌曲表：歌曲 ID、歌名、歌手、专辑；
    
- 播放记录表：谁在什么时候听了哪首歌；
    
- 收藏表：谁收藏了哪首歌；
    
- 评论表：谁评论了哪首歌。
    

这些数据不能只是随便放在几个 CSV 里，因为真实系统需要解决很多问题：

1. 数据量大时要能快速查询；
    
2. 多人同时修改时不能乱；
    
3. 数据之间要保持一致，比如播放记录中的用户 ID 必须真的存在；
    
4. 数据丢失后要能恢复；
    
5. 不同用户有不同权限；
    
6. 复杂查询要能稳定运行。
    

所以数据库的核心作用不是“存数据”这么简单，而是：

> **高效、可靠、安全、一致地管理数据。**

---

# 二、数据库管理系统 DBMS

我们平时说的 MySQL、PostgreSQL、Oracle、SQL Server，其实不是数据库本身，而是 **数据库管理系统**，英文叫 DBMS，Database Management System。

DBMS 负责：

|功能|说明|
|---|---|
|数据定义|创建数据库、创建表、定义字段类型|
|数据存储|把数据保存到磁盘中|
|数据查询|执行 SQL 查询|
|数据修改|插入、更新、删除数据|
|事务管理|保证多个操作要么都成功，要么都失败|
|并发控制|多人同时操作时避免数据冲突|
|权限控制|控制谁能看、谁能改|
|备份恢复|数据出错后恢复|

可以这样理解：

> 数据库是“仓库”，DBMS 是“仓库管理系统”，SQL 是你和仓库管理系统沟通的语言。

---

# 三、数据库的几种模型

数据库有很多类型，最重要的是关系型数据库。

## 1. 关系型数据库

关系型数据库把数据组织成一张张二维表。

典型代表：

- MySQL
    
- PostgreSQL
    
- Oracle
    
- SQL Server
    
- SQLite
    

关系型数据库的核心特点是：

1. 数据以表的形式存储；
    
2. 表和表之间通过主键、外键建立关系；
    
3. 使用 SQL 操作数据；
    
4. 强调数据一致性和事务。
    

例如：

用户表 `users`

|user_id|username|city|
|---|---|---|
|1|张三|上海|
|2|李四|北京|

订单表 `orders`

|order_id|user_id|amount|
|---|---|---|
|101|1|99|
|102|1|128|
|103|2|66|

这里 `orders.user_id` 指向 `users.user_id`，表示订单属于哪个用户。

---

## 2. 非关系型数据库 NoSQL

NoSQL 不是“不用 SQL”，而是“不完全按照关系表模型组织数据”。

常见类型：

|类型|代表|适合场景|
|---|---|---|
|文档数据库|MongoDB|JSON 文档、灵活结构|
|键值数据库|Redis|缓存、高速读写|
|列族数据库|HBase|海量分布式数据|
|图数据库|Neo4j|社交网络、知识图谱|

不过你现在学习数据库基础和 SQL，重点还是关系型数据库。

---

# 四、关系型数据库的基本概念

关系数据库里的几个核心概念一定要分清楚。

## 1. 表 Table

表是存储同一类对象的数据集合。

例如学生表：

```sql
students
```

|student_id|name|gender|age|major|
|---|---|---|---|---|
|1|张三|男|20|数据科学|
|2|李四|女|19|金融学|

这张表描述的是“学生”这种实体。

---

## 2. 行 Row / 记录 Record

表中的一行代表一个具体对象。

比如：

|student_id|name|gender|age|major|
|---|---|---|---|---|
|1|张三|男|20|数据科学|

这一行就是一条学生记录。

---

## 3. 列 Column / 字段 Field

表中的一列代表对象的一个属性。

例如：

- `student_id`：学号；
    
- `name`：姓名；
    
- `age`：年龄；
    
- `major`：专业。
    

---

## 4. 数据类型 Data Type

每个字段都要有数据类型。

常见类型：

|类型|含义|示例|
|---|---|---|
|INT|整数|1, 100|
|BIGINT|大整数|用户 ID、订单 ID|
|DECIMAL|精确小数|金额、价格|
|FLOAT / DOUBLE|浮点数|科学计算|
|CHAR|固定长度字符串|性别、状态码|
|VARCHAR|可变长度字符串|姓名、标题|
|TEXT|长文本|评论、简介|
|DATE|日期|2026-06-03|
|DATETIME / TIMESTAMP|日期时间|2026-06-03 10:30:00|
|BOOLEAN|布尔值|true / false|

注意一个很重要的点：

> 金额一般不要用 FLOAT，而要用 DECIMAL。

因为 FLOAT 是近似存储，可能出现精度误差。比如 0.1 + 0.2 不一定精确等于 0.3。金额、价格、利率等需要精确表达的数据，最好用 `DECIMAL(10,2)` 这类类型。

---

# 五、主键、外键和约束

这是数据库基础里最重要的一块。

---

## 1. 主键 Primary Key

主键是表中用于唯一标识一条记录的字段。

例如学生表：

```sql
student_id
```

每个学生的学号不同，所以可以作为主键。

```sql
CREATE TABLE students (
    student_id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT
);
```

主键有两个核心要求：

1. 唯一；
    
2. 不能为空。
    

也就是说，主键字段不能出现重复值，也不能是 `NULL`。

---

## 2. 为什么需要主键

假设你有一个学生表：

|name|age|major|
|---|---|---|
|张三|20|数据科学|
|张三|20|金融学|

只靠姓名无法区分这两个张三。这个时候就需要一个唯一 ID。

|student_id|name|age|major|
|---|---|---|---|
|1001|张三|20|数据科学|
|1002|张三|20|金融学|

所以主键本质上解决的是：

> 如何唯一定位一条记录。

---

## 3. 自增主键

很多数据库会使用自增 ID。

```sql
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50)
);
```

插入数据时，不需要手动写 `user_id`，数据库会自动生成。

```sql
INSERT INTO users (username)
VALUES ('张三');
```

---

## 4. 自然主键和代理主键

主键有两类：

### 自然主键

自然存在、具有业务含义的字段。

比如：

- 身份证号；
    
- 学号；
    
- 邮箱；
    
- 手机号。
    

### 代理主键

人为创建的无业务含义 ID。

比如：

- `user_id`
    
- `order_id`
    
- `song_id`
    

实际项目中更推荐使用代理主键，因为自然主键可能变化。

例如用户换手机号、换邮箱，都会导致自然主键不稳定。

---

## 5. 外键 Foreign Key

外键用于建立表与表之间的关系。

例如：

学生表：

|student_id|name|
|---|---|
|1|张三|
|2|李四|

选课表：

|enroll_id|student_id|course_id|
|---|---|---|
|1|1|101|
|2|2|102|

这里 `enrollments.student_id` 指向 `students.student_id`。

建表时可以写：

```sql
CREATE TABLE enrollments (
    enroll_id INT PRIMARY KEY,
    student_id INT,
    course_id INT,
    FOREIGN KEY (student_id) REFERENCES students(student_id)
);
```

外键的作用是保证引用完整性。

也就是说：

> 选课表里不能出现一个不存在的学生 ID。

如果 `students` 表里没有 `student_id = 999`，那么 `enrollments` 里就不能插入 `student_id = 999` 的选课记录。

---

## 6. 常见约束

|约束|含义|
|---|---|
|PRIMARY KEY|主键，唯一且非空|
|FOREIGN KEY|外键，引用其他表|
|NOT NULL|不允许为空|
|UNIQUE|唯一约束|
|CHECK|检查条件|
|DEFAULT|默认值|

例子：

```sql
CREATE TABLE products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    price DECIMAL(10, 2) CHECK (price >= 0),
    status VARCHAR(20) DEFAULT 'active',
    sku VARCHAR(50) UNIQUE
);
```

含义是：

- `product_id` 是主键；
    
- `product_name` 不能为空；
    
- `price` 不能小于 0；
    
- `status` 默认是 active；
    
- `sku` 不能重复。

# 六、实体、属性、关系：E-R 模型

在真正建表之前，通常要先做概念设计，也就是 E-R 模型。

E-R 是 Entity-Relationship 的缩写，即实体-关系模型。

---

## 1. 实体 Entity

实体就是现实世界中可以独立描述的对象。

比如：

- 学生；
    
- 课程；
    
- 教师；
    
- 用户；
    
- 商品；
    
- 订单；
    
- 歌曲。
    

对应到数据库里，实体通常会变成一张表。

---

## 2. 属性 Attribute

属性是实体的特征。

学生实体的属性：

- 学号；
    
- 姓名；
    
- 性别；
    
- 年龄；
    
- 专业。
    

歌曲实体的属性：

- 歌曲 ID；
    
- 歌名；
    
- 歌手；
    
- 时长；
    
- 发布时间。
    

---

## 3. 关系 Relationship

关系描述实体之间的联系。

例如：

- 学生选修课程；
    
- 用户购买商品；
    
- 用户收藏歌曲；
    
- 歌手发布歌曲；
    
- 订单包含商品。
    

---

# 七、关系的三种类型

实体之间的关系通常有三种。

---

## 1. 一对一关系 1:1

一个 A 对应一个 B，一个 B 也只对应一个 A。

例如：

- 一个学生对应一张校园卡；
    
- 一个人对应一份身份证信息。
    

数据库设计时，可以把两个实体合并成一张表，也可以拆成两张表。

---

## 2. 一对多关系 1:N

一个 A 可以对应多个 B，但一个 B 只能对应一个 A。

例如：

- 一个班级有多个学生；
    
- 一个用户有多个订单；
    
- 一个歌手有多首歌曲。
    

实现方法：

> 在“多”的一方添加外键。

例如一个用户有多个订单：

```sql
users(user_id, username)
orders(order_id, user_id, amount)
```

`orders.user_id` 是外键，指向 `users.user_id`。

---

## 3. 多对多关系 M:N

一个 A 可以对应多个 B，一个 B 也可以对应多个 A。

例如：

- 学生和课程：一个学生可以选多门课，一门课也可以被多个学生选；
    
- 用户和歌曲：一个用户可以收藏多首歌，一首歌也可以被多个用户收藏；
    
- 订单和商品：一个订单可以包含多个商品，一个商品也可以出现在多个订单里。
    

多对多关系不能直接用一个外键解决，需要增加一张中间表。

例如学生选课：

```sql
students(student_id, name)
courses(course_id, course_name)
enrollments(student_id, course_id, enroll_time, grade)
```

`enrollments` 就是中间表，也叫关联表。

---

# 八、表设计的核心原则

数据库表设计最重要的是：

> 不要重复存储同一事实，避免数据冗余和更新异常。

比如有一张表：

|student_id|student_name|course_id|course_name|teacher|
|---|---|---|---|---|
|1|张三|101|数据库|王老师|
|2|李四|101|数据库|王老师|
|3|王五|101|数据库|王老师|

这里 `course_name = 数据库`、`teacher = 王老师` 重复出现多次。

如果王老师换成李老师，你要改很多行。万一漏改一行，就会数据不一致。

更合理的设计是拆成：

学生表：

```text
students(student_id, student_name)
```

课程表：

```text
courses(course_id, course_name, teacher)
```

选课表：

```text
enrollments(student_id, course_id)
```

这样课程信息只存一次。

---

# 九、范式：1NF、2NF、3NF、BCNF

范式是数据库表设计的规范，用来减少冗余、避免异常。

你不需要把它学成纯理论，但必须理解它解决什么问题。

---

## 1. 第一范式 1NF

第一范式要求：

> 每个字段都是不可再分的原子值。

错误例子：

|student_id|name|phones|
|---|---|---|
|1|张三|138xxx, 139xxx|

`phones` 里放了多个手机号，不符合 1NF。

更合理的设计：

学生表：

|student_id|name|
|---|---|
|1|张三|

手机号表：

|student_id|phone|
|---|---|
|1|138xxx|
|1|139xxx|

一句话：

> 一个字段里不要塞多个值。

---

## 2. 第二范式 2NF

第二范式要求：

> 在满足 1NF 的基础上，非主属性必须完全依赖于整个主键，不能只依赖组合主键的一部分。

这个主要出现在组合主键中。

例如选课表：

|student_id|course_id|student_name|course_name|grade|
|---|---|---|---|---|
|1|101|张三|数据库|90|

主键是 `(student_id, course_id)`。

但是：

- `student_name` 只依赖 `student_id`；
    
- `course_name` 只依赖 `course_id`；
    
- `grade` 才依赖 `(student_id, course_id)`。
    

所以这张表不满足 2NF。

应该拆成：

```text
students(student_id, student_name)
courses(course_id, course_name)
enrollments(student_id, course_id, grade)
```

一句话：

> 表里每个非主键字段都应该依赖整个主键，而不是只依赖主键的一部分。

---

## 3. 第三范式 3NF

第三范式要求：

> 在满足 2NF 的基础上，非主属性不能传递依赖于主键。

例子：

|student_id|student_name|dept_id|dept_name|
|---|---|---|---|
|1|张三|D01|信息学院|

主键是 `student_id`。

依赖关系是：

```text
student_id -> dept_id
dept_id -> dept_name
```

所以：

```text
student_id -> dept_name
```

这是传递依赖。

问题是：学院名称其实不应该放在学生表里，因为它依赖的是 `dept_id`，不是直接依赖学生。

更合理的设计：

```text
students(student_id, student_name, dept_id)
departments(dept_id, dept_name)
```

一句话：

> 非主键字段不要依赖另一个非主键字段。

---

## 4. BCNF

BCNF 比 3NF 更严格。

它要求：

> 每一个决定因素都必须是候选键。

用更直观的话说：

> 只要某个字段或字段组能决定其他字段，它本身就应该能唯一标识一条记录。

BCNF 常用于处理一些更复杂的依赖关系。课程里如果讲 BCNF，通常重点是判断函数依赖和分解表。

一般项目中，你至少要做到 3NF；如果存在复杂组合依赖，再考虑 BCNF。

# 十、SQL 是什么

SQL，全称 Structured Query Language，结构化查询语言。

SQL 是关系型数据库的标准操作语言，用来：

1. 创建数据库和表；
    
2. 插入数据；
    
3. 查询数据；
    
4. 更新数据；
    
5. 删除数据；
    
6. 管理权限；
    
7. 控制事务。
    

SQL 可以分成几类：

|类别|英文|作用|常见命令|
|---|---|---|---|
|DDL|Data Definition Language|定义结构|CREATE, ALTER, DROP|
|DML|Data Manipulation Language|操作数据|INSERT, UPDATE, DELETE|
|DQL|Data Query Language|查询数据|SELECT|
|DCL|Data Control Language|权限控制|GRANT, REVOKE|
|TCL|Transaction Control Language|事务控制|COMMIT, ROLLBACK|

---

# 十一、DDL：创建、修改、删除表

DDL 用来定义数据库结构。

---

## 1. 创建数据库

```sql
CREATE DATABASE school;
```

使用数据库：

```sql
USE school;
```

---

## 2. 创建表

```sql
CREATE TABLE students (
    student_id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    gender CHAR(1),
    age INT,
    major VARCHAR(100),
    enroll_date DATE
);
```

---

## 3. 查看表结构

MySQL 中：

```sql
DESC students;
```

或者：

```sql
SHOW COLUMNS FROM students;
```

---

## 4. 修改表结构

新增字段：

```sql
ALTER TABLE students
ADD email VARCHAR(100);
```

修改字段类型：

```sql
ALTER TABLE students
MODIFY email VARCHAR(150);
```

删除字段：

```sql
ALTER TABLE students
DROP COLUMN email;
```

---

## 5. 删除表

```sql
DROP TABLE students;
```

`DROP TABLE` 是删除整张表，包括结构和数据，风险很大。

---

## 6. 清空表

```sql
TRUNCATE TABLE students;
```

`TRUNCATE` 删除所有数据，但保留表结构。

和 `DELETE` 的区别：

```sql
DELETE FROM students;
```

|命令|作用|是否保留表结构|是否可加 WHERE|
|---|---|---|---|
|DELETE|删除数据|保留|可以|
|TRUNCATE|清空整表|保留|不可以|
|DROP|删除整张表|不保留|不可以|

---

# 十二、DML：插入、更新、删除数据

---

## 1. 插入数据 INSERT

```sql
INSERT INTO students (student_id, name, gender, age, major)
VALUES (1, '张三', '男', 20, '数据科学');
```

一次插入多行：

```sql
INSERT INTO students (student_id, name, gender, age, major)
VALUES 
(1, '张三', '男', 20, '数据科学'),
(2, '李四', '女', 19, '金融学'),
(3, '王五', '男', 21, '统计学');
```

---

## 2. 更新数据 UPDATE

```sql
UPDATE students
SET major = '数据科学与大数据技术'
WHERE student_id = 1;
```

注意：

> `UPDATE` 一定要小心 `WHERE`。

如果不写 `WHERE`：

```sql
UPDATE students
SET major = '数据科学与大数据技术';
```

这会把所有学生的专业都改掉。

---

## 3. 删除数据 DELETE

```sql
DELETE FROM students
WHERE student_id = 1;
```

如果不写 `WHERE`：

```sql
DELETE FROM students;
```

会删除整张表的数据。

---

# 十三、SELECT 查询基础

SQL 学习最核心的是 `SELECT`。

假设有一张学生表：

```text
students(student_id, name, gender, age, major, score)
```

---

## 1. 查询所有列

```sql
SELECT *
FROM students;
```

`*` 表示查询所有字段。

实际项目中不推荐频繁使用 `SELECT *`，因为：

1. 不知道具体查了哪些字段；
    
2. 数据量可能很大；
    
3. 表结构变化后可能影响代码；
    
4. 无法充分利用覆盖索引。
    

更推荐写清楚字段：

```sql
SELECT student_id, name, age
FROM students;
```

---

## 2. 起别名

```sql
SELECT 
    student_id AS 学号,
    name AS 姓名,
    score AS 分数
FROM students;
```

`AS` 可以省略：

```sql
SELECT name 姓名
FROM students;
```

但为了可读性，建议保留 `AS`。

---

## 3. 去重 DISTINCT

## 7. NULL 的判断

这是 SQL 里非常容易错的地方。

不能写：

按分数从高到低排序：

```sql
SELECT *
FROM students

```

```sql
SELECT *

SELECT *

```

含义是：

2. 同一专业内再按分数降序。

聚合函数用于对多行数据做统计。

|函数|含义|
|---|---|
|COUNT|计数|
|SUM|求和|
|AVG|平均值|
|MAX|最大值|
|MIN|最小值|

例子：

```sql
SELECT COUNT(*) AS student_count

```

```sql
SELECT AVG(score) AS avg_score

```

查询最高分和最低分：

```sql
SELECT

```

---

## COUNT(*) 和 COUNT(字段) 的区别

```sql
COUNT(*)
```

统计行数，包括字段为 NULL 的行。

```sql
COUNT(email)
```

统计 `email` 不为 NULL 的行数。

例如：

|student_id|email|
|---|---|
|1|[a@qq.com](mailto:a@qq.com)|
|2|NULL|
|3|[b@qq.com](mailto:b@qq.com)|

结果：

```sql

```

# 十七、分组 GROUP BY

分组用于“按某个维度统计”。

例如，按专业统计人数：

```sql
SELECT 
    major,
    COUNT(*) AS student_count
FROM students
GROUP BY major;
```

结果可能是：

|major|student_count|
|---|---|
|数据科学|30|
|金融学|45|
|统计学|25|

按专业统计平均分：

```sql
SELECT 
    major,
    AVG(score) AS avg_score
FROM students
GROUP BY major;
```

---

## GROUP BY 的核心理解

`GROUP BY major` 的意思是：

> 先把所有学生按 major 分成若干组，然后每组分别计算统计量。

没有分组时：

```sql
SELECT AVG(score)
FROM students;
```

是全体学生的平均分。

有分组时：

```sql
SELECT major, AVG(score)
FROM students
GROUP BY major;
```

是每个专业各自的平均分。

---

## GROUP BY 的重要规则

使用 `GROUP BY` 时，`SELECT` 后面通常只能放：

1. 分组字段；
    
2. 聚合函数。
    

比如：

```sql
SELECT major, AVG(score)
FROM students
GROUP BY major;
```

这是合理的。

但是：

```sql
SELECT major, name, AVG(score)
FROM students
GROUP BY major;
```

一般是不合理的。

因为每个专业里有很多学生，数据库不知道应该显示哪个 `name`。

---

# 十八、HAVING：对分组后的结果筛选

`WHERE` 是分组前筛选。

`HAVING` 是分组后筛选。

比如查询平均分大于 85 的专业：

```sql
SELECT 
    major,
    AVG(score) AS avg_score
FROM students
GROUP BY major
HAVING AVG(score) > 85;
```

不能写：

```sql
WHERE AVG(score) > 85
```

因为 `WHERE` 执行时还没有分组，也还没有计算平均分。

---

## WHERE 和 HAVING 的区别

|比较|WHERE|HAVING|
|---|---|---|
|作用对象|原始行|分组后的结果|
|是否可用聚合函数|一般不可以|可以|
|执行顺序|GROUP BY 之前|GROUP BY 之后|

例子：

查询每个专业中，年龄大于 18 的学生的平均分，并只保留平均分大于 85 的专业：

```sql
SELECT 
    major,
    AVG(score) AS avg_score
FROM students
WHERE age > 18
GROUP BY major
HAVING AVG(score) > 85;
```

逻辑是：

1. 先筛选年龄大于 18 的学生；
    
2. 再按专业分组；
    
3. 计算每组平均分；
    
4. 保留平均分大于 85 的专业。
    

---

# 十九、SQL 的逻辑执行顺序

这是非常重要的知识点。

虽然 SQL 写法是：

```sql
SELECT
FROM
WHERE
GROUP BY
HAVING
ORDER BY
LIMIT
```

但实际逻辑执行顺序大致是：

```text
FROM
WHERE
GROUP BY
HAVING
SELECT
ORDER BY
LIMIT
```

也就是：

1. 先确定从哪张表取数据；
    
2. 再筛选原始行；
    
3. 再分组；
    
4. 再筛选分组；
    
5. 再选择输出字段；
    
6. 再排序；
    
7. 最后限制行数。
    

所以为什么 `WHERE` 不能使用 `SELECT` 里定义的别名？

例如：

```sql
SELECT score * 1.1 AS new_score
FROM students
WHERE new_score > 90;
```

很多数据库不支持，因为 `WHERE` 执行时，`SELECT` 还没执行，`new_score` 这个别名还不存在。

可以改成：

```sql
SELECT score * 1.1 AS new_score
FROM students
WHERE score * 1.1 > 90;
```

或者用子查询。

# 二十、多表查询 JOIN

真实项目中，数据通常分散在多张表里，所以必须使用 JOIN。

假设有两张表：

用户表 `users`

|user_id|username|
|---|---|
|1|张三|
|2|李四|
|3|王五|

订单表 `orders`

|order_id|user_id|amount|
|---|---|---|
|101|1|99|
|102|1|128|
|103|2|66|

---

## 1. INNER JOIN 内连接

查询每个订单对应的用户：

```sql
SELECT 
    o.order_id,
    u.username,
    o.amount
FROM orders o
INNER JOIN users u
    ON o.user_id = u.user_id;
```

结果：

|order_id|username|amount|
|---|---|---|
|101|张三|99|
|102|张三|128|
|103|李四|66|

`INNER JOIN` 只保留两张表能匹配上的记录。

王五没有订单，所以不会出现。

---

## 2. LEFT JOIN 左连接

查询所有用户及其订单：

```sql
SELECT 
    u.user_id,
    u.username,
    o.order_id,
    o.amount
FROM users u
LEFT JOIN orders o
    ON u.user_id = o.user_id;
```

结果：

|user_id|username|order_id|amount|
|---|---|---|---|
|1|张三|101|99|
|1|张三|102|128|
|2|李四|103|66|
|3|王五|NULL|NULL|

`LEFT JOIN` 会保留左表的所有记录，即使右表没有匹配。

所以王五也会出现，只是订单相关字段为 NULL。

---

## 3. RIGHT JOIN 右连接

```sql
SELECT *
FROM users u
RIGHT JOIN orders o
    ON u.user_id = o.user_id;
```

右连接保留右表所有记录。

实际项目中，`RIGHT JOIN` 用得相对少，因为大多数情况可以改写成 `LEFT JOIN`，可读性更统一。

---

## 4. FULL JOIN 全连接

全连接保留左右两张表的全部记录，匹配不上就填 NULL。

```sql
SELECT *
FROM users u
FULL JOIN orders o
    ON u.user_id = o.user_id;
```

MySQL 不直接支持 `FULL JOIN`，通常用 `LEFT JOIN UNION RIGHT JOIN` 模拟。

---

## 5. CROSS JOIN 笛卡尔积

```sql
SELECT *
FROM users
CROSS JOIN products;
```

结果是用户和商品的所有组合。

如果 users 有 100 行，products 有 50 行，结果就是 5000 行。

如果普通 JOIN 忘了写连接条件，也可能产生笛卡尔积，导致结果暴增。

---

## 6. SELF JOIN 自连接

同一张表自己和自己连接。

例如员工表：

|emp_id|emp_name|manager_id|
|---|---|---|
|1|张总|NULL|
|2|李经理|1|
|3|王员工|2|

查询每个员工的上级：

```sql
SELECT 
    e.emp_name AS employee,
    m.emp_name AS manager
FROM employees e
LEFT JOIN employees m
    ON e.manager_id = m.emp_id;
```

---

# 二十一、JOIN 的核心本质

JOIN 的本质是：

> 根据某个条件，把两张表中有关联的行拼接起来。

比如：

```sql
orders.user_id = users.user_id
```

就是把订单和用户连起来的桥。

一定要注意：

> JOIN 之后的行数不一定等于原表行数。

如果一个用户有多个订单，那么用户信息会重复出现多次。

例如张三有两个订单，JOIN 后张三会出现两行。

这是正常的，不是数据错误。

---

# 二十二、LEFT JOIN 的常见坑

假设你想查询所有用户以及他们金额大于 100 的订单。

错误写法：

```sql
SELECT 
    u.username,
    o.order_id,
    o.amount
FROM users u
LEFT JOIN orders o
    ON u.user_id = o.user_id
WHERE o.amount > 100;
```

这会把没有订单的用户过滤掉，因为没有订单的用户 `o.amount` 是 NULL，而 `NULL > 100` 不成立。

这样 `LEFT JOIN` 实际上变成了类似 `INNER JOIN`。

更合理的写法：

```sql
SELECT 
    u.username,
    o.order_id,
    o.amount
FROM users u
LEFT JOIN orders o
    ON u.user_id = o.user_id
   AND o.amount > 100;
```

区别在于：

- 放在 `ON` 里：限制右表匹配条件，仍保留左表所有用户；
    
- 放在 `WHERE` 里：对连接后的结果整体筛选，可能把左表无匹配记录删掉。
    

这是 SQL 面试和项目里非常常见的坑。

# 二十三、子查询 Subquery

子查询就是 SQL 里面嵌套 SQL。

---

## 1. WHERE 子查询

查询分数高于平均分的学生：

```sql
SELECT *
FROM students
WHERE score > (
    SELECT AVG(score)
    FROM students
);
```

里面这个：

```sql
SELECT AVG(score)
FROM students
```

会先算出平均分，然后外层查询用它筛选学生。

---

## 2. IN 子查询

查询选过数据库课程的学生：

```sql
SELECT *
FROM students
WHERE student_id IN (
    SELECT student_id
    FROM enrollments
    WHERE course_id = 101
);
```

---

## 3. EXISTS 子查询

```sql
SELECT *
FROM students s
WHERE EXISTS (
    SELECT 1
    FROM enrollments e
    WHERE e.student_id = s.student_id
);
```

含义是：

> 查询至少有一条选课记录的学生。

`EXISTS` 更强调“是否存在”，不关心子查询返回什么具体值。

---

## 4. FROM 子查询

可以把一个查询结果当成临时表。

```sql
SELECT major, avg_score
FROM (
    SELECT 
        major,
        AVG(score) AS avg_score
    FROM students
    GROUP BY major
) t
WHERE avg_score > 85;
```

# 二十四、CTE：WITH 子句

CTE 可以让复杂 SQL 更清晰。

```sql
WITH major_avg AS (
    SELECT 
        major,
        AVG(score) AS avg_score
    FROM students
    GROUP BY major
)
SELECT *
FROM major_avg
WHERE avg_score > 85;
```

它和 FROM 子查询类似，但可读性更好。

复杂数据分析 SQL 中，CTE 很常用。

例如：

```sql
WITH user_order AS (
    SELECT 
        user_id,
        COUNT(*) AS order_count,
        SUM(amount) AS total_amount
    FROM orders
    GROUP BY user_id
),
high_value_user AS (
    SELECT *
    FROM user_order
    WHERE total_amount > 1000
)
SELECT *
FROM high_value_user;
```

逻辑会比层层嵌套子查询清晰很多。

# 二十五、集合操作：UNION、UNION ALL

## 1. UNION

合并两个查询结果，并自动去重。

```sql
SELECT user_id FROM online_users
UNION
SELECT user_id FROM offline_users;
```

---

## 2. UNION ALL

合并两个查询结果，不去重。

```sql
SELECT user_id FROM online_users
UNION ALL
SELECT user_id FROM offline_users;
```

区别：

|操作|是否去重|性能|
|---|---|---|
|UNION|去重|较慢|
|UNION ALL|不去重|较快|

实际项目中，能用 `UNION ALL` 就不要随便用 `UNION`，除非你明确需要去重。

# 二十六、窗口函数：数据分析非常重要

窗口函数不是最基础的 SQL，但对数分、数据科学、面试非常重要，建议你尽早掌握。

窗口函数的特点是：

> 不把多行压缩成一行，而是在保留原始行的基础上计算排名、累计值、组内统计量等。

普通聚合：

```sql
SELECT major, AVG(score)
FROM students
GROUP BY major;
```

结果是一组一行。

窗口函数：

```sql
SELECT 
    student_id,
    name,
    major,
    score,
    AVG(score) OVER (PARTITION BY major) AS major_avg_score
FROM students;
```

结果仍然每个学生一行，但额外多了一列“本专业平均分”。

---

## 1. 排名函数

按分数排名：

```sql
SELECT 
    name,
    score,
    RANK() OVER (ORDER BY score DESC) AS ranking
FROM students;
```

常见排名函数：

|函数|特点|
|---|---|
|ROW_NUMBER|不管是否并列，都给唯一序号|
|RANK|并列同名次，后面跳号|
|DENSE_RANK|并列同名次，后面不跳号|

例如分数：

|name|score|
|---|---|
|A|100|
|B|90|
|C|90|
|D|80|

结果：

|name|ROW_NUMBER|RANK|DENSE_RANK|
|---|---|---|---|
|A|1|1|1|
|B|2|2|2|
|C|3|2|2|
|D|4|4|3|

---

## 2. 分组内排名

查询每个专业内部的分数排名：

```sql
SELECT 
    name,
    major,
    score,
    RANK() OVER (
        PARTITION BY major
        ORDER BY score DESC
    ) AS major_rank
FROM students;
```

`PARTITION BY major` 表示每个专业内部单独排名。

---

## 3. 累计求和

查询每个用户订单金额的累计值：

```sql
SELECT 
    user_id,
    order_time,
    amount,
    SUM(amount) OVER (
        PARTITION BY user_id
        ORDER BY order_time
    ) AS cumulative_amount
FROM orders;
```
- **`SUM(amount)`**：对金额进行累加求和。
    
- **`OVER`**：表示这是一个窗口函数，定义它作用的“窗口”范围。
    
- **`PARTITION BY user_id`**：**分组**。意思是按不同用户分别计算，每个用户的累积金额互不干扰。
    
- **`ORDER BY order_time`**：**排序**。在每个用户内部，按订单时间从早到晚排序。
    
- **累加逻辑**：`SUM(amount)` 配合 `ORDER BY`，会从该用户的**第一笔订单开始**，**逐行累加**到当前行。
    

 **结果示例**

假设 `orders` 表数据如下：

|user_id|order_time|amount|
|---|---|---|
|101|2025-01-01|100|
|101|2025-01-15|200|
|102|2025-01-10|50|
|101|2025-02-01|150|

执行这条 SQL 后，结果会是：

| user_id | order_time | amount | cumulative_amount        |
| ------- | ---------- | ------ | ------------------------ |
| 101     | 2025-01-01 | 100    | **100** (100)            |
| 101     | 2025-01-15 | 200    | **300** (100+200)        |
| 101     | 2025-02-01 | 150    | **450** (100+200+150)    |
| 102     | 2025-01-10 | 50     | **50** (50，因为是另一个用户重新累加) |
这在用户生命周期价值、累计消费、留存分析中很常见。

# 二十七、视图 View

视图可以理解为保存下来的查询。

```sql
CREATE VIEW high_score_students AS
SELECT *
FROM students
WHERE score >= 90;
```

之后可以像查表一样查视图：

```sql
SELECT *
FROM high_score_students;
```

视图的作用：

1. 简化复杂查询；
    
2. 提高复用性；
    
3. 隐藏底层表结构；
    
4. 做权限控制。
    

但视图通常不直接存储数据，它只是一个查询定义。查询视图时，数据库会根据视图定义去查底层表。

# 二十八、索引 Index

索引是数据库性能优化里最核心的概念。

## 1. 索引是什么

索引可以理解为数据库表的“目录”。

没有索引时，查一条记录可能要从头扫到尾。

比如：

```sql
SELECT *
FROM students
WHERE student_id = 1001;
```

如果没有索引，数据库可能要扫描整张表。

有索引后，数据库可以快速定位到 `student_id = 1001` 的位置。

就像查字典：

- 没目录：从第一页翻到最后；
    
- 有目录：直接根据拼音或部首定位。
    

---

## 2. 常见索引类型

| 类型   | 说明        |
| ---- | --------- |
| 主键索引 | 主键自动建立索引  |
| 唯一索引 | 保证字段唯一    |
| 普通索引 | 提高查询速度    |
| 联合索引 | 多个字段组合建索引 |
| 全文索引 | 用于文本搜索    |

---

## 3. 创建索引

```sql
CREATE INDEX idx_students_major
ON students(major);
```

联合索引：

```sql
CREATE INDEX idx_orders_user_time
ON orders(user_id, order_time);
```

---

## 4. 索引的优点和缺点

优点：

1. 加快查询；
    
2. 加快排序；
    
3. 加快分组；
    
4. 加快连接。
    

缺点：

1. 占用额外存储空间；
    
2. 插入、更新、删除会变慢；
    
3. 索引太多会增加维护成本。
    

所以索引不是越多越好。

---

## 5. 什么字段适合建索引

适合建索引的字段：

1. 经常用于 `WHERE` 的字段；
    
2. 经常用于 `JOIN` 的字段；
    
3. 经常用于 `ORDER BY` 的字段；
    
4. 经常用于 `GROUP BY` 的字段；
    
5. 区分度较高的字段。
    

例如：

```sql
WHERE user_id = ?
```

`user_id` 适合建索引。

```sql
WHERE order_time BETWEEN ? AND ?
```

`order_time` 也适合建索引。

---

## 6. 什么字段不适合建索引

不太适合建索引：

1. 区分度很低的字段；
    
2. 经常更新的字段；
    
3. 很少查询的字段；
    
4. 小表字段。
    

例如性别字段只有“男/女”，区分度很低，单独建索引通常意义不大。

---

## 7. 联合索引和最左前缀原则

假设有联合索引：

```sql
CREATE INDEX idx_user_time_status
ON orders(user_id, order_time, status);
```

这个索引可以较好支持：

```sql
WHERE user_id = 1
```

```sql
WHERE user_id = 1 AND order_time >= '2026-01-01'
```

```sql
WHERE user_id = 1 AND order_time >= '2026-01-01' AND status = 'paid'
```

但不一定能很好支持：

```sql
WHERE order_time >= '2026-01-01'
```

因为没有从最左边的 `user_id` 开始使用。

这就是最左前缀原则。

# 二十九、事务 Transaction

事务是数据库可靠性的核心。

## 1. 什么是事务

事务是一组操作，要么全部成功，要么全部失败。

经典例子：银行转账。

张三给李四转账 100 元：

```text
1. 张三账户 -100
2. 李四账户 +100
```

这两个操作必须绑定在一起。

不能出现：

- 张三扣钱了，李四没收到；
    
- 李四收到钱了，张三没扣钱。
    

所以转账必须放在一个事务里。

---

## 2. 事务的 ACID 特性

事务有四个特性：ACID。

|特性|英文|含义|
|---|---|---|
|原子性|Atomicity|要么全做，要么全不做|
|一致性|Consistency|事务前后数据满足约束|
|隔离性|Isolation|并发事务之间互不干扰|
|持久性|Durability|提交后数据永久保存|

---

## 3. 原子性 Atomicity

事务中的操作不可分割。

例如转账：

```sql
BEGIN;

UPDATE accounts
SET balance = balance - 100
WHERE user_id = 1;

UPDATE accounts
SET balance = balance + 100
WHERE user_id = 2;

COMMIT;
```

如果中间出错：

```sql
ROLLBACK;
```

全部回滚。

---

## 4. 一致性 Consistency

事务执行前后，数据库必须从一个合法状态变成另一个合法状态。

比如账户余额不能小于 0，订单必须对应真实用户，这些约束在事务前后都应该成立。

---

## 5. 隔离性 Isolation

多个事务同时执行时，不能互相干扰到错误结果。

比如两个人同时购买库存只剩 1 件的商品，不能都购买成功。

---

## 6. 持久性 Durability

事务一旦提交，即使数据库崩溃，数据也应该能恢复。

---

# 三十、并发问题和隔离级别

当多个事务并发执行时，会出现一些问题。

---

## 1. 脏读 Dirty Read

一个事务读到了另一个事务还没提交的数据。

例如：

1. 事务 A 把余额从 100 改成 50，但还没提交；
    
2. 事务 B 读取余额为 50；
    
3. 事务 A 回滚，余额恢复为 100。
    

事务 B 读到的 50 就是脏数据。

---

## 2. 不可重复读 Non-repeatable Read

同一个事务里，两次读取同一行数据，结果不一样。

例如：

1. 事务 A 第一次读余额是 100；
    
2. 事务 B 修改余额为 80 并提交；
    
3. 事务 A 第二次读余额变成 80。
    

---

## 3. 幻读 Phantom Read

同一个事务里，两次查询一个范围，结果行数不一样。

例如：

1. 事务 A 查询年龄大于 18 的学生，有 10 人；
    
2. 事务 B 插入一个年龄 20 的学生并提交；
    
3. 事务 A 再查，变成 11 人。
    

---

## 4. 四种隔离级别

|隔离级别|脏读|不可重复读|幻读|
|---|---|---|---|
|READ UNCOMMITTED|可能|可能|可能|
|READ COMMITTED|不会|可能|可能|
|REPEATABLE READ|不会|不会|可能|
|SERIALIZABLE|不会|不会|不会|

隔离级别越高，数据一致性越强，但并发性能通常越差。

MySQL InnoDB 默认通常是 `REPEATABLE READ`。

---

# 三十一、SQL 查询优化基础

数据库查询慢时，通常从几个角度排查。

## 1. 是否扫描了太多行

例如：

```sql
SELECT *
FROM orders
WHERE user_id = 1001;
```

如果 `user_id` 没有索引，可能全表扫描。

可以建索引：

```sql
CREATE INDEX idx_orders_user_id
ON orders(user_id);
```

---

## 2. 是否使用了 SELECT *

只需要部分字段时，不要查所有字段。

错误：

```sql
SELECT *
FROM orders;
```

更好：

```sql
SELECT order_id, user_id, amount
FROM orders;
```

---

## 3. 是否在索引字段上使用函数

假设 `order_time` 有索引。

不推荐：

```sql
WHERE DATE(order_time) = '2026-06-03'
```

因为对字段使用函数，可能导致索引失效。

更推荐：

```sql
WHERE order_time >= '2026-06-03'
  AND order_time < '2026-06-04'
```

---

## 4. LIKE 前置通配符可能无法用索引

```sql
WHERE name LIKE '%张'
```

前面有 `%`，一般不容易利用普通索引。

```sql
WHERE name LIKE '张%'
```

更可能利用索引。

---

## 5. OR 可能影响索引使用

```sql
WHERE major = '数据科学'
   OR age > 20
```

有时优化器不容易高效使用索引，可以考虑改写成 `UNION ALL`，具体要看数据库执行计划。

---

## 6. 使用 EXPLAIN 看执行计划

```sql
EXPLAIN
SELECT *
FROM orders
WHERE user_id = 1001;
```

`EXPLAIN` 可以看数据库如何执行这条 SQL，比如：

- 是否使用索引；
    
- 扫描了多少行；
    
- JOIN 顺序是什么；
    
- 查询类型是否高效。
    

这是 SQL 性能优化的入口。

---

# 三十二、数据库设计中的常见异常

范式之所以重要，是因为低质量表设计会产生异常。

## 1. 插入异常

如果课程信息和选课信息放在一张表中：

|student_id|student_name|course_id|course_name|
|---|---|---|---|

当一门新课还没有学生选时，你可能无法插入这门课的信息。

这就是插入异常。

---

## 2. 删除异常

如果最后一个选某门课的学生退课了，你删除这条记录时，课程信息也被删掉。

这就是删除异常。

---

## 3. 更新异常

如果课程老师改了，表里有 100 条相关记录，你要改 100 次，容易漏改。

这就是更新异常。

范式设计就是为了减少这些异常。

---

# 三十三、数据库完整性

数据库完整性指的是数据必须正确、有效、一致。

主要有三类。

## 1. 实体完整性

每张表的每条记录要能唯一标识。

靠主键保证。

```sql
student_id INT PRIMARY KEY
```

---

## 2. 参照完整性

表与表之间引用关系要正确。

靠外键保证。

```sql
FOREIGN KEY (student_id) REFERENCES students(student_id)
```

---

## 3. 用户定义完整性

业务规则要满足。

例如：

- 年龄不能小于 0；
    
- 成绩在 0 到 100 之间；
    
- 订单金额不能为负。
    

可以用 `CHECK` 实现：

```sql
score INT CHECK (score BETWEEN 0 AND 100)
```

---

# 三十四、SQL 中 NULL 的深入理解

`NULL` 是 SQL 中最容易出错的概念之一。

`NULL` 不是 0，不是空字符串，而是未知值。

例如：

```sql
score = NULL
```

含义不是“分数为 NULL”，而是“分数是否等于未知”，这个结果仍然是未知。

所以要写：

```sql
score IS NULL
```

---

## NULL 和聚合函数

大多数聚合函数会忽略 NULL。

例如：

|score|
|---|
|90|
|NULL|
|80|

```sql
AVG(score)
```

结果是：

```text
(90 + 80) / 2 = 85
```

不是除以 3。

---

## 用 COALESCE 处理 NULL

```sql
SELECT COALESCE(score, 0)
FROM students;
```

含义是：

> 如果 score 是 NULL，就用 0 替代。

在数据分析中很常用。

---

# 三十五、SQL 写作顺序建议

写复杂 SQL 时，不要一上来就写完整。

建议按这个顺序：

1. 先确定主表；
    
2. 再确定需要 JOIN 哪些表；
    
3. 再写连接条件；
    
4. 再写 WHERE 筛选；
    
5. 再写 GROUP BY；
    
6. 再写聚合指标；
    
7. 再写 HAVING；
    
8. 最后写 ORDER BY 和 LIMIT。
    

例如要统计每个用户的订单数和总消费金额：

```sql
SELECT 
    u.user_id,
    u.username,
    COUNT(o.order_id) AS order_count,
    COALESCE(SUM(o.amount), 0) AS total_amount
FROM users u
LEFT JOIN orders o
    ON u.user_id = o.user_id
GROUP BY u.user_id, u.username
ORDER BY total_amount DESC;
```

这条 SQL 的逻辑是：

1. 从用户表出发；
    
2. 左连接订单表；
    
3. 按用户分组；
    
4. 统计订单数和总金额；
    
5. 没有订单的用户金额补 0；
    
6. 按总金额降序排序。

# 三十六、数据分析常用 SQL 模板

## 1. 统计总量

```sql
SELECT COUNT(*) AS total_count
FROM table_name;
```

---

## 2. 按类别统计数量

```sql
SELECT 
    category,
    COUNT(*) AS count
FROM table_name
GROUP BY category
ORDER BY count DESC;
```

---

## 3. 按日期统计趋势

```sql
SELECT 
    DATE(create_time) AS dt,
    COUNT(*) AS count
FROM orders
GROUP BY DATE(create_time)
ORDER BY dt;
```

---

## 4. 用户维度汇总

```sql
SELECT 
    user_id,
    COUNT(*) AS order_count,
    SUM(amount) AS total_amount,
    AVG(amount) AS avg_amount,
    MAX(order_time) AS last_order_time
FROM orders
GROUP BY user_id;
```

---

## 5. 找 Top N

```sql
SELECT 
    product_id,
    SUM(amount) AS total_sales
FROM order_items
GROUP BY product_id
ORDER BY total_sales DESC
LIMIT 10;
```

---

## 6. 找每组 Top N

比如找每个专业分数前三名：

```sql
WITH ranked_students AS (
    SELECT 
        student_id,
        name,
        major,
        score,
        ROW_NUMBER() OVER (
            PARTITION BY major
            ORDER BY score DESC
        ) AS rn
    FROM students
)
SELECT *
FROM ranked_students
WHERE rn <= 3;
```

# 三十七、SQL 面试和考试高频问题

## 1. WHERE 和 HAVING 的区别

`WHERE` 用于分组前筛选原始行，`HAVING` 用于分组后筛选聚合结果。

```sql
WHERE age > 18
GROUP BY major
HAVING AVG(score) > 85
```

---

## 2. INNER JOIN 和 LEFT JOIN 的区别

`INNER JOIN` 只保留两边匹配的数据。

`LEFT JOIN` 保留左表全部数据，右表没有匹配则补 NULL。

---

## 3. 主键和唯一索引的区别

主键：

1. 唯一；
    
2. 非空；
    
3. 一张表只能有一个主键；
    
4. 用来唯一标识记录。
    

唯一索引：

1. 保证唯一；
    
2. 一张表可以有多个唯一索引；
    
3. 是否允许 NULL 要看具体数据库实现。
    

---

## 4. DELETE、TRUNCATE、DROP 的区别

|命令|作用|
|---|---|
|DELETE|删除数据，可加 WHERE|
|TRUNCATE|清空整张表，保留结构|
|DROP|删除整张表，包括结构|

---

## 5. 什么是事务

事务是一组数据库操作，要么全部成功，要么全部失败，具有 ACID 特性。

---

## 6. 什么是索引

索引是帮助数据库快速查找数据的数据结构，类似书的目录。优点是加快查询，缺点是占空间，并且会降低写入和更新速度。

---

## 7. 为什么要做范式化

为了减少数据冗余，避免插入异常、删除异常、更新异常，提高数据一致性。

---

## 8. 什么时候反范式化

有时为了查询性能，会故意保留冗余字段。

例如订单表里保存商品当时的价格，而不是每次都去商品表查当前价格。

因为商品价格会变，订单中应该保留历史成交价。

所以真实业务中不是范式越高越好，而是要平衡：

> 数据一致性、存储冗余、查询性能、业务语义。

# 三十九、一个完整例子：用户订单分析

假设有三张表。

用户表：

```sql
users(
    user_id,
    username,
    register_time,
    city
)
```

订单表：

```sql
orders(
    order_id,
    user_id,
    order_time,
    amount,
    status
)
```

商品明细表：

```sql
order_items(
    item_id,
    order_id,
    product_id,
    quantity,
    price
)
```

---

## 1. 查询每个用户的订单数和消费金额

```sql
SELECT 
    u.user_id,
    u.username,
    COUNT(o.order_id) AS order_count,
    COALESCE(SUM(o.amount), 0) AS total_amount
FROM users u
LEFT JOIN orders o
    ON u.user_id = o.user_id
GROUP BY u.user_id, u.username
ORDER BY total_amount DESC;
```

---

## 2. 查询最近 30 天每天的订单量

```sql
SELECT 
    DATE(order_time) AS dt,
    COUNT(*) AS order_count
FROM orders
WHERE order_time >= CURRENT_DATE - INTERVAL 30 DAY
GROUP BY DATE(order_time)
ORDER BY dt;
```

---

## 3. 查询消费金额前 10 的用户

```sql
SELECT 
    u.user_id,
    u.username,
    SUM(o.amount) AS total_amount
FROM users u
JOIN orders o
    ON u.user_id = o.user_id
WHERE o.status = 'paid'
GROUP BY u.user_id, u.username
ORDER BY total_amount DESC
LIMIT 10;
```

---

## 4. 查询每个城市的用户平均消费

```sql
SELECT 
    u.city,
    AVG(user_total.total_amount) AS avg_user_amount
FROM users u
JOIN (
    SELECT 
        user_id,
        SUM(amount) AS total_amount
    FROM orders
    WHERE status = 'paid'
    GROUP BY user_id
) user_total
    ON u.user_id = user_total.user_id
GROUP BY u.city
ORDER BY avg_user_amount DESC;
```

这里要注意：  
如果直接写：

```sql
SELECT city, AVG(amount)
FROM users u
JOIN orders o ON u.user_id = o.user_id
GROUP BY city;
```

算的是“订单平均金额”，不是“用户平均消费金额”。

这就是数据分析 SQL 中很关键的粒度问题。

# 四十、数据库里最重要的思维：粒度

粒度就是一行数据代表什么。

例如：

用户表：

```text
一行 = 一个用户
```

订单表：

```text
一行 = 一个订单
```

订单明细表：

```text
一行 = 一个订单中的一个商品
```

播放记录表：

```text
一行 = 一次播放行为
```

评论表：

```text
一行 = 一条评论
```

SQL 分析前一定要先问：

> 当前表的一行代表什么？

否则很容易重复计数。

例如订单表和订单明细表 JOIN 后：

一个订单如果有 3 个商品，JOIN 后这个订单会变成 3 行。

如果这时候直接：

```sql
SUM(order_amount)
```

就会把订单金额重复算 3 次。

所以分析前必须确认数据粒度。

这是数据分析中比语法更重要的能力。

# 四十一、你应该优先掌握的 SQL 能力

按实用优先级排序：

1. `SELECT + WHERE`：能筛选数据；
    
2. `GROUP BY + 聚合函数`：能做统计；
    
3. `JOIN`：能合并多表；
    
4. `HAVING`：能筛选分组结果；
    
5. `CASE WHEN`：能做分类变量和条件统计；
    
6. `子查询 / CTE`：能拆解复杂问题；
    
7. `窗口函数`：能做排名、累计、组内比较；
    
8. `索引和 EXPLAIN`：能理解性能；
    
9. `事务和约束`：能理解数据库可靠性；
    
10. `范式和 E-R 设计`：能做数据库建模。
    

补一个非常常用的 `CASE WHEN`：

```sql
SELECT 
    user_id,
    CASE
        WHEN total_amount >= 10000 THEN '高价值用户'
        WHEN total_amount >= 1000 THEN '中价值用户'
        ELSE '低价值用户'
    END AS user_level
FROM user_summary;
```

条件统计也常用：

```sql
SELECT 
    COUNT(*) AS total_orders,
    SUM(CASE WHEN status = 'paid' THEN 1 ELSE 0 END) AS paid_orders,
    SUM(CASE WHEN status = 'cancelled' THEN 1 ELSE 0 END) AS cancelled_orders
FROM orders;
```

> 先从表中取数据，再筛选，再连接，再分组，再聚合，再排序，最后得到分析结果。
