# 一、网页的三大要素：HTML、CSS、JavaScript

你的第一张 PPT 讲的是网页的三大要素，这一页其实非常重要，因为它决定了你后面学每个技术时的定位。

可以把一个网页想象成一座房子：

- **HTML** 决定房子的**结构和内容**
    
- **CSS** 决定房子的**外观和装修**
    
- **JavaScript** 决定房子的**交互和行为**
    

这是最经典的理解方式。

---

## 1. HTML：负责结构和内容

HTML，全称 **HyperText Markup Language**，中文叫**超文本标记语言**。

这里有三个词都要理解：

### （1）HyperText：超文本

“超文本”不是普通文本，它可以包含跳转关系，比如点击一个链接跳到另一个页面。  
这就是网页和普通 txt 文本文件的重要区别之一。

### （2）Markup：标记

HTML 不是“编程语言”意义上的那种语言，它更像一种**标记语言**。  
它通过一对一对的标签（tag）告诉浏览器：

- 这里是标题
    
- 这里是段落
    
- 这里是图片
    
- 这里是链接
    
- 这里是表格
    
- 这里是表单
    

也就是说，HTML 的本质是：**给内容加语义和结构标记**。

### （3）Language：语言

虽然它不擅长计算和逻辑控制，但它有自己的语法规则，所以叫语言。

---

## 2. CSS：负责样式和美观

CSS，全称 **Cascading Style Sheets**，中文叫**层叠样式表**。

它的作用是控制：

- 颜色
    
- 字体
    
- 大小
    
- 间距
    
- 边框
    
- 布局
    
- 动画
    
- 响应式展示
    

HTML 只负责告诉浏览器“这是一段文字”“这是一个标题”，  
CSS 负责决定：

- 标题是红色还是蓝色
    
- 字体多大
    
- 居中还是左对齐
    
- 页面是两栏还是三栏
    

所以可以说：

> HTML 管“是什么”，CSS 管“长什么样”。

---

## 3. JavaScript：负责交互和行为

JavaScript 是真正的编程语言（programming language）。  
它能让网页“动起来”。

比如：

- 点击按钮后弹出提示
    
- 输入表单时实时检查内容
    
- 点击菜单展开或收起
    
- 页面异步加载数据
    
- 修改 HTML 内容
    
- 动态修改 CSS 样式
    
- 响应用户事件（event）
    

所以：

> HTML 提供骨架，CSS 提供外观，JavaScript 提供灵魂和动作。

---

## 4. 三者关系的更准确理解

很多初学者会误以为这三个是完全独立的。其实不是。

它们之间的关系是：

- HTML 创建页面元素
    
- CSS 选中这些元素并设置样式
    
- JavaScript 获取这些元素并修改内容、样式或行为
    

例如一个按钮：

- HTML：定义按钮存在
    
- CSS：让按钮变成蓝色圆角
    
- JavaScript：点击按钮后提交数据
    

所以你要建立一个很重要的前端认知：

> 网页不是某一门技术单独完成的，而是三者协作完成的。

# 二、HTML 到底是什么：不是“会显示文字”这么简单

很多初学者刚学 HTML 时，会觉得 HTML 不就是写点标签吗？  
其实 HTML 的关键不在“显示”，而在于**结构化表达内容**。

---

## 1. HTML 的核心职责：描述文档结构

浏览器看到 HTML 后，不是把它当作普通文本，而是把它解析成一棵树，这棵树叫：

**DOM Tree（Document Object Model Tree，文档对象模型树）**

比如：

```html
<h1>标题</h1>
<p>这是一个段落</p>
<a href="https://example.com">链接</a>
```

浏览器不会只看到三行代码，而会理解成：

- 有一个一级标题节点
    
- 有一个段落节点
    
- 有一个链接节点
    

所以 HTML 的真正本质是：

> 用标签把内容组织成一个有层次的文档结构。

---

## 2. HTML 强调“语义”（semantic）

HTML 不只是“让文字大一点小一点”，更重要的是表达内容的意义。

例如：

- `<h1>` 表示一级标题
    
- `<p>` 表示段落
    
- `<a>` 表示链接
    
- `<table>` 表示表格
    
- `<form>` 表示表单
    
- `<img>` 表示图像
    

这叫 **semantic markup（语义化标记）**。

为什么语义很重要？

### 原因一：方便浏览器理解页面

浏览器能更准确地渲染页面。

### 原因二：方便搜索引擎理解

搜索引擎（如 Google、Bing）会利用 HTML 结构理解页面内容。

### 原因三：方便无障碍访问

屏幕阅读器（screen reader）会依赖语义标签帮助视障用户理解页面。

### 原因四：方便程序员维护

语义清晰的 HTML 更容易读懂和修改。

所以现代前端开发很强调：

> 不要只是“把页面做出来”，而要“用正确的标签表达正确的内容”。

---

# 三、HTML 元素、标签、属性：这三个概念必须分清

这部分是你 PPT 里“HTML 元素”和“HTML 属性”的重点。

---

## 1. 什么是标签（tag）

标签（tag）是 HTML 的基本语法单位。  
通常长这样：

```html
<p>
</p>
```

一个开始标签（start tag）和一个结束标签（end tag）构成一对。

例如：

```html
<h1>这是标题</h1>
```

这里：

- `<h1>` 是开始标签
    
- `</h1>` 是结束标签
    

---

## 2. 什么是元素（element）

元素（element）通常指：

> 从开始标签到结束标签之间的整体

例如：

```html
<p>My first paragraph</p>
```

整个这一段就是一个 **paragraph element（段落元素）**。

所以要注意：

- 标签是“符号”
    
- 元素是“完整结构”
    

简单说：

```html
<p>内容</p>
```

- `<p>`、`</p>` 是标签
    
- `<p>内容</p>` 是元素
    

---

## 3. HTML 元素可以嵌套（nesting）

你的 PPT 提到“HTML 元素允许嵌套”，这很关键。

例如：

```html
<p>这是一个 <a href="https://example.com">链接</a></p>
```

这里：

- 外层是 `<p>` 段落元素
    
- 内层是 `<a>` 链接元素
    

这说明 HTML 是层次化结构，而不是平铺的字符串。

但嵌套必须正确，不能乱套。比如：

```html
<p><b>文字</p></b>
```

这是错误的，因为标签闭合顺序不对。

正确应该是：

```html
<p><b>文字</b></p>
```

这和括号匹配很像。

---

## 4. 空元素（empty element）

有些 HTML 元素没有内容，也不需要结束标签，称为空元素（empty element）。

例如：

```html
<br>
<img src="a.jpg">
```

它们只负责插入某种功能，不包裹文字内容。

常见空元素有：

- `<br>`：换行
    
- `<img>`：图像
    
- `<input>`：输入框
    
- `<hr>`：水平线
    
- `<meta>`
    
- `<link>`
    

---

## 5. 什么是属性（attribute）

属性（attribute）是给 HTML 元素提供附加信息的。

例如：

```html
<a href="https://www.example.com">访问网站</a>
```

这里的 `href` 就是属性名，`https://www.example.com` 是属性值。

一般格式是：

```html
name="value"
```

也就是：

> 属性名 = 属性值

你的 PPT 里给的例子也符合这个格式，比如：

- `href="http://www.w3school.com.cn"`
    
- `bgcolor="yellow"`
    
- `align="right"`
    

---

## 6. 属性通常写在开始标签里

这一点非常基础但必须记牢：

> 属性总是在开始标签中定义，而不是结束标签中。

例如：

```html
<img src="cat.jpg" width="200" height="100">
```

这里 `src`、`width`、`height` 都写在 `<img ...>` 里面。

---

## 7. 标签大小写问题

PPT 里提到 HTML 标签大小写不敏感，像 `<P>` 等同于 `<p>`。  
这在历史上是对的，浏览器通常能兼容。

但是在现代开发里，**强烈建议全部用小写**：

```html
<p>这是段落</p>
```

原因是：

- 风格统一
    
- 更符合 HTML5 规范习惯
    
- 和 XHTML、XML 风格更兼容
    
- 更便于团队协作
    

所以实际开发请记住：

> 现代 HTML 标签、属性名通常都写小写。

---

# 四、HTML 文档的基本结构

你的 PPT 里多次出现这样的代码结构：

```html
<!DOCTYPE html>
<html lang="zh-cn">
<head>
    <title>网页标题</title>
</head>
<body>
    ...
</body>
</html>
```

这就是一个 HTML 文档的基本骨架。

---

## 1. `<!DOCTYPE html>`

这不是 HTML 标签，而是文档类型声明（document type declaration）。

它告诉浏览器：

> 这是一个 HTML5 文档。

如果没有它，浏览器可能进入兼容模式（quirks mode），导致渲染行为异常。

所以现代 HTML 页面几乎都以它开头。

---

## 2. `<html>`

这是整个 HTML 文档的根元素（root element）。

所有其他内容通常都包含在 `<html>` 内。

例如：

```html
<html lang="zh-CN">
...
</html>
```

其中 `lang="zh-CN"` 表示页面语言是中文，这对搜索引擎和辅助工具很有帮助。

---

## 3. `<head>`

`<head>` 不是页面上直接显示的内容区域。  
它主要放的是文档的**元信息（metadata）**，例如：

- 页面标题 `<title>`
    
- 字符编码 `<meta charset="UTF-8">`
    
- 样式表 `<link>`
    
- 内部 CSS `<style>`
    
- JavaScript `<script>`
    
- 搜索引擎描述信息
    

你可以把它理解成页面的“配置区”。

---

## 4. `<title>`

`<title>` 定义网页标题，显示在：

- 浏览器标签页
    
- 收藏夹标题
    
- 搜索引擎结果标题中
    

例如：

```html
<title>我的第一个网页</title>
```

它非常重要，既影响用户体验，也影响 SEO（搜索引擎优化）。

---

## 5. `<body>`

`<body>` 里的内容才是网页可见的主体内容。  
比如：

- 标题
    
- 段落
    
- 图片
    
- 链接
    
- 表单
    
- 表格
    

都放在 `<body>` 中。

你看到的绝大多数页面内容，本质上都是 body 中元素渲染出来的结果。

---

# 五、标题、段落、换行：最基础但最常用的文本结构

你的 PPT 有一页专门讲“HTML 标题与段落”。

---

## 1. 标题：`<h1>` 到 `<h6>`

HTML 提供了六级标题：

- `<h1>`
    
- `<h2>`
    
- `<h3>`
    
- `<h4>`
    
- `<h5>`
    
- `<h6>`
    

其中：

- `<h1>` 最大，层级最高
    
- `<h6>` 最小，层级最低
    

例如：

```html
<h1>主标题</h1>
<h2>章节标题</h2>
<h3>小节标题</h3>
```

---

## 2. 标题不仅是“大字”，更是“层级结构”

初学者最容易犯的错误是把标题当成“调大字体的工具”。

这不对。

标题真正的作用是表示文档结构层级。

例如一篇文章：

- `<h1>`：整篇文章标题
    
- `<h2>`：第一章、第二章
    
- `<h3>`：章内的小节
    
- `<h4>`：更细的子部分
    

所以标题的本质不是视觉，而是结构与语义。

### 一个重要实践原则

一般一个页面最好只有一个核心 `<h1>`，表示页面主主题。  
下面再按逻辑使用 `<h2>`、`<h3>` 递进。

---

## 3. 段落：`<p>`

段落用 `<p>` 定义：

```html
<p>这是一个段落。</p>
<p>这是另一个段落。</p>
```

浏览器会自动在段落之间留出一定空白。

注意：

> `<p>` 表示的是“独立的段落”，不是随便包一行文字。

如果一段文字语义上是完整段落，就用 `<p>`。

---

## 4. 换行：`<br>`

如果你不想新起一个段落，只想在同一段内换行，可以使用 `<br>`：

```html
<p>这是第一行<br>这是第二行</p>
```

这里要理解：

- `<p>` 是段落分隔
    
- `<br>` 是行内换行
    

二者语义不同，不能混用。

### 常见误区

不要为了增加空白连续写很多 `<br><br><br>`。  
页面间距应该尽量交给 CSS 控制，而不是靠 HTML 换行堆出来。

---

# 六、HTML 链接：网页“超文本”特性的核心

如果没有链接，网页就很难称为“Web”。

---

## 1. 链接元素：`<a>`

HTML 中链接用锚元素（anchor element）定义：

```html
<a href="https://www.example.com">访问网站</a>
```

这里：

- `<a>` 是链接标签
    
- `href` 是最核心的属性
    
- 标签中间的文字是用户点击时看到的内容
    

---

## 2. `href` 的作用

`href` 全称通常理解为 **hypertext reference**，表示链接目标地址。

它可以指向：

### （1）外部网页

```html
<a href="https://www.example.com">外部网站</a>
```

### （2）站内页面

```html
<a href="about.html">关于我们</a>
```

### （3）页面中的某个位置

```html
<a href="#tips">跳到提示部分</a>
```

### （4）邮箱、电话等

```html
<a href="mailto:test@example.com">发邮件</a>
<a href="tel:123456789">拨号</a>
```

---

## 3. 书签 / 锚点跳转

你 PPT 里讲了通过 `name` 属性创建书签。  
这是早期写法，例如：

```html
<a name="tips">基本的注意事项</a>
<a href="#tips">有用的提示</a>
```

但在现代 HTML 中，更常用的是给元素设置 `id`：

```html
<h2 id="tips">有用的提示</h2>
<a href="#tips">跳转到提示</a>
```

也就是说，现代更推荐：

> 用 `id` 作为页面内跳转锚点，而不是 `name`

因为 `id` 更通用，也能被 CSS 和 JavaScript 使用。

---

## 4. 链接文本要有意义

这是一个很实用的前端规范：

尽量不要写“点击这里”“点我”。

更好的是：

```html
<a href="guide.html">查看 HTML 入门指南</a>
```

这样用户和搜索引擎都更容易理解链接内容。

---

# 七、HTML 图像：`<img>` 不是“插图那么简单”

你的 PPT 讲到图像 `<img>` 标签时提到它是空标签，这一点是对的。

---

## 1. 图像标签：`<img>`

基本写法：

```html
<img src="cat.jpg" alt="一只猫">
```

其中最重要的两个属性是：

- `src`：图片地址
    
- `alt`：替代文本
    

---

## 2. `src`：图片来源

`src` 表示图片文件位置：

```html
<img src="images/logo.png">
```

浏览器会去这个路径加载图片。

---

## 3. `alt`：非常重要

PPT 里更多强调了 `align`，但在现代 HTML 中，`alt` 比 `align` 重要得多。

`alt` 是替代文本（alternative text），作用包括：

### （1）图片加载失败时显示说明

### （2）给屏幕阅读器读取

### （3）有利于搜索引擎理解图片内容

例如：

```html
<img src="campus.jpg" alt="上海财经大学校园主楼">
```

如果你以后写正式网页，没有 `alt` 往往会被认为是不规范的。

---

## 4. 关于 `align` 属性

你 PPT 里展示了：

```html
<img src="..." align="bottom">
<img src="..." align="middle">
<img src="..." align="top">
```

这在早期 HTML 中确实常见，但现在属于**过时写法（deprecated）**。  
现代开发中，图片对齐通常用 CSS 来做，而不是靠 HTML 的 `align` 属性。

例如：

```html
<img src="cat.jpg" style="vertical-align: middle;">
```

或者用更规范的 CSS 文件控制。

所以你要知道：

- PPT 里的写法有历史意义，帮助理解
    
- 但现代开发更推荐用 CSS 管理样式
    

---

# 八、HTML 表格：用来展示“表格数据”，不是用来做页面布局

你的 PPT 有一页“HTML 表格”。

---

## 1. 表格的基本结构

表格由这些主要元素组成：

- `<table>`：表格整体
    
- `<tr>`：表格行（table row）
    
- `<td>`：单元格（table data）
    
- `<th>`：表头单元格（table header）
    

例如：

```html
<table>
  <tr>
    <th>姓名</th>
    <th>年龄</th>
  </tr>
  <tr>
    <td>张三</td>
    <td>20</td>
  </tr>
</table>
```

---

## 2. 如何理解表格结构

你可以这样想：

- `<table>` 是整张表
    
- 每个 `<tr>` 是一行
    
- 每行里有若干 `<td>` 或 `<th>`
    

所以表格本质上是二维数据结构在网页中的表示。

---

## 3. 表格适合什么场景

表格适合展示规则数据，比如：

- 课程表
    
- 成绩表
    
- 财务报表
    
- 商品清单
    
- 数据统计结果
    

不适合用来做整个页面排版。

---

## 4. 为什么现代网页不用表格布局

早期网页确实经常用 `<table>` 做整体布局，因为那时候 CSS 布局能力很弱。  
但现在已经不推荐了。

原因：

- 语义不正确
    
- 代码难维护
    
- 响应式适配差
    
- 不利于无障碍访问
    

现代布局一般使用：

- Flexbox
    
- Grid
    
- 语义化结构标签 + CSS
    

所以请记住一句非常重要的话：

> 表格用于展示数据，不用于页面布局。

# 九、HTML 表单：网页和用户“交互输入”的入口

你 PPT 后面很多页都在讲 HTML 表单（form），这是非常重要的一块，因为它是网页收集用户输入的核心方式。

---

## 1. 什么是表单（form）

表单用于收集用户输入（collect user input）。

例如：

- 登录
    
- 注册
    
- 搜索
    
- 提交问卷
    
- 填写联系方式
    
- 上传信息
    

HTML 用 `<form>` 定义表单区域。

---

## 2. `<form>` 的作用

例如：

```html
<form action="/submit" method="post">
  ...
</form>
```

它相当于一个提交容器，里面放各种输入控件。  
当用户点击提交按钮时，表单中的数据会被发送到服务器。

---

## 3. `action` 属性

`action` 指定表单提交到哪里。

例如：

```html
<form action="/login">
```

表示提交到 `/login` 这个地址。

你可以理解为：

> action 决定“数据送去哪里”。

---

## 4. `method` 属性：GET 和 POST

这是表单非常重要的知识点。

### （1）GET

GET 会把表单数据拼接到 URL 后面。

例如：

```html
/search?keyword=html
```

特点：

- 参数可见
    
- 适合查询、搜索
    
- 不适合敏感信息
    
- 长度有限制
    

### （2）POST

POST 会把数据放在请求体（request body）中。

特点：

- 参数通常不直接显示在地址栏
    
- 适合提交敏感或较多数据
    
- 常用于注册、登录、表单提交
    

你的 PPT 写的是：

- Get：网页地址可见
    
- Post：网页地址不可见
    

这个说法作为入门可以接受，但更准确一点应该是：

> GET 的参数通常出现在 URL 中，POST 的数据通常在请求体中，不直接显示在地址栏。

注意：“不可见”不等于“绝对安全”，只是**不显示在地址栏**。真正安全还要看 HTTPS、后端处理等。

---

## 5. `target` 属性

PPT 里也讲了 `target`，它控制提交结果显示的位置。

常见值：

- `_self`：当前窗口
    
- `_blank`：新窗口或新标签页
    
- `_parent`
    
- `_top`
    

最常见的是：

```html
<form target="_blank">
```

表示提交后在新页面打开结果。

---

# 十、表单中的常见输入控件：`<input>`

PPT 里多页都在讲 `<input>`，因为它是最核心的表单元素之一。

---

## 1. 文本框：`type="text"`

```html
<input type="text" name="username">
```

用于单行文本输入，比如：

- 用户名
    
- 姓名
    
- 学号
    
- 关键词搜索
    

---

## 2. 密码框：`type="password"`

```html
<input type="password" name="psw">
```

输入内容会被掩码显示（星号或圆点）。

这只是“显示上隐藏”，并不代表数据天然安全。真正安全还依赖：

- HTTPS
    
- 后端正确处理
    
- 不明文存储密码
    

---

## 3. 单选按钮：`type="radio"`

```html
<input type="radio" name="sex" value="male">Male
<input type="radio" name="sex" value="female">Female
```

单选按钮的关键点是：

> 同一组单选项必须有相同的 `name`

这样浏览器才知道它们属于同一组选项，只能选一个。

---

## 4. 复选框：`type="checkbox"`

```html
<input type="checkbox" name="hobby" value="music">音乐
<input type="checkbox" name="hobby" value="sports">运动
```

复选框允许多选。

---

## 5. 提交按钮：`type="submit"`

```html
<input type="submit" value="Submit">
```

点击后会提交整个表单。

---

## 6. `name` 属性为什么重要

很多初学者只关注页面显示，却忽略 `name`。

实际上在表单提交时，真正提交给服务器的通常是：

```text
name=value
```

例如：

```html
<input type="text" name="firstname" value="Mickey">
```

提交时数据类似于：

```text
firstname=Mickey
```

所以：

> `name` 是数据字段名，后端通常根据它取值。

---

## 7. `value` 属性

`value` 可以表示：

- 文本框的默认值
    
- 单选框/复选框提交时对应的值
    
- 按钮上显示的文字
    

例如：

```html
<input type="text" value="默认内容">
<input type="submit" value="提交">
```

---

# 十一、下拉框、文本域：表单不只有 input

你的 PPT 还讲了 `<select>`、`<option>`、`<textarea>`、`<output>`。

---

## 1. 下拉框：`<select>` 和 `<option>`

例如：

```html
<select name="cars">
  <option value="volvo">Volvo</option>
  <option value="saab">Saab</option>
  <option value="fiat">Fiat</option>
</select>
```

这里：

- `<select>` 定义下拉列表
    
- `<option>` 定义一个具体选项
    

适用于：

- 选省份
    
- 选学院
    
- 选车型
    
- 选年级
    

---

## 2. 文本域：`<textarea>`

用于多行文本输入：

```html
<textarea name="message" rows="10" cols="30">
The cat was playing in the garden.
</textarea>
```

适用于：

- 留言
    
- 反馈
    
- 备注
    
- 评论
    
- 自我介绍
    

它和 `input type="text"` 的区别是：

- `input text`：单行
    
- `textarea`：多行
    

---

## 3. `<output>`

PPT 提到了 `<output>` 用于计算结果输出。  
它在现代 HTML 中确实存在，但日常初学阶段用得不多。

一般用于表单计算结果展示，例如配合 JavaScript 做即时计算。

---

# 十二、style 属性：HTML 也能写样式，但这只是“内联样式”

你的 PPT 讲了“HTML 样式”和 `style` 属性。

---

## 1. `style` 属性是什么

`style` 是 HTML 的一个属性，用来直接给某个元素写 CSS 样式。

例如：

```html
<p style="color:red; font-size:20px;">这是一个段落</p>
```

这里本质上已经不是 HTML 在定义样式，而是在 HTML 标签里嵌入了一段 CSS。

---

## 2. style 的作用

它可以控制：

- 文字颜色
    
- 字号
    
- 背景色
    
- 对齐方式
    
- 边框
    
- 间距等
    

例如：

```html
<h2 style="background-color:red">This is a heading</h2>
<p style="background-color:green">This is a paragraph.</p>
```

---

## 3. 为什么说 `style` 只是“过渡写法”

虽然 `style` 很方便，但现代开发中通常不推荐大量使用内联样式（inline style）。

因为这样会导致：

- HTML 和样式混在一起
    
- 代码难维护
    
- 样式难复用
    
- 页面风格不统一
    

例如如果你有 100 个标题都想改颜色，  
写 100 次 `style="color:red"` 很麻烦。

所以更好的方式是用独立 CSS：

```html
<style>
h1 {
  color: red;
}
</style>
```

或者单独写到 `.css` 文件中。

---

# 十三、CSS 真正的意义：把“样式”从 HTML 中分离出来

这也是你的 PPT 从“HTML 样式”过渡到“CSS”的目的。

---

## 1. CSS 为什么出现

早期网页中，很多样式直接写在 HTML 里，比如：

- `bgcolor`
    
- `align`
    
- `color`
    
- `font`
    

这样会带来一个严重问题：

> HTML 既要描述结构，又要描述样式，职责混乱。

CSS 的出现就是为了把“表现”（presentation）从“内容结构”（structure）中分离出来。

---

## 2. CSS 的优点

### （1）集中管理样式

一个规则可以影响很多元素。

### （2）易维护

改一处，多个页面都更新。

### （3）代码更清晰

HTML 只写结构，CSS 只写样式。

### （4）更适合复杂布局

比如 Flexbox、Grid。

---

## 3. CSS 的基本语法

例如：

```css
body {
  background-color: lightblue;
}

h1 {
  color: white;
  text-align: center;
}

p {
  font-family: verdana;
  font-size: 20px;
}
```

它的结构通常是：

```css
选择器 {
  属性: 值;
}
```

例如：

- `h1` 是选择器（selector）
    
- `color` 是属性（property）
    
- `white` 是值（value）
    

---

# 十四、CSS 选择器：PPT 中的 id 选择器

你最后一页 PPT 讲的是 **id selector（id 选择器）**。

---

## 1. `id` 是什么

HTML 元素可以有 `id` 属性：

```html
<p id="para1">Hello World!</p>
```

`id` 用于唯一标识一个元素。  
在同一页面中，理论上一个 id 只能出现一次。

---

## 2. CSS 如何选中 id

写法是：

```css
#para1 {
  text-align: center;
  color: red;
}
```

这里 `#para1` 的意思就是：

> 选中 id 为 para1 的元素

所以：

- `#` 表示 id 选择器
    
- `para1` 是元素 id 的值
    

---

## 3. 为什么 id 必须唯一

因为它常用于：

- 精确 CSS 样式控制
    
- JavaScript 获取元素
    
- 页面内跳转锚点
    

如果重复使用，会引起歧义。

---

## 4. id 和 class 的区别

虽然你的 PPT 还没讲到 `class`，但这里应该顺手补充一下，因为这对系统理解很重要。

### `id`

- 唯一
    
- 用 `#id名` 选中
    
- 一个页面中通常只出现一次
    

### `class`

- 可重复
    
- 用 `.class名` 选中
    
- 常用于一组相似元素
    

例如：

```html
<p class="warning">警告1</p>
<p class="warning">警告2</p>
```

```css
.warning {
  color: red;
}
```

实际开发中：

- `id` 用于唯一元素
    
- `class` 用于复用样式
    

而且 **class 比 id 更常用**。

给这些元素添加：

# 十八、一个更本质的问题：HTML 是不是编程语言？

这是很多老师和初学者常讨论的问题。

严格来说：

> HTML 不是传统意义上的编程语言（programming language）

原因是它通常不具备：

- 变量
    
- 条件分支
    
- 循环
    
- 函数
    
- 算法逻辑
    

它更准确地说是：

> 标记语言（markup language）

但这不代表 HTML 不重要。  
恰恰相反，HTML 是网页世界的基础协议之一。没有 HTML，浏览器就难以理解页面结构。

所以可以这样说：

- JavaScript 负责“算”和“控制”
    
- HTML 负责“描述和组织”

# 一、先建立总框架：什么叫 TCP/IP 协议族

你的第一页 PPT 写的是：

> 互联网 TCP/IP 协议：是一系列协议的总称

这句话非常关键。  
这里的意思不是“TCP/IP”只指两个协议 TCP 和 IP，而是说：

> **TCP/IP 协议族（protocol suite）** 是互联网运行所依赖的一整套协议体系。

也就是说：

- IP 只是其中一个协议
    
- TCP 是其中一个协议
    
- UDP、HTTP、DNS、ARP、ICMP 等也都在这套体系里
    

所以你不能把 TCP/IP 理解成“只有 TCP + IP”，而应该理解成：

> 它是互联网通信的整体规则系统。

---

# 二、为什么网络协议必须“分层”

这是理解整部分内容的核心。

现实中，计算机之间通信非常复杂，要解决很多问题：

- 数据怎么表示？
    
- 应该发给哪台机器？
    
- 应该发给那台机器上的哪个程序？
    
- 数据丢了怎么办？
    
- 数据乱序怎么办？
    
- 用户输入的网址怎么变成服务器地址？
    
- 浏览器和服务器怎么定义“请求”和“响应”？
    

如果把这些问题全塞进一个大协议里，会非常混乱。  
所以网络设计采用了一个非常重要的思想：

> **分层（layering）**

即：把整个通信过程拆成若干层，每层只负责自己那一部分工作。

---

## 1. 分层的直觉理解

你可以把寄快递想成网络通信：

- 你写信的内容是什么 —— 类似应用层
    
- 用什么格式打包 —— 类似传输层/网络层的数据格式
    
- 收件人住哪条街哪栋楼 —— 类似 IP 地址
    
- 楼里具体哪一户 —— 类似端口
    
- 中途由哪些物流站转运 —— 类似路由
    
- 对方收到后怎么拆包 —— 类似解封装
    

分层的好处是：

- 每层职责清楚
    
- 便于替换和扩展
    
- 上层不用过多关心下层细节
    
- 便于标准化和互操作
    

---

# 三、OSI 七层和 TCP/IP 分层：你 PPT 里两套图怎么理解

第一页左边的图把 **OSI 七层模型** 和 **TCP/IP 分层** 放在了一起。  
这很常见，因为很多教材会同时讲这两个体系。

---

## 1. OSI 七层模型（理论模型）

OSI 七层从上到下是：

1. 应用层（Application Layer）
    
2. 表示层（Presentation Layer）
    
3. 会话层（Session Layer）
    
4. 传输层（Transport Layer）
    
5. 网络层（Network Layer）
    
6. 数据链路层（Data Link Layer）
    
7. 物理层（Physical Layer）
    

这个模型更偏**教学和理论**，方便把网络通信的各种功能分得很细。

---

## 2. TCP/IP 模型（实际互联网体系）

实际互联网更常用 TCP/IP 模型。教材版本略有差异，常见有四层或五层说法。

### 常见四层说法

1. 应用层
    
2. 传输层
    
3. 网际层 / 网络层
    
4. 网络接口层
    

### 常见五层说法

1. 应用层
    
2. 传输层
    
3. 网络层
    
4. 数据链路层
    
5. 物理层
    

你 PPT 的图更接近“OSI 七层 + TCP/IP 对照”的展示方式。

---

## 3. 你在学习中怎么用这两套模型

对于你现在学 Web 和互联网基础，最重要的是掌握这个简化版本：

- **应用层**：HTTP、DNS、SMTP、FTP……
    
- **传输层**：TCP、UDP
    
- **网络层**：IP、ICMP、ARP（严格说 ARP 常放在链路/网络边界）
    
- **链路层**：以太网、Wi-Fi 等
    
- **物理层**：网线、电信号、无线电波
    

只要这个框架清楚，后面内容就很好串起来。

---

# 四、第二个核心思想：封装（encapsulation）与解封装（decapsulation）

你第一页右边的图其实就是在讲这个。

---

## 1. 什么叫封装

发送端发送数据时，不是直接把“应用数据”丢出去，而是会一层一层加头部（header）。

例如浏览器要发送 HTTP 请求时：

### 第一步：应用层

先形成 HTTP 数据

### 第二步：传输层

TCP 给它加一个 TCP 头部  
于是变成：**TCP 段（segment）**

### 第三步：网络层

IP 再加一个 IP 头部  
于是变成：**IP 数据报（datagram / packet）**

### 第四步：链路层

以太网再加帧头帧尾  
于是变成：**帧（frame）**

然后才真正通过网卡、网线或无线网络发出去。

这就叫：

> **封装**：每一层都把上一层数据当作“载荷（payload）”，再加上自己的控制信息。

---

## 2. 什么叫解封装

接收端收到后会反过来：

- 链路层拆掉帧头帧尾
    
- 网络层拆掉 IP 头
    
- 传输层拆掉 TCP / UDP 头
    
- 应用层得到真正的 HTTP 数据
    

这叫：

> **解封装（decapsulation）**

---

## 3. 为什么封装很重要

因为每一层要解决的问题不同，它必须在头部里写入自己需要的信息。

例如：

- IP 头部要写源 IP、目标 IP
    
- TCP 头部要写源端口、目标端口、序号、确认号
    
- HTTP 头部要写 Host、User-Agent、Accept 等
    

所以你以后看到“包头”“请求头”“响应头”，本质上都和这个思想有关。

---

# 五、网络层协议：这一层到底负责什么

你的第二页 PPT 列出了网络层常见协议：

- IP
    
- ARP
    
- ICMP
    
- NAT
    
- RIP
    
- OSPF
    
- BGP
    

这些协议名字很多，初学者容易一下子乱掉。其实只要先抓住网络层的核心任务就行：

> **网络层的核心职责是：把数据从一台主机送到另一台主机。**

换句话说，网络层关心的是：

- 目标机器是谁
    
- 该往哪个方向转发
    
- 中途怎么经过多个路由器到达对方
    

---

## 1. IP：负责“寻址 + 转发”

IP 是网络层最核心的协议。  
它解决的是：

- 数据该发给哪台主机
    
- 数据如何跨网络传递
    

这就是为什么 IP 地址在网络层这么关键。

---

## 2. ARP：负责“已知 IP，找 MAC”

ARP（Address Resolution Protocol，地址解析协议）主要用于局域网内，把 IP 地址解析成 MAC 地址。

因为在局域网真正发帧时，网卡层面需要知道对方的硬件地址（MAC 地址），而上层往往只知道 IP 地址。

所以 ARP 可以理解成：

> **IP 到 MAC 的翻译器**

---

## 3. ICMP：负责差错和控制信息

ICMP（Internet Control Message Protocol，互联网控制报文协议）常用于：

- 报告错误
    
- 反馈网络状态
    
- 辅助诊断网络问题
    

最经典的例子就是：

- `ping` 命令背后常用 ICMP Echo Request / Echo Reply
    
- `traceroute` 也会利用 ICMP 相关机制
    

所以 ICMP 不是用来传输正常业务数据的，而是网络层的“系统提示信息”。

---

## 4. NAT：网络地址转换

NAT（Network Address Translation）严格来说更偏路由器功能机制，但通常也放在网络层相关讨论里。

它的作用是：

> 把内网私有地址转换成公网地址

例如你家里很多设备：

- 手机
    
- 电脑
    
- 平板
    

都可以用私有 IP 上网，但对外通常共享同一个公网 IP，这背后就常有 NAT。

---

## 5. RIP / OSPF / BGP：路由协议

这些主要用于“路由器之间如何决定路径”。

### RIP

较早、较简单，按跳数选路

### OSPF

开放最短路径优先协议，常见于自治系统内部

### BGP

边界网关协议，用于大型网络/自治系统之间交换路由信息，是互联网骨干路由的关键协议

你现在入门阶段只要知道：

> 它们都是“决定数据该怎么走”的协议。

不必一开始深究算法细节。

---

# 六、IP 协议：互联网寻址的核心

你的第三页 PPT 专门讲 IP 协议，这一页是重点。

---

# 七、IP 地址：网络世界中的“机器地址”

PPT 说：

- IPv4 是 32 位地址
    
- IPv6 是 128 位地址
    

这是最基本的定义。

---

## 1. IPv4

IPv4 地址本质上是一个 **32 位二进制数**。  
为了方便人读，通常写成四段十进制数，用点分隔：

```text
192.168.0.1
```

其实它底层还是二进制，只是我们平常看到的是点分十进制表示法（dotted decimal notation）。

例如：

```text
192.168.0.1
```

对应四个字节：

- 192
    
- 168
    
- 0
    
- 1
    

每段范围是 0 到 255。

---

## 2. IPv6

IPv6 是 **128 位地址**，远比 IPv4 长很多。  
例如：

```text
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

它出现的主要原因之一是：

> IPv4 地址不够用了。

因为互联网设备越来越多，32 位地址空间有限。

---

## 3. IP 地址的本质作用

IP 地址不是“给程序用的名字”，而是：

> **在网络层标识一台主机或网络接口的位置**

所以以后你要区分：

- 域名（domain name）：给人记的，比如 `www.baidu.com`
    
- IP 地址：给网络路由和寻址用的，比如 `110.242.68.66`
    

域名最终通常要通过 DNS 解析成 IP 才能通信。

---

# 八、IP 包：为什么网络传输要“分块”

PPT 中提到：

> 网络传输数据被分割成小块，打包成 IP 包发送

这很重要。

---

## 1. 为什么不能一次传一个超大数据

因为网络并不是为单一通信服务的，很多设备、很多应用都在同时使用网络。  
如果一次传一个巨大的整体，效率和管理都会很差。

所以通常需要把数据切成若干小块逐步发送。

---

## 2. IP 包的特点：尽力而为（best effort）

PPT 里说：

> 不保证到达，也不保证顺序到达

这是理解 IP 协议的关键。

IP 协议本身提供的是一种：

> **尽力而为（best-effort）的无连接数据报服务**

意思是：

- 它尽量帮你送
    
- 但不承诺一定送到
    
- 不承诺按原顺序到达
    
- 不承诺不会重复
    
- 不承诺中途不丢失
    

这说明：

> **IP 只负责“发往目标方向”，不负责“可靠送达”**

那“可靠性”谁负责？  
通常由上面的 **TCP** 来负责。

---

## 3. 为什么同一批数据可能走不同路径

互联网是一个复杂网络，数据从 A 到 B 可能经过很多路由器。  
不同的数据包在网络拥塞、路由变化等情况下，可能走不同路径。

所以同一个文件的几个数据包：

- 包 1 走甲路线
    
- 包 2 走乙路线
    
- 包 3 走丙路线
    

并不是不可能。

这就是为什么：

- 可能乱序
    
- 可能丢失
    
- 可能延迟不同
    

---

# 九、端口：为什么一台机器能同时运行很多网络程序

你的 PPT 在 IP 页里又提到了“端口”，虽然从严格分层来说端口更偏传输层概念，但这里一起讲很自然。

---

## 1. IP 地址只定位“机器”，端口定位“程序”

这一点一定要记牢：

- **IP 地址**：定位到哪台主机
    
- **端口号（port）**：定位到主机上的哪个应用程序
    

因为一台服务器上可能同时运行很多服务：

- Web 服务
    
- 数据库服务
    
- SSH 服务
    
- 邮件服务
    

如果只有 IP，没有端口，那系统不知道数据该交给哪个程序。

所以你可以把“IP + 端口”理解成：

> “一栋楼 + 房间号”

- IP = 大楼地址
    
- 端口 = 房间号
    

---

## 2. 典型端口例子

例如：

- HTTP 默认端口：80
    
- HTTPS 默认端口：443
    
- SSH 默认端口：22
    
- DNS 常用端口：53
    

所以：

```text
http://example.com:8080
```

这里 `8080` 就是端口号。

---

## 3. 源端口和目标端口

PPT 里写：

> IP 包包含源 IP 地址、目标 IP 地址、源端口和目标端口

从更严格的角度说，端口字段通常在 TCP/UDP 头部中，不是 IP 头部本身字段。  
但你可以先把整体理解为“通信双方既要知道对方机器，也要知道对方程序”。

例如浏览器访问服务器时：

- 目标 IP：服务器地址
    
- 目标端口：通常 80 或 443
    
- 源 IP：你的机器地址
    
- 源端口：你本地临时分配的一个端口
    

---

# 十、传输层协议：TCP 和 UDP 到底负责什么

你的第四页和第五页都在讲传输层。

传输层的核心任务可以概括为：

> **在主机到主机通信之上，进一步提供进程到进程的通信能力。**

更通俗地说：

- 网络层解决“发到哪台机器”
    
- 传输层解决“交给那台机器上的哪个程序，并决定如何传”
    

传输层里最重要的两个协议就是：

- TCP
    
- UDP
    

---

# 十一、TCP 和 UDP 的总体区别

你的 PPT 写了：

- TCP 负责建立可靠连接，保证数据按顺序到达，流式通信
    
- UDP 是无连接，不保证顺序到达，报文式通信
    

这几句话是正确的，但我们需要把它彻底讲透。

---

## 1. TCP：面向连接（connection-oriented）

TCP 通信前通常要先建立连接。  
这意味着双方先确认：

- 我准备好了
    
- 你准备好了
    
- 那我们开始传吧
    

建立连接后，再正式传输数据。

### 面向连接的含义

它不是说真的有一根“物理专线”，而是说双方在协议层维护一种逻辑上的连接状态。

---

## 2. UDP：无连接（connectionless）

UDP 不需要提前建立连接。  
应用要发数据时，直接把一个 UDP 数据报发出去即可。

所以 UDP 更像：

- 直接投递一封信
    
- 不先打招呼，不建立会话状态
    

---

## 3. TCP：可靠（reliable）

PPT 说 TCP 是可靠的。更准确说法是：

> TCP 提供可靠传输服务。

它通过多种机制尽量保证：

- 数据不丢
    
- 数据按顺序交付
    
- 重复数据被识别
    
- 流量不把对方压垮
    
- 网络拥塞时做调整
    

这背后涉及：

- 序号（sequence number）
    
- 确认应答（ACK）
    
- 超时重传（retransmission）
    
- 滑动窗口（sliding window）
    
- 流量控制（flow control）
    
- 拥塞控制（congestion control）
    

你当前阶段先重点理解前面三个。

---

## 4. UDP：不保证可靠

UDP 不负责：

- 丢包重传
    
- 顺序保证
    
- 连接状态维护
    

所以它的特点是：

- 简单
    
- 开销小
    
- 延迟低
    
- 适合实时性场景
    

但“UDP 不可靠”不等于“UDP 一定很差”，而是：

> 它把可靠性责任交给应用自己决定是否处理。

---

# 十二、TCP 三次握手：为什么要握三次

你的第五页 PPT 提到了：

> TCP 通信之前需要进行三次握手

这是网络基础里非常经典的知识点。

---

## 1. 三次握手是什么

通常过程是：

### 第一次

客户端发送 SYN  
表示：我想建立连接

### 第二次

服务器回复 SYN + ACK  
表示：我收到了你的请求，我也同意建立连接

### 第三次

客户端再发 ACK  
表示：我也收到了你的确认，那连接建立成功

---

## 2. 为什么不是两次

根本原因在于：

> 双方都要确认“我能发、也能收，对方也能发、也能收”。

三次握手的目标是建立**双向通信的确认**。

简单说：

- 第一次：客户端告诉服务器“我能发”
    
- 第二次：服务器告诉客户端“我收到了，而且我也能发”
    
- 第三次：客户端告诉服务器“我知道你也准备好了”
    

这样双方都能确认连接状态。

---

## 3. 三次握手的本质

不要把它只当成考试流程图。  
本质上它是在做两件事：

- 协商建立连接状态
    
- 同步初始序列号等关键控制信息
    

所以它是 TCP 可靠通信的起点。

---

# 十三、TCP 为什么可靠：直观理解

---

## 1. 序号机制

TCP 会给发送的数据编号。  
这样接收方收到后就知道：

- 哪段先到
    
- 哪段后到
    
- 哪段缺了
    

这可以解决乱序和缺失问题。

---

## 2. ACK 确认

接收方收到数据后会回复确认。  
发送方知道“对方确实收到了”。

---

## 3. 超时重传

如果发送方等了一段时间还没收到确认，就会认为可能丢了，然后重发。

---

## 4. 按序重组

即使中间网络导致包顺序乱了，TCP 也能按照序号重新排序，再交给上层应用。

所以浏览器最终看到的网页内容通常是有序完整的。

---

# 十四、UDP 为什么仍然很有用

很多初学者一看到“TCP 可靠，UDP 不可靠”，就以为 TCP 一定更高级，UDP 只是简化版。  
其实不是。

UDP 很有价值，因为它：

- 头部小
    
- 开销低
    
- 不需要建立连接
    
- 时延通常更低
    

适合场景例如：

- 实时音视频
    
- 在线游戏
    
- 语音通话
    
- DNS 查询
    
- 一些直播传输
    
- 某些监控与广播场景
    

为什么这些场景常用 UDP？

因为它们更在意：

> **快**，有时比“绝对完整”更重要。

例如视频通话中，某一帧丢了，通常宁可跳过去，也不要停下来等重传。  
否则画面会卡顿得很厉害。

---

# 十五、TCP 头部和 UDP 头部：为什么说 TCP 开销更大

你的第六页 PPT 说：

> TCP 头部开销大，UDP 头部开销小

这句话背后的原因是：TCP 要承担更多功能。

---

## 1. TCP 头部为什么复杂

TCP 头部要包含很多字段，例如：

- 源端口
    
- 目标端口
    
- 序号
    
- 确认号
    
- 首部长度
    
- 标志位（SYN、ACK、FIN 等）
    
- 窗口大小
    
- 校验和
    
- 紧急指针
    
- 可选项
    

因为它要管理：

- 连接建立与关闭
    
- 可靠传输
    
- 顺序控制
    
- 流量控制
    

所以头部自然更大。

---

## 2. UDP 头部为什么简单

UDP 头部通常只包含：

- 源端口
    
- 目标端口
    
- 长度
    
- 校验和
    

非常精简。

所以 UDP 的协议开销较低，更适合轻量快速传输。

# 十七、HTTP 和 HTTPS：Web 世界最关键的应用层协议

你的第八页 PPT 进入了 Web 核心内容。

---

## 1. HTTP 是什么

HTTP 全称 **HyperText Transfer Protocol**，中文叫**超文本传输协议**。

它是浏览器和 Web 服务器之间交流的规则。

也就是说，当你在浏览器地址栏输入网址，浏览器和服务器之间通常会用 HTTP 或 HTTPS 进行通信。

---

## 2. HTTP 的两个重要特点

PPT 提到：

- 请求响应模式（request-response model）
    
- 无状态（stateless）
    

这两个词必须真正理解。

---

### （1）请求响应模式

HTTP 的基本工作方式是：

- 客户端发请求（request）
    
- 服务器回响应（response）
    

例如：

浏览器请求一个网页：

```http
GET /index.html HTTP/1.1
Host: example.com
```

服务器返回：

```http
HTTP/1.1 200 OK
Content-Type: text/html

<html>...</html>
```

所以 HTTP 不是“服务器主动一直推给你内容”的主模式，而是“你请求，我响应”。

---

### （2）无状态（stateless）

所谓无状态，是指：

> HTTP 协议本身默认不记得上一次请求是谁发的、做过什么。

例如：

- 你第一次请求首页
    
- 你第二次请求个人中心
    

从纯 HTTP 协议角度看，这两个请求彼此独立。  
服务器不会天然记得“这是同一个用户”。

这就会引出后面的 Cookie / Session 机制。

---

## 3. HTTP 为什么通常基于 TCP

PPT 说 HTTP 常基于 TCP 连接，这很对。

因为网页传输通常要求：

- 内容尽量完整
    
- 顺序正确
    
- 文本不要乱序缺失
    

TCP 提供可靠传输，很适合 HTTP。

当然现代 HTTP/3 底层引入 QUIC，基于 UDP 思想实现可靠传输机制，这是更进阶的话题。你当前先记住：

> 传统 HTTP/1.1、HTTP/2 通常基于 TCP。

---

# 十八、HTTP URL 的结构

PPT 写了一个 URL 格式：

```text
http://host[":"port][abs_path]
```

可以扩展理解成更完整形式：

```text
scheme://host:port/path?query#fragment
```

例如：

```text
https://www.example.com:443/search?q=html#part1
```

各部分含义：

- `https`：协议 scheme
    
- `www.example.com`：主机 host
    
- `443`：端口 port
    
- `/search`：路径 path
    
- `q=html`：查询参数 query
    
- `part1`：片段标识 fragment
    

---

# 十九、HTTP 请求报文结构：为什么你抓包时能看到那么多字段

PPT 提到 HTTP 请求由这些部分组成：

- 请求行
    
- 请求头信息
    
- 空行
    
- 请求数据
    

这是标准结构。

---

## 1. 请求行（request line）

例如：

```http
GET /index.html HTTP/1.1
```

包含三部分：

- 请求方法（GET / POST / PUT / DELETE 等）
    
- 请求路径
    
- HTTP 版本
    

---

## 2. 请求头（request headers）

例如：

```http
Host: www.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html
Connection: keep-alive
```

这些头部用于告诉服务器更多上下文信息。

---

### Host

目标主机。  
在一台服务器承载多个网站时尤其重要。

### User-Agent

客户端身份信息，通常表示浏览器或程序类型。

### Accept

客户端能接受的数据类型。

### Connection

是否保持连接，例如 `keep-alive`。

---

## 3. 空行

空行用于分隔“头部”和“消息体”。

---

## 4. 请求体（request body）

通常在 POST 等请求里更常见，用来携带表单数据、JSON 数据等。

例如：

```http
name=Tom&age=20
```

# 二十、HTTP 和 HTTPS 的区别：这页 PPT 标题有 HTTPS，但正文讲得不多，我给你补全

PPT 标题写“HTTP 和 HTTPS 协议”，但正文主要展开了 HTTP。  
这里必须系统补一下 HTTPS。

---

## 1. HTTPS 是什么

HTTPS 可以理解为：

> **HTTP + TLS/SSL 加密层**

也就是在 HTTP 通信外面再套一层安全机制。

---

## 2. HTTPS 解决什么问题

HTTP 明文传输时，中间如果被截获，内容可能直接被看见。  
HTTPS 主要解决：

- **机密性（confidentiality）**：防窃听
    
- **完整性（integrity）**：防篡改
    
- **身份认证（authentication）**：确认服务器身份
    

---

## 3. 为什么登录、支付必须 HTTPS

因为如果是明文 HTTP：

- 用户名密码可能被窃听
    
- 请求内容可能被篡改
    
- 甚至可能被冒充服务器
    

所以现代网站，尤其涉及账号、支付、隐私信息时，HTTPS 基本是必须的。

---

# 二十一、Cookie：为什么 HTTP 无状态还能“记住你”

你的第九页、第十页 PPT 进入 Cookie 和 Session。

这是 Web 基础里非常重要的一块，因为它直接关系到：

- 登录状态
    
- 购物车
    
- 用户身份
    
- 个性化设置
    

---

## 1. Cookie 是什么

PPT 中的定义大意是：

> Cookie 是保存在浏览器端的一小段文本数据，在发送 HTTP 请求时会默认携带。

这个定义基本正确。

更准确地说：

> Cookie 是服务器让浏览器保存的一小段键值信息，之后浏览器访问同一站点时，会按规则自动附带上去。

---

## 2. Cookie 的作用

它的核心作用是：

> 给“无状态的 HTTP”增加“状态记忆能力”

例如：

- 记住你是否登录过
    
- 记住购物车内容标识
    
- 记住用户偏好设置
    
- 记住会话标识
    

---

## 3. Cookie 的基本工作流程

### 第一步

浏览器第一次请求服务器

### 第二步

服务器响应时通过 `Set-Cookie` 告诉浏览器保存一个 Cookie

例如：

```http
Set-Cookie: sessionId=abc123
```

### 第三步

浏览器保存后，以后再访问该站点时自动带上：

```http
Cookie: sessionId=abc123
```

### 第四步

服务器根据这个值识别用户状态

---

## 4. Cookie 为什么能用于身份识别

因为它相当于浏览器随请求自动携带的“身份线索”。

但注意：

> Cookie 本身不一定直接存放完整用户信息，很多时候只是存一个标识符。

真正用户数据可能还在服务器端。

---

# 二十二、Session：为什么很多网站更依赖 Session 维护登录态

PPT 做了 Cookie 机制和 Session 机制对比，这一页非常重要。

---

## 1. Session 是什么

Session 可以理解为：

> 服务器端保存的一段会话状态数据

服务器给某个用户创建一个 session，并分配一个 session ID。  
然后把这个 ID 通常通过 Cookie 发给浏览器。

之后浏览器每次请求时带上 session ID，服务器就能查到对应 session 数据。

---

## 2. Session 的本质

所以你可以这样理解：

- **Cookie**：常存客户端
    
- **Session**：常存服务器端
    
- **Session ID**：常通过 Cookie 传来传去
    

也就是说，很多场景下二者不是对立关系，而是协作关系。

---

## 3. Cookie 机制 vs Session 机制

你 PPT 表格里对比了几个方面，我逐个展开。

---

### （1）存储位置

- Cookie：客户端（浏览器）
    
- Session：服务器端
    

这个是最本质的区别。

---

### （2）安全性

PPT 写 Cookie 安全性较低，Session 更安全。  
这个说法总体可以接受，但要稍微更严谨一些：

- Cookie 因为存在浏览器端，更容易被窃取、篡改或滥用
    
- Session 数据在服务器端，敏感内容通常不直接暴露给客户端，因此相对更安全
    

但如果 session ID 被偷走，攻击者仍可能冒充用户。  
所以“Session 更安全”不是绝对无条件的，它依赖：

- HTTPS
    
- HttpOnly
    
- Secure
    
- SameSite
    
- 合理的过期控制
    
- 服务端校验
    

---

### （3）存储数据量

- Cookie 通常大小有限制，单个约几 KB
    
- Session 在服务器端，容量更灵活
    

所以复杂用户状态一般更适合存在 Session 或数据库中，而不是直接塞进 Cookie。

---

### （4）生命周期

- Cookie 可设置过期时间
    
- Session 通常依赖服务器端会话管理策略，常在会话结束或超时后失效
    

---

### （5）典型应用

- Cookie：记住登录名、主题偏好、追踪标识
    
- Session：登录状态、购物车、权限上下文
    

---

# 二十三、Cookie 和 Session 的真实协作方式

这一点非常重要，很多入门 PPT 会把它们对比得像二选一，但实际开发里常是这样：

1. 用户登录
    
2. 服务器创建 Session
    
3. 服务器生成 Session ID
    
4. 服务器通过 `Set-Cookie` 把 Session ID 发给浏览器
    
5. 浏览器后续请求自动带上这个 Cookie
    
6. 服务器根据 Session ID 找回对应 Session 数据
    

所以常见情况是：

> **Cookie 负责带“身份证号”，Session 负责存“档案内容”。**

这个比喻很形象。

# 二十四、把整部分串起来：一次浏览器访问网页到底发生了什么

现在我们把这几页 PPT 的知识串成一个完整流程。  
这是最关键的整合理解。

假设你在浏览器输入：

```text
https://www.example.com
```

会发生什么？

---

## 第一步：应用层发起请求

浏览器准备访问这个网址，本质上是要发一个 HTTP/HTTPS 请求。

---

## 第二步：DNS 解析域名

浏览器需要把 `www.example.com` 解析成 IP 地址。  
这通常依赖 DNS 协议。

---

## 第三步：确定目标 IP 和端口

- IP：目标服务器的网络地址
    
- 端口：HTTPS 通常是 443
    

---

## 第四步：传输层建立连接

如果是基于 TCP 的通信，浏览器和服务器会先进行 TCP 三次握手，建立连接。

如果是 HTTPS，还会进一步进行 TLS 握手，协商加密参数。

---

## 第五步：构造 HTTP 请求报文

浏览器生成请求行、请求头、可能的请求体。

如果浏览器之前保存过该网站 Cookie，也会自动加到请求头中。

---

## 第六步：网络层与链路层封装并发送

HTTP 数据交给 TCP  
TCP 加头部  
再交给 IP  
IP 加头部  
再交给链路层封装成帧

然后经过路由器、交换机等网络设备转发出去。

---

## 第七步：服务器接收并解封装

服务器从链路层一路向上拆包，最后得到 HTTP 请求。

---

## 第八步：服务器处理请求并返回响应

服务器根据 URL、Cookie、Session、业务逻辑等生成响应内容。

---

## 第九步：浏览器收到响应并渲染页面

HTML、CSS、JavaScript 再被浏览器解析执行，最终形成你看到的网页。

---

# 二十五、你作为 Web 学习者，需要特别抓住哪些点

因为你前面一直在学 HTML、Web 基础，所以这里我帮你从 Web 视角再提炼一下重点。

---

## 1. 你不用一开始把所有协议都背全

现阶段最重要的是抓住这条主线：

- **IP**：找到机器
    
- **端口**：找到程序
    
- **TCP**：可靠传输
    
- **HTTP**：浏览器和服务器交流网页内容
    
- **HTTPS**：安全版 HTTP
    
- **Cookie / Session**：在无状态 HTTP 上维护用户状态
    

这条主线已经足够支撑你理解大多数 Web 基础问题。

---

## 2. Web 开发里最常见的误区

### 误区一：把 IP 和域名混为一谈

域名是给人记的，IP 是给网络寻址的。

### 误区二：把端口当成网址的一部分装饰

端口不是装饰，而是定位服务进程的重要标识。

### 误区三：认为 TCP“可靠”就绝对不丢东西

TCP 提供可靠传输服务，但现实系统是否最终“业务成功”还涉及应用层逻辑。

### 误区四：认为 Cookie 就是不安全、不能用

Cookie 本身是机制，关键在于你存什么、如何保护、是否配合 HTTPS 与安全属性。

### 误区五：认为 Session 就完全不依赖 Cookie

很多时候 Session ID 仍通过 Cookie 传递。

## 一、这一部分 PPT 在讲什么

这一部分的核心主题其实是四句话：

**第一，爬虫本身不是天然违法。**  
网络爬虫是互联网中常见的数据获取技术，小规模抓取常用 Requests，中等规模常用 Scrapy，更大规模则需要定制化系统。问题不在“是不是爬虫”，而在于**抓什么、怎么抓、抓完做什么**。国家网信办的《网络爬虫的法律规制》也明确承认爬虫有开放共享、信息聚合等正面价值，但同时强调恶意抓取会带来服务器负担、隐私泄露、知识产权和经营秩序风险。([国家互联网信息办公室](https://www.cac.gov.cn/2019-06/16/c_1124630015.htm?utm_source=chatgpt.com "网络爬虫的法律规制"))

**第二，法律边界越来越清楚。**  
在中国，与你这部分课件最相关的三部基础法律分别是《网络安全法》（2017-06-01施行）、《数据安全法》（2021-09-01施行）和《个人信息保护法》（2021-11-01施行）。它们共同构成了爬虫合规的基本框架：不能破坏系统安全，不能违法处理数据，不能随意抓取、使用个人信息。([国家法律法规数据库](https://flk.npc.gov.cn/detail?fileId=&id=2c909fdd678bf17901678bf8276f093d&title=%E4%B8%AD%E5%8D%8E%E4%BA%BA%E6%B0%91%E5%85%B1%E5%92%8C%E5%9B%BD%E7%BD%91%E7%BB%9C%E5%AE%89%E5%85%A8%E6%B3%95&type=&utm_source=chatgpt.com "中华人民共和国网络安全法"))

**第三，Robots 协议是一种“规则声明”，不是技术防线。**  
站点可以通过 `robots.txt` 告诉爬虫哪些路径允许抓、哪些不允许抓，但它本质上只是一个“君子协定”。RFC 9309 明确写得很清楚：Robots Exclusion Protocol 只是告诉 crawler 应该如何访问 URI，**它不是访问授权机制**。([RFC Editor](https://www.rfc-editor.org/rfc/rfc9309.html?utm_source=chatgpt.com "RFC 9309: Robots Exclusion Protocol"))

**第四，真正的治理是“法律 + 平台规则 + 技术防护”一起上。**  
课件里提到的 User-Agent 审查、Robots 协议、法律责任，都只是整体治理的一部分。真正有效的数据保护，必须配合鉴权、限流、风控、最小暴露、日志审计等工程手段。中国已有因搜索引擎爬虫遍历请求导致系统文件和组织架构被抓取、进而造成数据泄露的处罚案例，说明“只讲公开、不做防护”是远远不够的。([汕尾市政府](https://www.shanwei.gov.cn/swsgzglh/gkmlpt/content/1/1184/post_1184919.html?utm_source=chatgpt.com "国家网信办发布近期网络安全、数据安全、个人信息保护相关执法 ..."))

---

## 二、Robots.txt 真的能“保护”网站数据不被爬取吗？

**不能。它能“表达规则”，不能“阻止抓取”。**  
最关键的依据就是 RFC 9309：Robots 规则**不是访问授权（not a form of access authorization）**。也就是说：

- 它只能告诉守规矩的爬虫“别抓这里”
    
- 但对恶意爬虫、定制脚本、伪装客户端，几乎没有强制拦截能力
    
- 更糟的是，`robots.txt` 还可能把“你不想被抓的目录”直接暴露出来
    

所以如果一个网站把“防抓取”完全寄托在 `robots.txt` 上，基本等于没设防。([RFC Editor](https://www.rfc-editor.org/rfc/rfc9309.html?utm_source=chatgpt.com "RFC 9309: Robots Exclusion Protocol"))

更准确地说，`robots.txt` 的作用主要有三个：

1. **对正规搜索引擎和合规爬虫发规则公告**
    
2. **减少无意义抓取，节省服务器资源**
    
3. **为平台后续维权提供一层“事先告知”证据**
    

但它做不到：

- 身份校验
    
- 速率控制
    
- 反自动化识别
    
- 阻止登录后数据被滥抓
    
- 阻止 API 被程序直接调用
    

所以结论可以直接记成一句话：

> **Robots.txt 是合规声明，不是安全防护。**

([RFC Editor](https://www.rfc-editor.org/rfc/rfc9309.html?utm_source=chatgpt.com "RFC 9309: Robots Exclusion Protocol"))

## 三、国内外对于爬虫态度的对比

这个问题不能简单说成“国内严、国外松”，更准确的说法是：

> **国内外都不支持恶意爬虫，但国外更强调“公开网页上的竞争与访问自由”，中国更强调“数据权益、平台秩序和个人信息保护”。**

### 1. 中国的总体态度：更强调秩序、安全与数据权利边界

中国官方讨论中，对爬虫的态度通常是“**承认技术中性，但强调规制必要性**”。网信办文章明确提出，恶意抓取会侵害经营自由、增加系统负担、带来欺诈、知识产权和商业秘密风险。检察机关文章也强调：如果抓取的是**非开放数据**，或者**绕过、突破反爬措施**，或者涉及**个人信息**，就可能违法，严重时甚至触犯刑法中的非法获取计算机信息系统数据罪、侵犯公民个人信息罪等。([国家互联网信息办公室](https://www.cac.gov.cn/2019-06/16/c_1124630015.htm?utm_source=chatgpt.com "网络爬虫的法律规制"))

所以中国语境下，平台常见的判断逻辑是：

- 是否公开
    
- 是否获得授权
    
- 是否突破技术保护
    
- 是否影响网站正常运行
    
- 是否涉及个人信息、商业秘密、数据安全
    

整体上偏审慎。

### 2. 美国的总体态度：对“公开网页抓取”相对更开放

美国最有代表性的就是 **hiQ v. LinkedIn**。美国第九巡回法院在 2022 年的判决中认为，针对**公开可访问（publicly accessible）**的网站数据，仅凭 cease-and-desist letter 及平台反对，并不当然构成美国《计算机欺诈和滥用法》（CFAA）意义上的“未经授权访问”；法院同时指出，受害方仍可能依据侵权、版权、合同、隐私等其他法律路径维权。([cdn.ca9.uscourts.gov](https://cdn.ca9.uscourts.gov/datastore/opinions/2022/04/18/17-16783.pdf?utm_source=chatgpt.com "hiQ Labs, Inc. v. LinkedIn Corp"))

这说明美国法在“公开网页数据抓取”上，相对更强调：

- 公开访问空间的竞争秩序
    
- 不能把反黑客法无限扩大成“反一切抓取工具”
    
- 但平台仍可通过合同、版权、隐私、州法侵权等方式维权
    

### 3. 欧盟的总体态度：比美国更看重个人数据保护

欧盟这边，核心不是“抓不抓”，而是**只要抓到的是个人数据，就必须满足 GDPR 要求**。欧洲数据保护委员会（EDPB）和法国 CNIL 的公开材料都强调：网页抓取如果涉及个人数据，必须有合法依据、透明告知、合理保存期限，并采取保护数据主体权利的措施。2024 年法国 CNIL 因 KASPR 抓取 LinkedIn 联系信息，对其处以 24 万欧元罚款，理由就包括缺乏合法依据、透明义务不足、保存期限不当等。([edpb.europa.eu](https://www.edpb.europa.eu/news/news/2025/data-scraping-french-supervisory-authority-fined-kaspr-eu240-000_en?utm_source=chatgpt.com "French Supervisory Authority fined KASPR €240 000"))

### 4. 一句话概括国内外差异

你可以这样记：

- **中国**：更强调平台秩序、数据安全、个人信息和技术突破边界
    
- **美国**：对公开网页抓取相对宽松，但不等于合法无风险
    
- **欧盟**：只要涉及个人数据，监管通常比美国更严格
    

([国家互联网信息办公室](https://www.cac.gov.cn/2019-06/16/c_1124630015.htm?utm_source=chatgpt.com "网络爬虫的法律规制"))

## 四、如果你是某平台的开发者，你会如何保护数据不被爬取？

我的思路会分成四层：**规则层、访问层、数据层、取证层**。  
重点不是“绝对防住”，而是把抓取成本抬高，把高价值数据保护住，把异常行为尽早识别出来。

---

### 1. 规则层：先把边界说清楚

我会做三件事：

**第一，明确 ToS / 开放政策 / robots.txt。**  
把哪些页面可以抓、哪些不行、允许什么频率、是否提供 API、禁止什么用途写清楚。`robots.txt` 不能真正防抓，但它对正规 crawler 有约束作用，对后续投诉、封禁、维权也有证据价值。([RFC Editor](https://www.rfc-editor.org/rfc/rfc9309.html?utm_source=chatgpt.com "RFC 9309: Robots Exclusion Protocol"))

**第二，把公开数据和受限数据分层。**  
例如：

- 首页、公开文章：可被索引
    
- 搜索结果、用户详情、评论列表：谨慎开放
    
- 登录后接口、个人资料、交易数据：默认严格受控
    

**第三，能给 API 的尽量给 API。**  
因为“完全封死”通常不现实，很多商业和生态合作场景更好的做法是**提供受限、计费、可审计的官方接口**，而不是逼着所有人都去扒网页。

---

### 2. 访问层：真正有效的反爬，核心在这里

这是最重要的一层。

**（1）鉴权与最小暴露**  
凡是高价值数据，尽量不要放在匿名可访问页面。  
能要求登录就要求登录，能分权限就分权限，能后端渲染就不要把完整敏感字段一次性吐给前端。

**（2）限流（rate limit）与配额**  
按 IP、账号、设备指纹、Cookie、User-Agent、接口维度做多层限流。  
因为恶意爬虫最典型的特征就是“高频、持续、规律性强”。

**（3）行为风控**  
不要只看 User-Agent。PPT 里讲检查 User-Agent，这个只能算最初级手段，因为它太容易伪装。更有效的是综合判断：

- 请求频率是否异常
    
- 是否遍历页码/ID
    
- 是否固定时间间隔
    
- 是否总在深夜批量拉取
    
- 是否存在异常跳转路径
    
- 是否大量命中搜索、导出、详情接口
    

**（4）验证码 / 二次校验**  
用于高风险操作，而不是全站乱加。  
比如搜索翻页过快、批量访问详情、短时间大范围查询时触发。

**（5）令牌、签名、接口时效控制**  
对 API 请求加入时效 token、签名校验、防重放机制。  
这样即使别人看到请求，也不容易长期批量复用。

**（6）WAF / Bot 管理 / CDN 防护**  
把通用型恶意 bot 挡在边缘层，不要全压到源站。

---

### 3. 数据层：就算被访问，也别“裸奔”

真正成熟的平台不会只想着“挡住请求”，还会做**数据最小化暴露**。

**（1）最小返回原则**  
前端只拿渲染必需字段，不返回额外敏感字段。

**（2）分级展示**  
例如：

- 未登录：只看摘要
    
- 登录用户：看完整内容
    
- 合作方：走专门 API
    
- 内部人员：最严格审计
    

**（3）敏感字段脱敏**  
手机号、邮箱、身份证、住址、联系人等能不展示就不展示，必须展示就做脱敏。

**（4）高价值数据水印/蜜罐字段**  
对于高价值文本、图片、表格，可以加入隐式标记，便于后续追踪泄露来源。  
有时也会设置“正常用户不会访问、但爬虫喜欢遍历”的蜜罐链接，用来识别 bot。

---

### 4. 取证层：防不住时，要有追责能力

一个现实问题是：**没有任何系统能百分之百防住所有爬虫。**  
所以最后必须有审计和取证能力。

我会重点做：

- 访问日志留存
    
- 异常请求画像
    
- 接口调用链追踪
    
- 账号/IP/设备关联分析
    
- 告警与自动封禁
    
- 合规证据留档
    

这点很重要，因为中国已有案例显示，系统若未依法采取必要技术措施、未留存相关日志，数据被搜索引擎爬虫遍历抓取后，平台自身也可能因为未履行安全义务而被处罚。([汕尾市政府](https://www.shanwei.gov.cn/swsgzglh/gkmlpt/content/1/1184/post_1184919.html?utm_source=chatgpt.com "国家网信办发布近期网络安全、数据安全、个人信息保护相关执法 ..."))

## 五、相关案例怎么理解

### 案例 1：LinkedIn v. hiQ（美国）

这个案子说明：

- 对**公开网页数据**，美国法院并不一定支持把抓取一概当成“黑客攻击”
    
- 但这不代表平台没办法，仍可能走合同、版权、隐私、州法侵权等路径
    

它体现的是美国在“公开网页抓取”上的相对开放态度。([cdn.ca9.uscourts.gov](https://cdn.ca9.uscourts.gov/datastore/opinions/2022/04/18/17-16783.pdf?utm_source=chatgpt.com "hiQ Labs, Inc. v. LinkedIn Corp"))

### 案例 2：KASPR 因抓取 LinkedIn 数据被法国 CNIL 罚款

这个案子说明：

- 即使数据“网上能看到”，只要涉及**个人数据**
    
- 也不等于你能随便抓、随便建库、随便商用
    
- 欧盟监管重点不只是“有没有登录墙”，而是**有没有合法依据、有没有履行透明义务、保存是否过度**
    

这体现的是欧盟的隐私法路径。([edpb.europa.eu](https://www.edpb.europa.eu/news/news/2025/data-scraping-french-supervisory-authority-fined-kaspr-eu240-000_en?utm_source=chatgpt.com "French Supervisory Authority fined KASPR €240 000"))

### 案例 3：中国系统被搜索引擎爬虫遍历导致数据泄露

这个案例说明：

- 平台不能把责任全甩给“是别人爬的”
    
- 如果你自己存在目录遍历、未授权访问、缺少日志、未配好防护策略
    
- 导致搜索引擎或 crawler 抓到数据，平台也会因为未履行数据安全义务被处罚
    

这对开发者很有现实意义：  
**“防爬”本质上也是“数据安全”问题。** ([汕尾市政府](https://www.shanwei.gov.cn/swsgzglh/gkmlpt/content/1/1184/post_1184919.html?utm_source=chatgpt.com "国家网信办发布近期网络安全、数据安全、个人信息保护相关执法 ..."))

## 六、最后给你一个适合课堂回答的短结论

你在课上可以这样答，比较稳：

> `robots.txt` 不能真正保护网站数据不被爬取，它只是对合规爬虫的规则声明，不是访问控制手段。国内对爬虫总体更强调数据安全、平台秩序和个人信息保护；美国对公开网页抓取相对更开放，但也不是没有法律风险；欧盟则对涉及个人数据的抓取更严格。作为平台开发者，真正有效的做法不是只靠 robots，而是结合权限控制、限流、风控、验证码、签名、脱敏、日志审计和官方 API，把高价值数据分级保护起来。` ([RFC Editor](https://www.rfc-editor.org/rfc/rfc9309.html?utm_source=chatgpt.com "RFC 9309: Robots Exclusion Protocol"))
