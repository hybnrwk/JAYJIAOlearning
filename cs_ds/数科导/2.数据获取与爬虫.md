这几份课件的主线其实很清楚：先学 **Requests 抓静态页面**，再学 **BeautifulSoup/XPath/正则解析页面**，接着学 **JavaScript/Ajax 动态数据**，然后用 **Selenium/Playwright 操作浏览器**，最后用 **Cookie/Session/JWT 模拟登录**和 **Scrapy 框架**做更系统的爬虫。课件本身也把数据获取从网站调研、Requests、正则，到 BeautifulSoup、XPath，再到 Ajax、Selenium、Playwright、模拟登录、Scrapy 串成了一条技术路线。


# 一、数据获取最重要的不是背代码，而是判断“数据在哪里”

你以后借助 AI 写爬虫，真正要会的是判断这几个问题：

**第一，数据是否已经在网页源代码里？**

如果右键“查看网页源代码”或用 `requests.get(url).text` 能看到目标数据，那么这是最简单的情况，用 **Requests + BeautifulSoup/XPath** 就可以。

**第二，数据是不是由 JavaScript 后加载出来的？**

如果网页打开后能看到数据，但源代码里没有，通常说明数据是 JavaScript 通过 Ajax/Fetch 请求后动态写入页面的。课件第 5 讲说明，浏览器会创建 DOM，JavaScript 可以改变 HTML 内容、属性、样式，也能发送数据、局部刷新页面、下拉加载更多、渲染页面。 这时优先看浏览器开发者工具里的 Network，而不是直接上 Selenium。

**第三，Network 里是否有清晰的 JSON 接口？**

如果有清晰接口，比如 `/api/movie/?limit=10&offset=0` 这类，优先用 **Requests 直接请求接口**，比 Selenium 快、稳定、可批量。

**第四，接口参数是否复杂、加密、需要页面环境？**

如果 Ajax 接口找到了，但参数有签名、加密、时间戳、复杂 token，或者页面结构极其复杂，课件也明确说这时可以使用 Selenium：当页面结构复杂、Ajax 数据分析困难、参数加密，需要模拟浏览器时，Selenium 可以实现“可见即可爬”。

**第五，是否需要登录？**

如果数据必须登录后才能看到，就要判断认证方式：Cookie/Session、JWT，还是浏览器登录。课件第 7 讲把模拟登录分成基于 Cookie/Session 和基于 JWT 两类，并说明登录认证后用户才能实现相应功能。

**第六，是否是大规模、多页面、长期采集？**

如果只是抓几页数据，用 Requests 脚本就够。如果是上万条、多层翻页、断点续爬、去重、清洗、存数据库，就考虑 **Scrapy**。Scrapy 官方文档和课件都强调它有 Engine、Scheduler、Downloader、Spider、Item Pipeline 等组件，适合把抓取、解析、调度、存储拆开。 ([Scrapy Documentation](https://scrapy.readthedocs.io/en/latest/topics/architecture.html?utm_source=chatgpt.com "Architecture overview — Scrapy 2.15.2 documentation"))

# 二、总决策表：什么情况用什么方法

| 场景                 | 优先方法                           | 判断依据                        | 典型任务             |
| ------------------ | ------------------------------ | --------------------------- | ---------------- |
| 页面源代码里直接有数据        | Requests + BeautifulSoup/XPath | 查看源代码能看到目标字段                | 新闻标题、豆瓣静态页面、表格数据 |
| 文本格式很简单，只提取固定模式    | 正则表达式                          | 数据不是完整 HTML，而是简单字符串模式       | 提取手机号、日期、ID、价格   |
| HTML 不规范但结构不复杂     | BeautifulSoup                  | 页面标签乱，但字段容易按 class/id 找     | 教材页面、博客列表        |
| HTML 结构规整、层级明确、大页面 | XPath/lxml                     | 需要快速定位节点或属性                 | 列表页批量提取标题、链接     |
| 页面打开后才出现数据         | 先查 Network 中的 XHR/Fetch        | 源代码没有，Network 返回 JSON       | SPA 网站、下拉加载、评论接口 |
| Ajax 接口清楚、参数简单     | Requests 请求接口                  | Network 里能复制请求 URL/参数       | 电影列表、评论分页、搜索结果   |
| Ajax 参数复杂、加密、页面依赖强 | Selenium/Playwright            | 接口难复现，但浏览器能正常看到             | 动态搜索页、复杂前端站点     |
| 需要点击、输入、切换页面       | Selenium/Playwright            | 任务本质是“操作网页”                 | 自动填表、自动查询、辅助选课   |
| 需要登录但登录简单          | requests.Session 或 Cookie      | POST 表单 + 维护 Cookie 即可      | 教学示例登录站点         |
| 登录后用 token 请求数据    | JWT/Authorization              | Network 请求头中有 Authorization | 前后端分离系统          |
| 大规模、多页面、可复用项目      | Scrapy                         | 需要调度、去重、管道、数据库              | 上万条商品/新闻/评论数据    |
| 对实时性和稳定性要求高的网页操作   | Playwright 优先                  | 自动等待、定位器更稳                  | 自动化网页流程、测试型脚本    |

你学习时不用把每个库的代码都背下来，而要能对着网页问出这句话：

**这个数据是“服务器直接返回的 HTML”，还是“浏览器运行 JavaScript 后通过接口拿到的 JSON”，还是“只能靠模拟浏览器操作才能拿到”？**

这句话判断对了，方法基本就选对了。

# 三、第一步永远不是写代码，而是“调研网站”

课件第 3 讲很强调网站调研：看 `robots.txt`、Sitemap、技术栈、whois 信息、浏览器开发者工具等。 这一步决定你后面少走多少弯路。

你以后面对一个目标网站，可以按这个顺序做：

**第一，看数据来源是否合法、是否公开。**

公开网页、公开接口、课程允许站点、官方 API，是优先选择。登录后个人可见数据，只能在个人授权范围内使用。不要采集隐私数据，不要绕过访问控制。

**第二，看 `robots.txt` 和网站条款。**

`robots.txt` 主要是告诉爬虫哪些路径可以访问、哪些路径不建议访问；Google 也说明它主要用于管理爬虫访问，不是隐藏页面或安全保护机制。([Google for Developers](https://developers.google.com/search/docs/crawling-indexing/robots/intro?utm_source=chatgpt.com "Robots.txt Introduction and Guide | Google Search Central")) 但对课程作业来说，你应当把它作为合规检查的一部分。

**第三，用开发者工具看三处。**

Elements：看 HTML 结构，找字段在不在 DOM 里。

Network：看数据请求，尤其是 XHR/Fetch。

Application：看 Cookie、LocalStorage、SessionStorage、Token。

课件第 3 讲专门展示了 Chrome 开发者工具中的 Elements、Console、Sources、Network、Application 等面板，这些面板是你判断爬虫方法的核心工具。

**第四，先复制少量样本验证。**

不要一上来就写大爬虫。先手动打开一个列表页、一个详情页、一个翻页请求，确认字段、分页规律、登录状态、请求参数，再让 AI 写代码。

# 四、Requests：最基础、最应该优先尝试的方法

Requests 的定位是：**像程序一样直接向服务器发 HTTP 请求**。官方文档也说，Requests 可以非常容易地发送 HTTP/1.1 请求，不需要手动拼 query string 或手动编码 POST 数据。([Requests](https://requests.readthedocs.io/?utm_source=chatgpt.com "Requests: HTTP for Humans - Read the Docs"))

课件第 3 讲把 Requests 的常用方法列出来：`requests.get()` 对应 HTTP GET，主要用于获取 HTML；`requests.post()` 对应 HTTP POST，常用于提交表单。

你不用背代码，但要知道这几个概念：

**GET**：向服务器要数据。比如打开新闻列表、电影列表、搜索结果页。

**POST**：向服务器提交数据。比如登录、提交搜索条件、提交表单。

**headers**：请求头，告诉服务器“我是谁、从哪里来、接受什么格式”。常见有 `User-Agent`、`Referer`、`Cookie`、`Authorization`。

**params**：URL 查询参数。比如 `?page=2&keyword=AI`。

**data/json**：POST 请求体。表单登录一般用 `data`，JSON 接口一般用 `json`。

**response**：服务器返回的结果。可能是 HTML、JSON、图片、文件、状态码等。

适合用 Requests 的典型情况：

你能在源代码里看到数据。

Network 里有清晰 JSON 接口。

登录可以通过 POST 表单复现。

分页 URL 有规律。

数据量较大但请求规则简单。

不适合只用 Requests 的情况：

页面内容完全由 JavaScript 渲染，源代码没有目标数据。

接口参数经过复杂加密。

需要真实点击、拖拽、悬停、滑动、iframe 切换。

需要处理验证码或强风控。

# 五、网页解析：BeautifulSoup、XPath、正则怎么选

Requests 只是把网页拿回来，拿回来之后要把字段提取出来，这就是网页解析。

## 1. BeautifulSoup：适合初学者和不规范 HTML

BeautifulSoup 的优点是简单、容错性好。课件第 4 讲说明，BeautifulSoup 是 HTML/XML 解析器，基于 HTML DOM，会载入整个文档并解析 DOM 树；它支持 CSS 选择器，也可以搭配 Python 标准库或 lxml 解析器。

你可以把 BeautifulSoup 理解成：

**把网页变成一棵标签树，然后按标签名、属性、class、CSS 选择器去找内容。**

适合：

HTML 结构不太规范。

你刚开始写爬虫。

字段数量不多。

更看重易写、易懂，而不是极限速度。

常见思路是：

先找包含一条数据的“卡片”节点。

再在每个卡片里面找标题、链接、评分、时间等字段。

最后整理成字典列表，存成 CSV。

## 2. XPath/lxml：适合结构明确、数据量较大

XPath 是路径语言，像给网页元素写地址。课件第 5 讲也解释了：XPath 是在 XML/HTML 中查找节点和内容的路径语言，lxml 是 Python 中用于解析 HTML/XML、支持 XPath 的高速解析库。

课件第 4 讲进一步说明，XPath 最初用于 XML，但同样适用于 HTML；XPath 返回值是列表，常用 `//` 从任意节点查找，`.` 表示当前节点，`..` 表示父节点，`@` 选取属性。

适合：

网页结构规整。

字段路径稳定。

页面大，标签多。

你需要速度。

你会使用浏览器复制 XPath 或 CSS Selector。

## 3. 正则表达式：只适合简单文本匹配，不适合复杂 HTML

课件第 5 讲给了一个比较表：正则表达式解析速度快，适合简单文本匹配，但上手难度高、容错低、提取属性复杂；BeautifulSoup 上手简单、容错高但较慢；XPath 速度快、属性提取简单，但要求结构较规范。

正则适合：

从一段文本里提取日期：`2026-05-04`

提取邮箱、手机号、价格。

提取固定格式 ID。

不适合：

解析复杂 HTML。

解析多层嵌套标签。

处理结构经常变化的网页。

一句话：**不要用正则去硬解析复杂网页结构。正则是补充工具，不是主力网页解析器。**

# 六、JavaScript、DOM、Ajax：动态网页到底是怎么回事

现在大量网页不是服务器直接返回完整 HTML，而是先返回一个“空壳页面”，再由 JavaScript 请求数据、渲染页面。课件第 5 讲说，现在网页大量依靠 JavaScript 实现点击切换、页面不刷新发送数据、下拉加载更多、页面渲染等。

你需要理解三个词：

**DOM（Document Object Model）**：浏览器把 HTML 文档变成一棵树，JavaScript 可以读取、修改、添加、删除树上的元素。课件中也说，浏览器加载网页时会创建 DOM，借助 DOM 可以操作 HTML 元素。

**Ajax**：网页不用整体刷新，也能向服务器请求数据，然后局部更新页面。MDN 对 XMLHttpRequest 的解释是：它可以与服务器交互，不需要完整刷新页面就能从 URL 获取数据，从而更新页面的一部分。([MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest?utm_source=chatgpt.com "XMLHttpRequest - Web APIs | MDN"))

**Fetch**：现代网页更常用的请求 API。MDN 说明 Fetch API 是进行 HTTP 请求并处理响应的 JavaScript 接口，是 XMLHttpRequest 的现代替代方案。([MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API/Using_Fetch?utm_source=chatgpt.com "Using the Fetch API - MDN Web Docs"))

所以，当你看到页面“下拉加载更多”“点击按钮后局部刷新”“搜索结果无刷新出现”，就要想到：

**这背后很可能有 Ajax/Fetch 接口。优先找接口，而不是直接 Selenium 截页面。**

# 七、Ajax 数据抓取：优先级通常高于 Selenium

很多初学者看到动态网页就想用 Selenium。其实更专业的顺序是：

**先找 Ajax 接口，找不到或复现不了，再用 Selenium/Playwright。**

为什么？

因为 Ajax 接口通常直接返回 JSON，字段清晰、速度快、稳定性好。

Selenium 是模拟浏览器，开销大、速度慢、容易受页面变化影响。

抓 Ajax 的步骤是：

打开网页。

按 F12 打开开发者工具。

进入 Network。

筛选 XHR 或 Fetch。

刷新页面或点击加载更多。

找到返回数据的请求。

看 Preview/Response 是否有目标字段。

看 Headers 里的 Request URL、Method、Query String、Request Payload、Cookie、Authorization。

用 Requests 复现这个请求。

处理分页参数，如 `page`、`offset`、`limit`、`cursor`。

这就是你以后借助 AI 时最重要的能力：你不一定要自己写代码，但要能把 Network 中的关键信息复制给 AI，让 AI 帮你复现请求。

你可以这样问 AI：

> 我在 Network 里找到一个 XHR 请求，返回 JSON。请求方法是 GET，URL 是……，参数有 page、limit、keyword，请求头有 User-Agent 和 Referer。请帮我写一个低频、带异常处理、保存 CSV 的爬虫，并解释每一步。

这比直接说“帮我爬这个网站”有效得多。

# 八、Selenium：把浏览器当成机器人操作

Selenium 的定位不是“更高级的 Requests”，而是**浏览器自动化工具**。它适合你说的“设计简单的操作网页程序”。

课件第 6 讲把 Selenium 的流程概括为五步：创建浏览器对象、访问页面、定位网页元素、网页交互、获取网页内容。 Selenium 可以定位单个或多个节点，支持按 ID、Name、Class、Tag、CSS Selector、XPath 查找元素；也可以执行输入、清空、点击等交互。

Selenium 适合：

自动登录练习网站。

自动填写表单。

自动点击按钮。

自动搜索并下载文件。

自动翻页。

自动操作网页查询系统。

页面必须真实浏览器执行 JS 才能显示数据。

不适合：

高频大规模爬取。

能通过接口直接拿数据的场景。

强反爬、验证码、复杂风控场景。

官方 Selenium 文档也强调了等待机制：显式等待会等待某个条件发生后再继续，隐式等待则是在查找元素时轮询 DOM 一段时间。([Selenium Python 文档](https://selenium-python.readthedocs.io/waits.html?utm_source=chatgpt.com "5. Waits — Selenium Python Bindings 2 documentation")) 这点对网页自动化非常关键，因为网页元素不是瞬间出现的。如果你让 AI 写 Selenium 程序，一定要提醒它：**不要只用 `sleep`，要用显式等待或可操作条件等待。**

# 九、Playwright：更适合新手做稳定网页自动化

你的课件也比较了 Selenium 和 Playwright：Selenium 通过 WebDriver 协议，需要浏览器驱动；Playwright 直接通过浏览器协议通信，通常更快，并且内置智能等待机制。

Playwright 官方文档也说，Locator 是自动等待和重试能力的核心；Playwright 在执行点击等操作前会自动检查元素是否可见、稳定、可接收事件等。([Playwright](https://playwright.dev/python/docs/locators?utm_source=chatgpt.com "Locators | Playwright Python")) ([Playwright](https://playwright.dev/python/docs/actionability?utm_source=chatgpt.com "Auto-waiting | Playwright Python"))

对你的目标来说，我建议：

如果课程要求 Selenium，就按课件学 Selenium。

如果你自己要做网页自动化小工具，优先考虑 Playwright。

因为 Playwright 对“页面还没加载完”“按钮还不能点”“元素刚出现”等问题处理更自然，更适合写稳定脚本。

它适合：

自动打开教务系统页面。

自动筛选课程。

自动点击搜索。

自动填写固定表单。

自动截图、导出页面信息。

自动监控某个按钮是否出现，然后提醒你。

但依然要遵守网站规则，不要用它绕过验证码、排队系统、限流机制或访问限制。

# 十、模拟登录：Cookie/Session、JWT、浏览器登录怎么选

登录不是一个单独技术，而是“认证状态如何保存”的问题。课件第 7 讲讲得很实用。

## 1. Cookie/Session：传统登录常见

课件说明，Cookie 由服务器产生并保存在客户端，浏览器下次向同一服务器发送请求时会携带；Session 保存在服务器，SessionID 保存在客户端 Cookie 中。

这种场景一般是：

你提交用户名密码。

服务器验证通过。

服务器返回 SessionID 或 Cookie。

之后请求自动带 Cookie，服务器知道你是谁。

对应方法：

简单登录表单：用 `requests.Session()` 发送 POST，并维护 Cookie。

复杂登录页面：用 Selenium/Playwright 登录后提取 Cookie。

自己手动登录：复制 Cookie，短期使用。

## 2. JWT：前后端分离系统常见

课件说明，JWT 适用于前后端分离模式，是经过签名的字符串，包含 Header、Payload、Signature 三段；访问时通常在请求头中新增 `Authorization` 字段。

这种场景一般是：

登录接口返回 token。

后续接口请求头带 `Authorization: jwt <token>` 或 `Bearer <token>`。

token 过期后要重新登录获取。

对应方法：

找到登录接口。

发送用户名密码获取 token。

后续请求加 Authorization。

token 过期时重新获取。

## 3. 浏览器自动登录：最像人，但最重

如果登录过程有动态 JS、二维码、短信验证码、多因素认证、滑块验证码，就不适合用纯 Requests 硬模拟。可以用浏览器自动化辅助：

打开登录页。

你手动输入验证码或扫码。

程序保存登录状态。

之后复用状态。

这种方法适合个人自动化，但不要尝试绕过验证码。验证码的目的就是区分人和自动程序，绕过它通常违反服务规则。

# 十一、Scrapy：什么时候需要上框架

Scrapy 不适合刚开始就用。它适合你已经明确知道：

要爬很多页。

URL 调度复杂。

要自动去重。

要断点、重试、日志。

要清洗、验证、存数据库。

要多个 Spider 管理不同网站。

课件第 7-2 讲说明 Scrapy 是基于 Python 的爬虫框架，组件式架构、逻辑清晰、扩展性强；它包含 Engine、Items、Scheduler、Spiders、Downloader、Item Pipeline、Downloader Middleware、Spider Middleware 等组件。 Scrapy 官方文档也说明，Item Pipeline 负责在 Spider 抽取 Item 后进行清洗、验证、持久化等处理。([Scrapy Documentation](https://scrapy.readthedocs.io/en/latest/topics/item-pipeline.html?utm_source=chatgpt.com "Item Pipeline — Scrapy 2.15.2 documentation"))

你可以把 Scrapy 理解成：

Requests + 解析 + 队列 + 去重 + 并发 + 管道 + 存储 + 日志 的工程化框架。

什么时候不用 Scrapy？

只爬一个页面。

只做一次性小作业。

需要复杂浏览器交互。

登录流程很复杂且依赖真实浏览器。

什么时候用 Scrapy？

新闻站列表页 + 详情页。

# 十二、针对“抢课/网页操作程序”的方法选择

你提到“如抢课等”，这里要非常明确：可以学习**网页自动化思路**，但不应做破坏公平或违反学校系统规则的脚本。合规方向可以是：

自动打开选课系统。

自动登录到指定页面，验证码/二次认证由你手动完成。

自动筛选课程名称、教师、时间段。

自动监控页面上某个课程状态是否变成“可选”。

出现可选状态后提醒你。

最后提交由你手动确认。

不建议也不能做：

高频刷新抢占服务器资源。

绕过验证码或排队。

伪造请求绕过前端限制。

批量抢多个账号。

绕过学校系统的选课规则。

从技术上说，如果只是合规辅助，优先用 **Playwright 或 Selenium**。理由是你的任务是“操作网页”，不是“批量抓取数据”。流程通常是：

启动浏览器。

进入登录页。

等待你手动完成登录。

保存登录状态。

进入选课页面。

按课程名/课程号筛选。

定位课程行。

检测按钮状态。

提醒或等待人工确认。

这个任务的核心不是爬虫，而是**浏览器自动化（browser automation）**。Selenium 和 Playwright 都能做；Playwright 的自动等待和 Locator 更适合这类稳定操作。([Playwright](https://playwright.dev/python/docs/locators?utm_source=chatgpt.com "Locators | Playwright Python"))

# 十三、你应该掌握的“AI 协作能力”

你不需要记住所有代码，但要学会把任务拆给 AI。以后你要让 AI 帮你爬数据，最好给它这些信息：

## 1. 任务描述

告诉 AI：

我要抓哪个网站。

抓哪些字段。

预计抓多少条。

是否需要登录。

是否只抓公开数据。

保存成什么格式。

示例：

> 我要抓取一个公开电影列表网站的电影名、评分、年份、简介，目标约 5000 条，保存为 CSV。请先帮我判断用 Requests、Ajax 接口、Selenium 还是 Scrapy，不要直接写代码。

## 2. 页面证据

你最好复制给 AI：

目标网页 URL。

Elements 中一条数据卡片的 HTML。

Network 中 XHR 请求的 URL、请求方法、参数、响应样例。

是否有 Cookie、Authorization。

分页参数变化规律。

示例：

> 我在 Network 里看到接口 `/api/movies/?limit=10&offset=20`，Response 是 JSON，里面有 name、score、cover、categories。请帮我写一个低频翻页抓取脚本，带重试、去重和 CSV 保存。

## 3. 合规限制

明确告诉 AI：

只抓公开数据。

每秒不超过多少请求。

遇到验证码停止。

不绕过登录限制。

遵守 robots 和网站规则。

这样 AI 生成的方案会更稳、更安全。

## 4. 调试信息

如果代码出错，不要只说“跑不了”。要给 AI：

报错全文。

状态码。

返回内容前 500 字。

目标字段示例。

当前代码。

浏览器中是否能看到数据。

这样 AI 才能判断是请求头问题、编码问题、选择器问题、登录失效、反爬、还是动态渲染问题。

# 十四、常见错误与排查方法

## 1. 浏览器能看到，Requests 抓不到

可能原因：

数据是 JS 动态加载。

需要 Cookie。

需要 Authorization。

需要 Referer/User-Agent。

接口参数缺失。

解决：

看 Network 的 XHR/Fetch。

优先复现接口。

复现不了再 Selenium/Playwright。

## 2. 返回 403

可能原因：

没有请求头。

访问频率过高。

需要登录。

触发风控。

路径被禁止。

解决：

降低频率。

补充合理 headers。

确认是否需要登录。

检查 robots 和条款。

不要尝试绕过强风控。

## 3. 抓到的 HTML 没有目标数据

可能原因：

目标数据在 JS 里渲染。

目标数据来自 JSON 接口。

目标数据在 iframe 中。

解决：

查看页面源代码是否有字段。

看 Network。

检查 iframe。

必要时浏览器自动化。

## 4. Selenium 找不到元素

可能原因：

元素还没加载完。

选择器写错。

元素在 iframe 里。

元素被遮挡。

页面动态变化。

解决：

使用显式等待。

优先用稳定的 CSS Selector 或文本定位。

切换 iframe。

检查元素是否可见、可点击。

## 5. 抓取结果重复或漏页

可能原因：

分页参数没理解。

下拉加载 cursor 不是 page。

接口有时间排序变化。

程序没有去重。

解决：

记录唯一 ID。

保存已抓 URL。

观察翻页参数变化。

使用去重集合或数据库唯一键。

# 十六、一个最实用的判断流程

以后面对任何网站，按这个顺序：

**第 1 步：先看有没有官方 API 或可下载数据。**

有官方 API 就用 API，不要爬网页。

**第 2 步：看 robots.txt、网站规则、数据是否公开。**

不能访问的别抓，涉及隐私的别抓。

**第 3 步：用 Requests 请求页面。**

如果 HTML 里有数据，用 BeautifulSoup/XPath 解析。

**第 4 步：如果 HTML 里没有，看 Network。**

如果有 JSON 接口，用 Requests 复现接口。

**第 5 步：如果接口复杂，再用 Selenium/Playwright。**

尤其是需要点击、输入、滚动、iframe、等待加载的场景。

**第 6 步：如果要上万条、长期抓，改成 Scrapy。**

用 Spider 负责抓取，Item 定义字段，Pipeline 清洗和存储。

**第 7 步：如果需要登录，先判断认证方式。**

Cookie/Session 用 Session 维护，JWT 用 Authorization，复杂登录用浏览器登录状态。

# 十七、针对你当前目标的结论

你现在不需要追求“会手写各种爬虫代码”。你应该重点掌握：

**Requests 是直接向服务器要数据。**

**BeautifulSoup/XPath 是从 HTML 里提字段。**

**正则只适合简单文本模式。**

**Ajax 是浏览器背后偷偷向接口要数据，优先从 Network 找接口。**

**Selenium/Playwright 是让浏览器像人一样点击、输入、等待、翻页。**

**Cookie/Session/JWT 是登录状态的保存方式。**

**Scrapy 是把大规模爬虫工程化的框架。**

对于课程作业的数据采集，优先路线是：

**公开网站 → Network 找接口 → Requests 批量抓 JSON → 保存 CSV。**

对于你想做的网页自动化小工具，优先路线是：

**Playwright/Selenium → 手动登录/合规登录 → 定位元素 → 输入/点击/等待 → 人工确认关键提交。**

真正要记住的不是代码，而是这句话：

**能直接请求接口，就不要模拟浏览器；能解析 HTML，就不要上动态自动化；需要真实交互，才用 Selenium/Playwright；规模大、流程复杂，才用 Scrapy。**
