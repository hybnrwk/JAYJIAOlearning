下面我把这部分 PPT 按“**调研网站 → 用浏览器观察请求 → 用 Requests 发请求 → 处理真实网页中的常见问题**”这一条主线，给你做一次系统讲解。你可以把它理解成：**在真正写爬虫之前，先学会像侦探一样看网站，再学会像客户端一样发请求。**

# 一、这部分 PPT 的整体逻辑是什么

这组 PPT 其实在讲一个非常标准的爬虫入门流程：

1. **先调研网站**  
    看 `robots.txt`、`Sitemap`、域名信息、技术栈，先搞清楚这个站大概是什么结构、允许什么、不允许什么。
    
2. **再用浏览器开发者工具观察**  
    看页面到底请求了哪些资源，哪些是 HTML，哪些是接口返回的 JSON，哪些需要带参数、Cookie、Header。
    
3. **最后才用 Python Requests 去模拟请求**  
    先会 `GET`、`POST`，再会带参数、带头信息、带 Cookie、处理二进制内容、处理超时。
    

也就是说，这部分不是在教你“盲写代码”，而是在教你：

> **写爬虫之前，先理解网页是怎么工作的。**

这和 Requests 官方文档的定位是一致的：Requests 是一个易用的 HTTP 客户端库，用来发送 HTTP 请求、拿到 `Response` 对象，并处理参数、表单、Cookie、超时等常见问题。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/user/quickstart/?utm_source=chatgpt.com "Quickstart — Requests 2.33.0.dev1 documentation"))

# 二、第一部分：调研网站，为什么要先调研

很多初学者一上来就 `requests.get(url)`，然后发现抓不到想要的数据，就开始怀疑自己代码写错了。其实很多时候不是代码问题，而是**网站结构还没看懂**。

调研网站的目的，是在真正写代码前回答这几个问题：

- 这个网站是否公开允许一部分爬虫访问
    
- 网站大致有哪些栏目和页面层次
    
- 页面是静态 HTML 为主，还是前端动态渲染
    
- 关键数据是放在 HTML 里，还是通过接口异步加载
    
- 网站是否有登录态、Cookie、Token、参数签名等机制
    

所以“调研”这一步，本质上是在做**信息收集（reconnaissance）**，但这里是合法合规、面向公开信息的调研。

---

## 1. `robots.txt`：网站对爬虫的公开说明书

PPT 里先讲 `robots.txt`，这是非常合理的，因为很多正规 crawler 在访问站点前都会先看它。

`robots.txt` 属于 **Robots Exclusion Protocol**。RFC 9309 把它正式标准化了，并明确说：它是网站告诉 crawler 哪些 URI 希望被访问、哪些不希望被访问的一套规则；同时也明确说明，**这些规则不是访问授权机制**。换句话说，`robots.txt` 是“规则声明”，不是“技术防线”。([RFC Editor](https://www.rfc-editor.org/rfc/rfc9309.html?utm_source=chatgpt.com "RFC 9309: Robots Exclusion Protocol"))

这句话你一定要吃透：

- 它告诉合规爬虫“别抓哪里”
    
- 它不等于真正阻止别人访问
    
- 它更像“门口贴的告示”，不是“门锁”
    

### 一个典型 `robots.txt` 会告诉你什么

常见内容包括：

- `User-agent`：这条规则对哪些爬虫生效
    
- `Disallow`：不希望抓取的路径
    
- `Allow`：允许抓取的路径
    
- `Sitemap`：站点地图地址
    

### 现实中的实用价值

你作为学习者，读 `robots.txt` 至少有三层价值：

第一，**合规意识**。  
先知道网站希望你别碰哪些路径。

第二，**结构线索**。  
被列出来的路径名本身就能帮助你理解网站模块，例如 `/search/`、`/member/`、`/api/`、`/service/` 等。

第三，**Sitemap 入口**。  
很多站点会直接在 `robots.txt` 里给出 `Sitemap` 地址，等于直接告诉你有哪些公开页面值得索引。RFC 9309 和 Sitemaps 协议都支持这种做法。([RFC Editor](https://www.rfc-editor.org/rfc/rfc9309.html?utm_source=chatgpt.com "RFC 9309: Robots Exclusion Protocol"))

### 你在实践里可以怎么做

你可以把“先看 robots”养成习惯：

```python
import requests

url = "https://example.com/robots.txt"
r = requests.get(url, timeout=10)
print(r.status_code)
print(r.text[:1000])
```

这个练习很适合你先在一些公开站点、教学站点或你自己的站点上做。

### 一个重要提醒

不要把 `robots.txt` 当成“绕过清单”去理解，而要把它当成“公开规则 + 站点结构线索”去理解。  
这是爬虫入门时非常重要的态度。

---

## 2. Sitemap：网站公开页面的地图

PPT 里把 `robots.txt` 和 `Sitemap` 放在一起，这是非常对的，因为它们在“网站调研”里常常配套出现。

Sitemaps.org 对 Sitemap 的定义很清楚：Sitemap 是一种告诉搜索引擎“我的站点有哪些 URL 可以被抓取”的协议。最常见形式是 XML 文件，里面列出 URL，并可附带更新时间、更新频率、相对重要性等元数据。协议要求 XML 中的值进行实体转义，文件用 UTF-8 编码，并可以使用 sitemap index 文件管理多个 sitemap。([sitemaps.org](https://www.sitemaps.org/protocol.html?utm_source=chatgpt.com "sitemaps.org - Protocol"))

### 为什么 Sitemap 很有用

对搜索引擎来说，它是抓取入口。  
对你做网站调研来说，它也是抓取入口。

它能帮助你快速知道：

- 站点有哪些公开页面
    
- 页面是否按栏目拆分
    
- 某些深层页面是否没有明显导航，但依然存在于站点地图中
    
- 站点规模大概有多大
    

### 它和 `robots.txt` 的关系

可以这样记：

- `robots.txt` 更偏“**排除规则**”
    
- `Sitemap` 更偏“**包含清单**”
    

也就是说，一个是“哪些别抓”，一个是“哪些可以索引”。

### 实践建议

调研站点时，可以按这个顺序看：

1. 先开 `https://站点域名/robots.txt`
    
2. 找里面有没有 `Sitemap`
    
3. 再看 sitemap 文件是 XML 还是 HTML 导航页
    
4. 观察 URL 结构有没有明显规律
    

这一步非常适合用来帮助你设计后续采集范围，而不是一上来就胡乱翻页。

## 3. 技术栈识别：为什么要知道网站用了什么前端框架

PPT 里用 `wad.detection` 举例，核心思想不是这个工具本身，而是：

> **先判断网站用了什么技术，能帮你更快判断“数据大概在哪儿”。**

如果一个网站主要是传统服务端渲染，那很多数据就在 HTML 里。  
如果网站用了大量前端框架、模块加载器、模板引擎，那么你就要警惕：

- HTML 初始页面可能只是骨架
    
- 真实数据可能在接口里
    
- 页面内容可能由 JS 运行后再填充到 DOM 中
    

### PPT 里提到的几个库怎么理解

你 PPT 里举了：

- jQuery
    
- RequireJS
    
- Handlebars
    

这几个名字本身就很有代表性：

**jQuery**  
主要简化 DOM 操作、事件处理、AJAX 请求等。看到它，说明页面可能有较多前端脚本交互。

**RequireJS**  
说明页面可能按模块加载 JS，不是一股脑全部写死在一个文件里。

**Handlebars**  
是模板引擎，说明前端可能会拿到数据后，再通过模板生成 HTML。

所以如果你发现一个页面用了这种技术组合，你就要想到：

> “我未必该直接解析最初 HTML，而应该优先看 Network 里有没有接口返回 JSON。”

这就是“技术栈调研”的现实意义。

### 实际做法

今天更常见的是用浏览器插件、Wappalyzer 一类工具、或直接看源码里的脚本引用来判断，而不是执着于某个 Python 检测库。  
重点不在“工具名”，重点在“能不能由技术栈推断抓取策略”。

## 4. 域名信息：WHOIS / RDAP 能告诉你什么

PPT 里写的是 `whois` 命令，这在概念上没问题，但你现在要知道一个更新的事实：ICANN 说明，**RDAP（Registration Data Access Protocol）是 WHOIS 的标准化替代方案**，用于访问域名注册数据，提供更结构化、标准化的查询结果。([icann.org](https://www.icann.org/en/contracted-parties/registry-operators/resources/registration-data-access-protocol?utm_source=chatgpt.com "Registration Data Access Protocol (RDAP)"))

### 域名信息调研有什么用

对爬虫学习者来说，主要不是拿来“抓数据”，而是拿来做背景判断，比如：

- 这是主站还是子域名
    
- 域名大概什么时候注册
    
- 是否是很新的域名
    
- 某些站点是否多个子域共属一个业务体系
    

### 现实里更实用的理解

你不一定非要深挖 WHOIS 字段，但你要建立一种习惯：

> 对目标站点，先搞清楚“我在访问谁”。

这在做网站分析、做竞品调研、做数据来源辨识时都很有帮助。

# 三、第二部分：浏览器开发者工具，为什么它是爬虫学习的核心工具

这部分 PPT 很关键。  
我甚至会说：

> **不会用 Chrome DevTools，就不会真正做网页爬取。**

因为 Requests 只是“发请求”的工具，而浏览器开发者工具是“看清楚浏览器到底发了什么请求”的工具。

Chrome 官方文档把 Network panel 定义为用于分析网络加载与资源请求的面板，可以查看请求、响应、Headers、Timing 等。([Chrome for Developers](https://developer.chrome.com/docs/devtools/network/overview?utm_source=chatgpt.com "Network panel: Analyze network load and resources"))

---

## 1. 为什么开发者工具这么重要

因为你想模拟浏览器，就得先知道浏览器做了什么。

例如你打开一个网页，表面上看到的是页面内容，但浏览器背后可能还发了：

- HTML 文档请求
    
- CSS 请求
    
- JS 请求
    
- 图片请求
    
- 字体请求
    
- XHR / Fetch 接口请求
    
- WebSocket 连接
    

对爬虫最重要的，通常不是图片和 CSS，而是：

- **Doc**：主 HTML
    
- **Fetch/XHR**：接口请求
    
- 有时是 **WS**：实时数据
    

---

## 2. 几个最值得你盯住的面板

### Elements

看渲染后的 DOM 结构。  
适合判断最终页面里到底有哪些节点、哪些类名、哪些属性。

### Console

看前端报错、试验一些 JS 表达式。  
适合辅助判断页面脚本是否运行正常。

### Sources

看站点加载了哪些 JS 文件。  
适合分析前端代码结构，但入门阶段不是最核心。

### Network

这是爬虫学习里最关键的。  
你要重点看它。

---

## 3. Network 面板到底看什么

你要重点看五件事：

### （1）请求 URL

数据是从哪个地址来的。

### （2）请求方法

是 `GET` 还是 `POST`，还是别的。

### （3）请求参数

参数是在 URL 上，还是在请求体里。

### （4）请求头

是否带了 `User-Agent`、`Cookie`、`Referer`、`Authorization` 等。

### （5）响应内容

返回的是 HTML、JSON、图片，还是报错信息。

### 一个很实用的判断技巧

如果页面内容是动态变化的，你先在 Network 里筛选 `Fetch/XHR`。  
很多时候你会直接看到一个 JSON 接口，这比你硬啃 HTML 省事得多。

---

## 4. 这一步在真实实践中的价值

一个页面抓不到，通常不是马上改代码，而是先回浏览器开发者工具检查：

- 这个数据是否本来就不在 HTML 里
    
- 是否请求后才加载
    
- 是否翻页时额外请求接口
    
- 是否提交表单用了 POST
    
- 是否必须有 Cookie 或某些 Header
    

你未来做任何稍微复杂一点的采集，开发者工具都是先手。

# 四、第三部分：Requests 库到底是什么

Requests 官方文档把它定位为一个非常易用的 HTTP 库，支持简单地发送 HTTP/1.1 请求；主接口包括 `request()`、`get()`、`head()`、`post()`、`put()`、`patch()`、`delete()`，它们都会返回 `Response` 对象。([requests.readthedocs.io](https://requests.readthedocs.io/?utm_source=chatgpt.com "Requests: HTTP for Humans™ — Requests 2.33.0.dev1 ..."))

你可以把 Requests 理解成：

> **Python 世界里，替你扮演“客户端”的一个工具。**

浏览器也是客户端。  
Requests 也是客户端。  
差别在于：

- 浏览器还会解析 HTML、执行 JS、渲染页面
    
- Requests 只负责收发 HTTP 数据，不负责“看懂网页并渲染出来”
    

这句话非常重要。

---

# 五、Requests 的 7 个主要方法怎么理解

PPT 里列了 7 个方法，这个分类和官方 API 一致。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/api/?utm_source=chatgpt.com "Developer Interface — Requests 2.33.0.dev1 documentation"))

---

## 1. `requests.request()`

这是最通用的底层入口。  
你传入方法名，它帮你构造并发送请求。

```python
requests.request("GET", url)
requests.request("POST", url, data=data)
```

适合理解原理，但日常直接用 `get()`、`post()` 更多。

---

## 2. `requests.get()`

最常用。  
用于获取资源，对应 HTTP 的 GET。

适合：

- 请求网页
    
- 请求公开接口
    
- 带查询参数搜索
    

---

## 3. `requests.head()`

请求响应头，不拿正文。  
适合快速看资源是否存在、内容类型是什么、长度多大。

现实中没有 `get()` 常用，但在做资源探测时很有价值。

---

## 4. `requests.post()`

提交数据，对应 HTTP 的 POST。

适合：

- 表单提交
    
- 登录接口
    
- 搜索接口
    
- 上传文件
    
- 调 API
    

---

## 5. `put()`、`patch()`、`delete()`

在 REST 风格 API 里更常见。  
普通网页抓取里不一定高频，但做接口调用时会见到。

---

# 六、`requests.get()`：最重要的第一步

PPT 把 `get()` 单独拿出来讲，是对的。  
因为爬虫入门的第一步几乎总是它。

官方 Quickstart 也是先用 `requests.get()` 举例，并强调返回的是 `Response` 对象。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/user/quickstart/?utm_source=chatgpt.com "Quickstart — Requests 2.33.0.dev1 documentation"))

---

## 1. `get()` 到底做了什么

```python
r = requests.get(url)
```

它做了两件事：

第一，构造一个 HTTP 请求并发出去。  
第二，把服务器返回内容封装成 `Response` 对象交给你。

所以 `r` 不是网页本身，而是“服务器回应你的结果”。

---

## 2. `Response` 对象里最常用的东西

### `r.status_code`

状态码，表示请求结果。

### `r.text`

按文本解码后的内容，常用于 HTML、JSON 文本。

### `r.content`

原始二进制内容，适合图片、文件。

### `r.headers`

响应头。

### `r.cookies`

服务器返回的 Cookie。

### `r.json()`

如果返回的是 JSON，可以直接解析。

---

## 3. 一个最基础但正确的写法

```python
import requests

r = requests.get("https://example.com", timeout=10)
print(r.status_code)
print(r.text[:500])
```

这已经比很多初学者的第一版代码更靠谱了，因为你加了 `timeout`。

---

# 七、HTTP 状态码：为什么不能只盯着 200

PPT 里列了几个最常见状态码。  
你不用一开始背全，但要把最核心的理解透。

### 200

成功，服务器正常返回内容。

### 201

创建成功，常见于 API 新建资源。

### 304

未修改。常见于缓存协商，不一定有正文。

### 400

请求有问题，参数错、格式错。

### 401

未认证。通常要登录或带认证信息。

### 403

禁止访问。不是“没找到”，而是“服务器知道你是谁，但不让你看”。

### 404

没找到。

### 429

请求过多。很典型的限流信号。

### 500

服务器内部错误。

### 503

服务暂不可用。

---

## 1. 初学者最容易犯的错

看到网页打不开，就只打印 `r.text`。  
其实第一步应该先看：

```python
print(r.status_code)
```

因为：

- 403 和 404 的处理思路完全不同
    
- 200 但返回的是报错 HTML，也和“成功拿到目标数据”不是一回事
    

---

## 2. 一个更实用的检查方式

Requests 官方文档推荐用 `raise_for_status()` 在 4xx/5xx 时抛出异常。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/user/quickstart/?utm_source=chatgpt.com "Quickstart — Requests 2.33.0.dev1 documentation"))

```python
r = requests.get(url, timeout=10)
r.raise_for_status()
```

这个习惯非常好，尤其在你以后写批量脚本时。

---

# 八、GET 参数：`params=` 到底是在干什么

PPT 里给了 `requests.get(..., params=data)` 的例子。  
这个是必须掌握的，因为很多搜索页、筛选页、API 查询页都在用它。

Requests 文档明确说明：你可以把查询参数作为字典传给 `params`，Requests 会自动拼到 URL 上。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/user/quickstart/?utm_source=chatgpt.com "Quickstart — Requests 2.33.0.dev1 documentation"))

例如：

```python
import requests

params = {
    "name": "germey",
    "age": 22
}
r = requests.get("https://httpbin.org/get", params=params, timeout=10)
print(r.url)
print(r.text)
```

它实际请求的 URL 会变成类似：

```text
https://httpbin.org/get?name=germey&age=22
```

---

## 1. 为什么 `params` 很重要

因为你在浏览器里看到的很多：

- 搜索关键词
    
- 页码
    
- 排序方式
    
- 分类条件
    

都体现在 URL 参数里。

比如：

```text
?page=2
?q=python
&sort=hot
```

这时候你只要改参数，不用改整段 URL。

---

## 2. 实际练习建议

你可以自己找一个公开的查询型接口，先在浏览器里改参数观察结果，再用 `requests.get(..., params=...)` 复现。  
这会让你非常快地理解“URL 参数驱动页面内容变化”这件事。

---

# 九、POST 参数：`data=` 和“提交表单”是什么关系

PPT 里用了 `requests.post(..., data=data)` 的例子。  
这在逻辑上对应“向服务器提交一组表单字段”。

Requests 文档说明：POST 数据可以通过 `data=` 传递，Requests 会替你完成表单编码。([requests.readthedocs.io](https://requests.readthedocs.io/?utm_source=chatgpt.com "Requests: HTTP for Humans™ — Requests 2.33.0.dev1 ..."))

例如：

```python
import requests

data = {
    "name": "germey",
    "age": "25"
}
r = requests.post("https://httpbin.org/post", data=data, timeout=10)
print(r.text)
```

---

## 1. `GET` 和 `POST` 在爬虫里怎么区分

### GET

通常是“拿数据”

- 看页面
    
- 翻页
    
- 搜索公开内容
    

### POST

通常是“提交数据”

- 登录
    
- 搜索表单
    
- 提交查询条件
    
- 上传文件
    

但这只是常见模式，不是绝对。  
真正判断还是要看浏览器开发者工具里的请求方法。

---

## 2. 一个实用认知

你看到浏览器里是 POST，就不要再硬写 GET 去猜。  
先按浏览器真实请求去模拟。

# 十、Header：为什么同一个 URL，浏览器能打开，Requests 却不行

PPT 里有一个很典型的例子：不带某些 header 时，请求结果和浏览器不同。

HTTP 请求头是客户端给服务器的附加信息。  
Chrome DevTools 能让你直接看到浏览器发了哪些请求头；Requests 也允许你通过 `headers=` 传入字典控制请求头。([Chrome for Developers](https://developer.chrome.com/docs/devtools/network/overview?utm_source=chatgpt.com "Network panel: Analyze network load and resources"))

---

## 1. 最常见的几个请求头

### `User-Agent`

告诉服务器“我是什么客户端”。

### `Accept`

我愿意接收什么类型的数据。

### `Referer`

我是从哪个页面来的。

### `Cookie`

我当前带着哪些站点状态信息。

### `Authorization`

认证信息，API 很常见。

---

## 2. 为什么 Header 很重要

因为服务器有时会根据 Header 决定：

- 返回 PC 页面还是移动页面
    
- 返回 HTML 还是 JSON
    
- 是否认为这是正常浏览器访问
    
- 是否需要登录态
    
- 是否触发某些风控策略
    

---

## 3. 正确理解“带浏览器头信息”

你不要把它理解成“专门为了绕过”。  
更准确地说，它是在**尽量复现真实客户端上下文**。

例如：

```python
headers = {
    "User-Agent": "Mozilla/5.0 ..."
}
r = requests.get("https://example.com", headers=headers, timeout=10)
```

这在很多公开页面测试中是正常做法，因为有些站点会根据客户端类型返回不同内容。

---

## 4. 现实里最实用的做法

不要手抄一大堆 Header。  
先在开发者工具里看真实请求，抓住最关键的几个：

- User-Agent
    
- Referer
    
- Cookie
    
- Authorization
    
- Content-Type
    

多数情况下，这就够了。

# 十一、非文本内容：为什么图片不能用 `r.text`

PPT 里拿 favicon 举例，这个例子非常好。

Requests Quickstart 明确区分了文本响应和二进制响应：文本常看 `r.text`，二进制内容要看 `r.content`。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/user/quickstart/?utm_source=chatgpt.com "Quickstart — Requests 2.33.0.dev1 documentation"))

---

## 1. 为什么会乱码

因为图片、音频、PDF、压缩包这些本质上不是文本。  
你强行 `print(r.text)`，就是拿文本解码器去解二进制，自然会乱。

---

## 2. 正确方法

```python
import requests

r = requests.get("https://github.com/favicon.ico", timeout=10)
with open("favicon.ico", "wb") as f:
    f.write(r.content)
```

这里的关键点有两个：

- 用 `r.content`
    
- 文件用二进制模式 `"wb"`
    

---

## 3. 一个很实用的判断方式

在写文件前，你可以先看响应头里的 `Content-Type`，判断这是图片、JSON，还是 HTML。  
这个习惯很专业。

# 十二、文件上传：为什么 `files=` 看起来像“反过来的爬虫”

PPT 里有 `requests.post(..., files=files)` 的例子。  
这不是抓网页本身，但它能帮你理解一个重要事实：

> Requests 不只是“下载数据”，也是“模拟客户端提交数据”。

如果某个网页表单支持上传文件，浏览器本质上也是发一个 HTTP POST，只不过正文里包含 multipart/form-data 格式的文件内容。  
Requests 支持直接用 `files=` 构造这种请求。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/api/?utm_source=chatgpt.com "Developer Interface — Requests 2.33.0.dev1 documentation"))

例如：

```python
import requests

with open("favicon.ico", "rb") as f:
    files = {"file": f}
    r = requests.post("https://httpbin.org/post", files=files, timeout=10)
    print(r.status_code)
```

---

## 1. 这部分对你有什么现实意义

即使你以后不常写上传代码，这个知识也有价值，因为它让你理解：

- 浏览器上传文件也是 HTTP 请求
    
- 前端交互最后常常会落到一条 API 请求上
    
- 爬虫和接口测试，本质上都在和 HTTP 打交道
    

---

# 十三、Cookie：为什么第一次请求和第二次请求会不一样

PPT 里专门讲了 Requests 对 Cookie 的查看与应用，这非常重要。

Requests 会把服务器返回的 Cookie 放在 `Response.cookies` 中；如果你使用 `Session`，它还能在多次请求之间持久化 Cookie。官方 Advanced Usage 明确说明：`Session` 会在多个请求之间持久化参数和 Cookie，并复用底层连接池。([requests.readthedocs.io](https://requests.readthedocs.io/en/master/user/advanced/?utm_source=chatgpt.com "Advanced Usage — Requests 2.33.0.dev1 documentation"))

---

## 1. 为什么 Cookie 对爬虫重要

因为很多网站的页面内容依赖状态：

- 是否已登录
    
- 是否访问过某页
    
- 是否通过某次校验
    
- 是否带某个站点标识
    

如果你不理解 Cookie，就很容易出现：

- 浏览器里有内容，Requests 没内容
    
- 第一次有内容，第二次失效
    
- 同一个接口在不同会话下结果不同
    

---

## 2. 查看 Cookie 的最简单方法

```python
import requests

r = requests.get("https://www.baidu.com", timeout=10)
print(r.cookies)
for k, v in r.cookies.items():
    print(k, v)
```

这一步不是为了“伪造身份”，而是为了理解：

> 服务器会在响应里下发状态信息，客户端会在后续请求中回传。

---

## 3. 为什么更推荐 `Session`

如果你要连续访问同一站点，最好用 `requests.Session()`：

```python
import requests

s = requests.Session()
r1 = s.get("https://example.com", timeout=10)
r2 = s.get("https://example.com/profile", timeout=10)
```

好处有两个：

- Cookie 自动继承
    
- 同主机连接可复用，效率更高
    

这也是 Requests 官方明确推荐的高级用法。([requests.readthedocs.io](https://requests.readthedocs.io/en/master/user/advanced/?utm_source=chatgpt.com "Advanced Usage — Requests 2.33.0.dev1 documentation"))

# 十四、Timeout：这是最容易被忽视，但最应该养成习惯的东西

PPT 最后一页讲 `timeout`，这一页非常重要，甚至可以说是“工程味”最强的一页。

Requests 官方文档明确说：

- 如果没有显式指定，Requests 默认**不会自动超时**
    
- `timeout` 不是整个下载总时间上限，而是底层 socket 在指定秒数内没有收到数据时触发异常
    
- 几乎所有生产环境代码都应该设置 timeout。([requests.readthedocs.io](https://requests.readthedocs.io/en/latest/user/quickstart/?utm_source=chatgpt.com "Quickstart — Requests 2.33.0.dev1 documentation"))
    

---

## 1. 为什么必须加 `timeout`

因为网络是不稳定的。  
如果服务器很慢、网络中断、对方卡住，你的程序可能一直挂着不返回。

所以不要写：

```python
requests.get(url)
```

应该写：

```python
requests.get(url, timeout=10)
```

---

## 2. 正确理解 timeout

它不是“10 秒内必须把整个网页下完”。  
更准确地说，是“如果连接或读取阶段长时间没有收到数据，就报超时”。

---

## 3. 实际工程里怎么写更稳

最好配合异常处理：

```python
import requests

try:
    r = requests.get("https://example.com", timeout=10)
    r.raise_for_status()
    print(r.text[:200])
except requests.exceptions.Timeout:
    print("请求超时")
except requests.exceptions.RequestException as e:
    print("请求失败：", e)
```

这段代码比单纯 `print(r.text)` 要成熟得多。

# 十九、给你一个比较规范的 Requests 入门模板

这是我建议你现在就开始习惯的写法：

```python
import requests

def fetch(url, params=None, headers=None):
    try:
        r = requests.get(
            url,
            params=params,
            headers=headers,
            timeout=10
        )
        r.raise_for_status()

        content_type = r.headers.get("Content-Type", "")
        print("状态码:", r.status_code)
        print("Content-Type:", content_type)

        if "application/json" in content_type:
            return r.json()
        else:
            return r.text

    except requests.exceptions.Timeout:
        print("请求超时")
    except requests.exceptions.RequestException as e:
        print("请求失败:", e)

result = fetch("https://httpbin.org/get", params={"name": "test"})
print(result)
```

这段模板的价值在于：

- 有 `timeout`
    
- 有异常处理
    
- 有状态码意识
    
- 有内容类型判断
    
- 不会一上来就把一切都当文本

# 一、先用一句话讲明白：什么是正则表达式

Python 官方文档对正则表达式的定义是：

> 正则表达式描述了一组字符串模式，用来检查某个字符串是否匹配这个模式。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

你可以把它理解成：

> **正则表达式 = 一套描述“什么样的字符串算符合条件”的规则。**

比如你想找：

- 所有手机号
    
- 所有邮箱
    
- 所有网页标题
    
- 所有数字
    
- 所有以 `http` 开头的链接
    
- 某个单词是否单独出现
    

都可以用正则来做。

---

# 二、你先别记语法，先记“正则解决什么问题”

你课件第一页叫“如何取出需要的信息”，这就是正则最核心的用法。

正则最常见的四类任务：

### 1. 判断有没有

例如：一个字符串里有没有邮箱格式。

### 2. 找出来

例如：从一段网页源码里找出所有标题。

### 3. 提取局部

例如：从 `手机号：13812345678` 中只取出 `13812345678`。

### 4. 替换 / 清洗

例如：把所有空白字符替换成一个空格。

所以你学正则时，不要把它当成“神秘符号学”，而要把它当成：

> **一种高级文本筛子。**

# 三、Python 里最常用的 `re` 函数：先学这 6 个就够了

你 PPT 里列了很多 `re` 函数。实际入门阶段，你先把下面 6 个学会，已经够解决大多数问题。

Python `re` 模块官方文档里给出了这些函数，它们既可以作为模块级函数直接用，也可以先 `compile()` 成正则对象再用。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 1. `re.match()`

`match()` 只从**字符串开头**开始匹配。官方 HOWTO 也专门区分了 `match()` 和 `search()`。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
import re

text = "Hello 123"
print(re.match(r"Hello", text))
print(re.match(r"123", text))
```

结果理解：

- 第一个能匹配，因为 `Hello` 就在开头
    
- 第二个不能，因为 `123` 不在开头
    

### 记忆法

**match = 只看开头**

---

## 2. `re.search()`

`search()` 会扫描整个字符串，找到**第一个**匹配位置。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
import re

text = "Hello 123"
print(re.search(r"123", text))
```

### 记忆法

**search = 全文搜第一个**

---

## 3. `re.findall()`

`findall()` 返回**所有非重叠匹配**，结果通常是列表。官方文档就是这样定义的。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
import re

text = "abc 123 def 456"
print(re.findall(r"\d+", text))
```

输出：

```python
['123', '456']
```

### 记忆法

**findall = 全找出来**

这是你以后最常用的一个。

---

## 4. `re.finditer()`

`finditer()` 返回一个迭代器，每个元素是一个 `Match` 对象。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

适合你不只想要内容，还想知道：

- 匹配了什么
    
- 在哪一段位置匹配到的
    

```python
import re

text = "abc 123 def 456"
for m in re.finditer(r"\d+", text):
    print(m.group(), m.span())
```

---

## 5. `re.sub()`

`sub()` 用于替换。官方文档中说明它会返回替换后的字符串。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
import re

text = "a   b\tc"
print(re.sub(r"\s+", " ", text))
```

可以把连续空白压缩成一个空格。

---

## 6. `re.split()`

`split()` 用正则做分割。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
import re

text = "apple,banana;orange grape"
print(re.split(r"[,;\s]+", text))
```

---

# 四、`Match` 对象怎么看

你 PPT 里举了 `span()` 和 `group()` 的例子，这很重要。

`match()`、`search()` 返回的是 `Match` 对象。它最常用的几个方法是：

- `group()`：拿到匹配到的文本
    
- `span()`：拿到匹配区间
    
- `start()` / `end()`：起止位置
    

这些都在 `Match Objects` 文档里有定义。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
import re

text = "Hello 123"
m = re.search(r"\d+", text)
print(m.group())   # 123
print(m.span())    # (6, 9)
```

### 记忆法

- `group()` 看“匹配到什么”
    
- `span()` 看“匹配到哪里”

# 五、原子：正则最基本的“积木”

你课件里讲“原子”，这个说法很适合初学者。

可以这样理解：

> **原子 = 正则里最基本的匹配单位**

最常见的原子有两类：

### 1. 普通字符

比如：

- `a`
    
- `b`
    
- `1`
    
- `_`
    

它们通常就匹配自己。Python 文档也明确说，大多数普通字符是最简单的正则，它们匹配自身。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

例如：

```python
re.search(r"Hello", "Hello world")
```

意思就是匹配字符串 `Hello`。

### 2. 特殊原子

比如：

- `\d`
    
- `\w`
    
- `.`
    
- `[abc]`
    

它们不是匹配自己，而是匹配某一类字符。

# 六、最重要的一组基础符号：你先把这 12 个学会

下面这些，基本就是你入门最核心的部分。Python 官方语法页都给了定义。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 1. `.` ：任意字符

`.` 默认匹配**除换行符以外的任意字符**。如果开了 `DOTALL`，连换行也能匹配。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.findall(r"a.c", "abc axc a-c")
```

会匹配：

- `abc`
    
- `axc`
    
- `a-c`
    

### 记忆法

`.` = 一个“任意位置的占位符”

---

## 2. `^` ：开头

`^` 匹配字符串开头；在 `MULTILINE` 模式下，也可匹配每一行开头。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.findall(r"^Hello", "Hello world")
```

---

## 3. `$` ：结尾

`$` 匹配字符串结尾；在多行模式下也可匹配行尾。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.findall(r"world$", "Hello world")
```

---

## 4. `*` ：前一个原子重复 0 次或多次

官方文档定义：`*` 匹配前一个 RE 的 0 次或多次。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.findall(r"ab*c", "ac abc abbc abbbc")
```

匹配：

- `ac`
    
- `abc`
    
- `abbc`
    
- `abbbc`
    

---

## 5. `+` ：前一个原子重复 1 次或多次

```python
re.findall(r"ab+c", "ac abc abbc")
```

匹配：

- `abc`
    
- `abbc`
    

但不匹配 `ac`。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 6. `?` ：前一个原子重复 0 次或 1 次

```python
re.findall(r"ab?c", "ac abc abbc")
```

匹配：

- `ac`
    
- `abc`
    

不匹配 `abbc`。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 7. `{n}` ：恰好 n 次

```python
re.findall(r"a{3}", "aa aaa aaaa")
```

匹配长度正好为 3 的 `a` 串。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 8. `{n,}` ：至少 n 次

```python
re.findall(r"a{2,}", "a aa aaa")
```

匹配至少两个 `a`。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 9. `{m,n}` ：至少 m 次，至多 n 次

```python
re.findall(r"a{2,4}", "a aa aaa aaaa aaaaa")
```

([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 10. `[]` ：字符集

字符集表示“这些字符里任意一个”。

Python 文档里明确说，字符集用来表示一个集合，匹配其中任意字符；范围可以写成 `[a-z]` 这种形式。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

例如：

```python
re.findall(r"[abc]", "apple banana cat")
```

会找出所有 `a`、`b`、`c`。

### 常见字符集

- `[0-9]`：任意数字
    
- `[a-z]`：任意小写字母
    
- `[A-Za-z]`：任意英文字母
    
- `[^0-9]`：不是数字的任意字符
    

---

## 11. `|` ：或

```python
re.findall(r"cat|dog", "cat dog pig cat")
```

表示匹配 `cat` 或 `dog`。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 12. `()` ：分组 / 捕获

括号最重要的作用有两个：

- 把多个原子组合成一个整体
    
- 捕获匹配内容
    

这和你课件写的 “Grouping / Capturing” 一致。Python 正则 HOWTO 也专门讲了分组与捕获。([Python documentation](https://docs.python.org/3/howto/regex.html "Regular Expression HOWTO — Python 3.14.3 documentation"))

# 七、最常见的快捷字符类：`\d \w \s`

这三个必须熟。

Python 文档把它们作为特殊序列来定义。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

## 1. `\d`

匹配十进制数字字符。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.findall(r"\d+", "电话 021-65901447")
```

## 2. `\w`

匹配“单词字符”，包括字母、数字、下划线；在 Unicode 模式下，范围比纯 ASCII 更广。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.findall(r"\w+", "Hello_123, world")
```

---

## 3. `\s`

匹配空白字符，比如空格、制表符、换行。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.sub(r"\s+", " ", text)
```

## 4. 它们的大写版本

- `\W`：非单词字符
    
- `\S`：非空白字符

# 八、字符集 `[]` 和括号 `()` 的区别：这是最容易混的地方

你课件专门讲了这一页，这里我给你讲透。

---

## 1. `[021]` 的意思

表示匹配：

- `0`
    
- `2`
    
- `1`
    

中的任意一个字符。

所以：

```python
re.findall(r"[021]", "021-65901447")
```

得到的是单个字符列表，比如 `['0', '2', '1', ...]`

---

## 2. `(021)` 的意思

表示把字符串 `021` 当成一个整体去匹配，并把它作为一个分组捕获出来。

```python
re.findall(r"(021)", "021-65901447")
```

得到的是：

```python
['021']
```

---

## 3. 一句话区分

- `[]`：**选一个字符**
    
- `()`：**把一串东西捆成一组**
    

这个区别非常重要。

# 九、边界：为什么 `hi` 和 `\bhi\b` 不一样

你课件里举了 `\bhi\b` 的例子，这非常经典。

Python 文档里把 `\b` 定义为“单词边界（word boundary）”——它不匹配具体字符，而匹配一个位置。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 1. `hi`

会匹配任何位置出现的 `hi`：

- `hi`
    
- `this`
    
- `behind`
    

里都可能匹配到一部分。

---

## 2. `\bhi\b`

表示前后都必须是单词边界。  
这样就只会匹配“独立出现的 hi”。

```python
re.findall(r"\bhi\b", "hi this is behind")
```

只会匹配前面的 `hi`。

---

## 3. 一句话理解

`\b` 不是字符，而是“单词边界位置”。

# 十、贪婪模式和懒惰模式：这在爬网页时特别重要

你课件里讲了“贪婪模式 vs 懒惰模式”，这是正则中最重要的坑之一。

Python 文档明确说：

- `*`、`+`、`?` 这些量词默认都是**贪婪的（greedy）**
    
- 在它们后面加 `?`，就会变成**非贪婪 / 最小匹配（non-greedy / minimal）**。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))
    

---

## 1. 什么叫贪婪

“尽可能多地吃字符”。

例如：

```python
html = "<p>First</p><p>Second</p>"
re.findall(r"<p>(.*)</p>", html)
```

结果会是：

```python
['First</p><p>Second']
```

因为 `.*` 太贪了，它会一路吃到最后一个 `</p>` 才停。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 2. 什么叫懒惰

“够用就停”。

```python
re.findall(r"<p>(.*?)</p>", html)
```

结果：

```python
['First', 'Second']
```

---

## 3. 你以后怎么记

- `.*`：贪婪
    
- `.*?`：懒惰
    

几乎所有网页提取里，只要你写 `.*`，都要先想一秒：  
**我是不是其实应该写 `.*?`？**

# 十一、模式修正符 / flags：最常用就 3 个

你课件里列了 `I M S U` 等。Python 文档把这些称为 flags。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

你入门先记最常用的 3 个。

---

## 1. `re.I` / `re.IGNORECASE`

忽略大小写。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

```python
re.findall(r"hello", "Hello HELLO", re.I)
```

---

## 2. `re.M` / `re.MULTILINE`

让 `^` 和 `$` 匹配每一行的开头和结尾，而不仅是整个字符串。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 3. `re.S` / `re.DOTALL`

让 `.` 连换行也匹配。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

这个在爬虫里特别有用，因为 HTML 源码常常跨很多行。

例如：

```python
re.findall(r"<div>(.*?)</div>", html, re.S)
```

# 十二、转义字符：什么时候必须加反斜杠

你课件最后讲“转义字符”，这是正则最基本但最容易出错的点。

Python 文档明确说，反斜杠 `\` 要么：

- 转义特殊字符
    
- 要么引入特殊序列
    

并且强烈建议正则模式用**原始字符串**写，即 `r"..."`，这样更不容易出错。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 1. 为什么要转义

因为这些字符本身有特殊含义：

- `.`
    
- `*`
    
- `+`
    
- `?`
    
- `|`
    
- `()`
    
- `[]`
    
- `{}`
    
- `^`
    
- `$`
    
- `\`
    

如果你想匹配它们本身，就要转义。

例如想匹配价格 `500$`：

```python
re.findall(r"\d+\$", "价格是 500$")
```

如果不写成 `\$`，`$` 就会被当成“结尾边界”，不是美元符号。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 2. 为什么推荐 `r"..."`

Python 文档专门强调：正则表达式最好写成 raw string，例如 `r"\n"`，这样反斜杠不会先被 Python 字符串机制吞掉。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

所以你以后写正则，形成习惯：

```python
pattern = r"\d+"
```

而不是：

```python
pattern = "\\d+"
```

后者也行，但更难读。

---

# 十三、`re.findall()` 为什么有时返回整串，有时只返回括号里的内容

这是很多初学者会突然卡住的地方。

当你的模式里**没有捕获组**时，`findall()` 返回完整匹配。

```python
re.findall(r"\d{3}-\d{8}", "电话 021-65901447")
```

得到：

```python
['021-65901447']
```

但如果你写了捕获组：

```python
re.findall(r"(\d{3})-(\d{8})", "电话 021-65901447")
```

得到的是：

```python
[('021', '65901447')]
```

这和 Python `re` 的行为定义有关。([Python documentation](https://docs.python.org/3/library/re.html "re — Regular expression operations — Python 3.14.3 documentation"))

---

## 解决思路

如果你只是为了分组控制优先级，不想让 `findall()` 改返回结构，就用**非捕获组**：

```python
(?:...)
```

这也是 Python 正则 HOWTO 里讲的内容。([Python documentation](https://docs.python.org/3/howto/regex.html "Regular Expression HOWTO — Python 3.14.3 documentation"))

例如：

```python
re.findall(r"(?:\d{3})-(\d{8})", "电话 021-65901447")
```

```python
print(re.findall(r"\w+@\w+\.\w+", text))             # 简单邮箱
print(re.findall(r"1[3-9]\d{9}", text))              # 手机号
```
