下面我按“**先建立整体图景，再逐节拆开讲**”的方式，把这一章完整讲清楚。你可以把这一章理解成：**写爬虫之前必须先补齐的底层常识**。它不是直接教你“怎么写代码”，而是在回答六个根本问题：

1. 浏览器访问网页时，到底发生了什么
    
2. 一个网页为什么能被抓到，抓到的到底是什么
    
3. 登录状态为什么会丢，Cookie 和 Session 怎么维持状态
    
4. 为什么有的网站直接 `requests.get()` 就行，有的却不行
    
5. 代理（proxy）为什么能帮爬虫隐藏来源、绕过限制
    
6. 为什么爬虫经常配合多线程、多进程来提速
    

这一章对应的内容，主要分成六部分：**HTTP 基本原理、Web 网页基础、爬虫的基本原理、Session 与 Cookie、代理的基本原理、多线程与多进程的基本原理**。整章的主线非常清晰：**先理解网络请求，再理解网页结构，再理解如何提取，再处理状态、反爬和效率问题。**

# 一、先给你一个总框架：这一章到底在学什么

很多初学者学爬虫时会有一个误区：以为“爬虫 = 写代码把网页下载下来”。  
其实不是。

更准确地说，爬虫（web crawler / web scraping）的本质是：

> **模拟客户端向服务器发出请求，获取响应数据，再从中抽取目标信息并保存。**

所以你会发现，这一章几乎全都围绕这个过程展开：

$发送请求 \rightarrow 获取响应 \rightarrow 理解网页结构 \rightarrow 提取数据 \rightarrow 保存数据 \rightarrow 提速与反封禁$

你可以把它看成一条流水线。后面每一节，都是在解释这条流水线中的一个环节。第 1.1 节告诉你“请求和响应”是什么；第 1.2 节告诉你“网页长什么样”；第 1.3 节告诉你“爬虫到底怎么工作”；第 1.4 和 1.5 节处理“状态”和“代理”；第 1.6 节处理“效率”。这就是整章的知识地图。

# 二、1.1 HTTP 基本原理：爬虫最底层的通信语言

这一节是全章最底层的一节。  
你只要明白一句话：

> **爬虫之所以能爬，本质上是因为它会“说 HTTP 这门话”。**

浏览器访问网页，本质上也是在和服务器用 HTTP 协议通信。爬虫只是把这个过程程序化了。第 1.1 节从 URL、HTTP、请求、响应、HTTP/2 这些内容切入，讲的都是这一层。书里前 12 页基本都在讲这一块。

## 2.1 URL 和 URI：先知道“访问地址”是什么

你在浏览器里输入的地址，比如：

`https://www.baidu.com`

这类东西叫 URL（Uniform Resource Locator，统一资源定位符）。书里也提到 URI（Uniform Resource Identifier，统一资源标识符），它是更大的概念。简单理解：

- **URI**：标识资源
    
- **URL**：不仅标识资源，还告诉你去哪里、怎么访问
    

所以 URL 可以看作 URI 的一种。书中还用图示说明了 URL 和 URN 同属于 URI 的范围。对爬虫来说，最常打交道的是 URL。

一个 URL 通常由这些部分组成：

$scheme://username:password@hostname:port/path;parameters?query#fragment$

你不用死记这个形式，但每一部分最好知道干什么：

### （1）scheme

协议类型，比如：

- `http`
    
- `https`
    
- `ftp`
    

爬虫里最常见的是 `http` 和 `https`。

### （2）hostname

主机名，也就是域名，比如 `www.baidu.com`。  
背后通常会解析到一个 IP 地址。

### （3）port

端口。  
HTTP 默认端口常见是 80，HTTPS 默认端口常见是 443。  
如果 URL 没写端口，通常使用默认值。

### （4）path

路径，表示服务器上的资源位置，比如：

`/index.html`

### （5）query

查询参数，也就是 `?` 后面那一串，比如：

`?wd=python&page=2`

这个对爬虫极其重要，因为很多列表页翻页、筛选、搜索关键词，实际上就是改 query 参数。

### （6）fragment

片段标识，也就是 `#` 后面的内容。  
它主要是给浏览器页面内部定位用的，通常**不会发给服务器**，因此对爬虫意义相对小一些。

你以后写爬虫时，看到 URL，不要只把它当字符串，而要知道：  
**里面往往藏着分页参数、搜索关键词、筛选条件、接口版本号等重要信息。** 这就是学 URL 结构的真正意义。

---

## 2.2 HTTP 和 HTTPS：它们是什么关系

HTTP（HyperText Transfer Protocol，超文本传输协议）是浏览器和服务器通信最常用的应用层协议。书里也给了 HTTP 发展的简单时间线：从 HTTP/0.9、HTTP/1.0、HTTP/1.1，再到 HTTP/2.0。

你可以把 HTTP 理解成一套约定：

- 客户端怎么发请求
    
- 服务器怎么回响应
    
- 请求和响应里面各个部分分别表示什么
    

HTTPS 则不是一种全新的业务协议，而是：

> **HTTP + SSL/TLS 加密层**

也就是在 HTTP 外面套一层安全传输机制。书里明确指出 HTTPS 的目标是提高安全性，保证数据传输的安全，包括身份认证和数据加密。

这对爬虫的启发是：

- `http://` 和 `https://` 在程序里都可以请求
    
- 但 HTTPS 连接会涉及证书校验、加密握手
    
- 一般现代网站大量使用 HTTPS，所以爬虫默认要适应 HTTPS 环境
    

---

## 2.3 HTTP 在哪一层：为什么书里提到了 TCP

书里讲到：HTTP 和 HTTPS 都属于**应用层（application layer）**协议，它们通常运行在 TCP 之上。  
这句话很关键。

简单理解：

- HTTP 负责规定“消息怎么组织”
    
- TCP 负责保证“消息怎么可靠地传输过去”
    

所以你写爬虫时，平常直接调用 `requests`、`httpx`、`aiohttp`，看上去像是在“发 HTTP 请求”，但更底层仍然依赖 TCP 建立连接、发送字节流。

## 2.4 浏览器打开网页，到底发生了什么

书里的图 1-2 把这个过程画得很清楚：  
**客户端发送 Request（请求），服务器返回 Response（响应）**。

把这个过程展开，可以理解为：

1. 你输入一个 URL
    
2. 浏览器解析域名，找到服务器
    
3. 浏览器向服务器发起 HTTP 请求
    
4. 服务器收到请求后处理
    
5. 服务器返回响应
    
6. 浏览器再把响应内容解析并显示出来
    

对爬虫来说，最核心的是第 3 步和第 5 步。  
因为程序做的事情就是：

- 模拟第 3 步发请求
    
- 接收第 5 步的响应
    
- 再从响应里提取数据
    

所以爬虫不是“神秘地把数据抓下来”，而是**自动化地重复浏览器和服务器之间的通信过程**。书里还通过 Chrome 开发者工具的 Network 面板示例，展示了如何观察一个网页加载时产生的各个请求。这个页面非常重要，因为这就是你以后分析网站时最核心的工具。

## 2.5 请求（Request）长什么样

书里把请求拆成三个部分：

- 请求方法（Request Method）
    
- 请求头（Request Headers）
    
- 请求体（Request Body）
    

这三个你一定要分清。

### （1）请求方法：你想“做什么”

最常见的是：

- `GET`：获取资源
    
- `POST`：提交数据
    

书里的表 1-2 还列了 `HEAD`、`PUT`、`DELETE`、`OPTIONS`、`CONNECT`、`TRACE` 等方法。你做基础爬虫时，最常见的还是 GET 和 POST。

#### GET 的特点

- 参数常放在 URL 里
    
- 常用于查询、获取页面内容
    
- 可直接在地址栏访问
    
- 浏览器容易缓存
    

#### POST 的特点

- 参数一般放在请求体里
    
- 常用于表单提交、登录、接口调用
    
- 地址栏不直接暴露数据
    
- 常见于 Ajax 接口请求
    

这里最容易混淆的一点是：

> **GET / POST 的区别不只是“一个明文一个不明文”**。

真正的区别在于**语义和数据放置方式**。  
HTTP 规范里，GET 更偏向“取数据”，POST 更偏向“提交数据”。  
当然，现实网站里也会有不那么规范的设计。

### （2）请求头：补充说明这次请求的“上下文”

书里列了很多常见请求头，比如：

- `Accept`
    
- `Accept-Language`
    
- `Accept-Encoding`
    
- `Host`
    
- `Cookie`
    
- `Referer`
    
- `User-Agent`
    
- `Content-Type`
    

这些头本质上是在告诉服务器：

- 我想接收什么格式
    
- 我来自哪里
    
- 我是什么客户端
    
- 我带没带登录状态
    
- 我提交的数据是什么格式
    

其中对爬虫最重要的几个是：

#### User-Agent

告诉服务器“我是谁”。  
浏览器会带上自己的标识。很多网站会检查这个头。

#### Cookie

告诉服务器“我是哪个会话里的用户”。  
登录后维持状态非常依赖它。

#### Referer

告诉服务器“我是从哪个页面跳过来的”。  
有些防盗链或接口校验会看它。

#### Content-Type

告诉服务器请求体是什么格式，比如：

- `application/x-www-form-urlencoded`
    
- `multipart/form-data`
    
- `application/json`
    

书里表 1-3 也总结了 Content-Type 和 POST 提交数据方式之间的关系。这个在你模拟表单提交、调用 Ajax 接口时非常重要。

### （3）请求体：真正提交的数据

请求体通常在 POST 请求中更常见。  
例如登录表单时，用户名和密码就可能放在请求体里。  
而 GET 请求很多时候没有请求体，或者即便有，也很少这样设计。

所以以后抓接口时，你一定要看清楚三件事：

- 参数是在 URL 里，还是 body 里
    
- body 是表单格式，还是 JSON
    
- 是否还依赖 header 里的 token / cookie / referer
    

这才是“会抓包”的核心。

## 2.6 响应（Response）长什么样

服务器返回的响应，书里同样拆成三部分：

- 状态码（Response Status Code）
    
- 响应头（Response Headers）
    
- 响应体（Response Body）
    

这三部分分别回答：

- 这次请求成功了吗
    
- 服务器附带了什么说明
    
- 真正的数据内容是什么
    

### （1）状态码：先判断有没有成功

书里表 1-4 列了很多常见状态码。你最常用、最该记住的有这些：

#### 2xx：成功

- `200 OK`：请求成功
    
- `201 Created`：资源创建成功
    
- `204 No Content`：成功但无返回内容
    

#### 3xx：重定向

- `301 Moved Permanently`
    
- `302 Found`
    
- `304 Not Modified`
    

#### 4xx：客户端错误

- `400 Bad Request`
    
- `401 Unauthorized`
    
- `403 Forbidden`
    
- `404 Not Found`
    
- `405 Method Not Allowed`
    

#### 5xx：服务器错误

- `500 Internal Server Error`
    
- `502 Bad Gateway`
    
- `503 Service Unavailable`
    
- `504 Gateway Timeout`
    

爬虫实战里最典型的是：

- `200`：大概率拿到数据了
    
- `403`：被拒绝访问，可能触发了反爬
    
- `404`：页面不存在
    
- `429`：有的网站会返回“请求过多”
    
- `500/502/503`：服务器问题，可能需要重试
    

书里没有把“反爬”这个词展开得很重，但你学状态码时就要已经能联想到它：  
**状态码不是死记，是用来诊断问题的。**

### （2）响应头：服务器附带的说明

书里提到的常见响应头包括：

- `Date`
    
- `Last-Modified`
    
- `Content-Encoding`
    
- `Server`
    
- `Content-Type`
    
- `Set-Cookie`
    
- `Expires`
    

其中最重要的几个：

#### Content-Type

告诉你响应体是什么类型。  
比如：

- `text/html`：HTML 页面
    
- `application/json`：JSON 数据
    
- `image/jpeg`：图片
    

做爬虫时，这是判断“我拿到的是网页还是接口数据”的重要线索。

#### Set-Cookie

服务器要求浏览器保存 Cookie。  
登录、会话保持、风控验证都常依赖它。

#### Content-Encoding

告诉你响应体是否被压缩，比如 gzip。  
这解释了为什么有时抓到的数据看着像乱码，本质是还没解压。

### （3）响应体：真正有用的数据

响应体才是你最后要解析的内容。  
可能是：

- HTML
    
- JSON
    
- 图片二进制
    
- 音视频流
    
- 文件
    

做网页内容爬取时，你最常处理的是：

- HTML：解析标签、文本、链接
    
- JSON：直接按字段取值
    

这也是为什么后面书里紧接着讲 Web 网页基础。因为响应体很多时候就是网页源码。

## 2.7 HTTP/2：为什么它更快

书里用了较大篇幅讲 HTTP/2，并画了图 1-8 的服务器推送示意图。这里你不必把协议细节背得很死，但要明白它为什么重要。

HTTP/1.1 的典型问题之一是：  
虽然可以复用 TCP 连接，但多个请求的组织效率仍然有限。

HTTP/2 的核心改进思路可以概括为四个词：

### （1）二进制分帧（binary framing）

HTTP/2 不再把消息简单当作纯文本块，而是拆成更结构化的二进制帧。

### （2）流（stream）

每个请求/响应可以属于不同的流。  
一个连接上可以同时承载多个流。

### （3）多路复用（multiplexing）

这是最关键的点。  
多个请求不必一个个排队完成，而可以在同一 TCP 连接上交错传输。

### （4）服务器推送（server push）

服务器可以在客户端还没明确请求时，主动把它预测会用到的资源先推过来，比如 JS、CSS。

书里还提到优先级、流量控制等机制。你不一定现在就用得到协议级细节，但要理解一个结论：

> **现代网站的请求组织方式已经比 HTTP/1.1 复杂很多，所以抓包时你看到很多并行请求、资源交错加载，是正常现象。**

对爬虫来说，你未必直接操作 HTTP/2 的底层帧，但你至少要知道：  
现在很多网站的数据加载是高并发、接口化、异步化的。不能再用“网页就是一个完整 HTML 文件”这种过于简单的视角去看。

# 三、1.2 Web 网页基础：响应体里的 HTML 到底是什么

如果说前一节是在讲“怎么拿到网页”，这一节就在讲“拿到网页之后你看到的是什么”。

书里从第 12 页到第 18 页，主要讲了前端三件套、HTML 结构、DOM 树、节点关系、CSS 选择器。对爬虫来说，这一节的任务只有一个：

> **让你知道如何在一堆网页源码里定位你真正要的数据。**

## 3.1 前端三件套：HTML、CSS、JavaScript 分别干什么

书里把网页比作由三部分构成：

- **HTML**：负责内容和结构
    
- **CSS**：负责样式和外观
    
- **JavaScript**：负责行为和交互
    

这个划分特别重要，因为你以后抓网页时，常常要判断：

- 我要的数据在 HTML 里吗
    
- 还是在 JS 动态请求的接口里
    
- 还是要靠执行 JS 才会生成
    

### HTML

HTML（HyperText Markup Language，超文本标记语言）负责定义页面结构，比如标题、段落、链接、表格、输入框等。

### CSS

CSS（Cascading Style Sheets，层叠样式表）负责外观，比如颜色、大小、排版、边框、位置。

### JavaScript

JavaScript 负责行为，比如点击按钮后加载内容、翻页、异步请求数据、局部刷新页面。

爬虫最直接处理的是 HTML，但真正难点很多时候来自 JavaScript。因为页面上看到的内容，不一定一开始就在 HTML 源码里。

---

## 3.2 HTML 的基本结构：网页不是一团字，而是一棵树

书里给了一个非常基础的 HTML 示例，大致包含：

- `<!DOCTYPE html>`
    
- `<html>`
    
- `<head>`
    
- `<meta charset="UTF-8">`
    
- `<title>`
    
- `<body>`
    
- `<div>`
    
- `<h1>`
    
- `<p>`
    

这个例子虽然简单，但足够你理解网页结构。

### head 和 body 的区别

- `head`：页面的“说明区”，放标题、编码、样式表、脚本引用等
    
- `body`：页面真正显示的主体内容
    

### id 和 class 的区别

书里也通过例子解释了：

- `id`：通常在一个页面中尽量唯一
    
- `class`：可重复，用来标记一类元素
    

爬虫里定位节点时，经常会优先找：

- `id`
    
- `class`
    
- 标签层级关系
    

因为这些通常比纯文本内容更稳定。

---

## 3.3 DOM：为什么说网页像一棵树

这是这一节最核心的概念。

DOM（Document Object Model，文档对象模型）可以理解为：

> **浏览器把 HTML 文档解析成的一棵树形结构。**

书里的图 1-11 和图 1-12 专门画了 DOM 树和节点之间的关系。

比如一个简单页面里：

- `html` 是根
    
- 下面分成 `head` 和 `body`
    
- `body` 下又有 `div`
    
- `div` 下又有 `h1` 和 `p`
    

于是就形成了父节点、子节点、兄弟节点的关系。

你以后写 XPath 或 CSS 选择器，本质上就是在这棵树上找节点。  
所以一定要把这几个关系理解透：

- parent：父节点
    
- child：子节点
    
- sibling：兄弟节点
    
- ancestor：祖先节点
    
- descendant：后代节点
    

这个理解一旦建立起来，网页解析就不再是“猜字符串”，而是“按树结构定位元素”。

---

## 3.4 CSS 选择器：如何在 DOM 树中定位元素

书里后面用了开发者工具示例，演示如何通过选择器找到网页中的具体节点，还给了表 1-5，列出大量常见 CSS 选择器规则。

你不需要一口气全背下来，但一定要掌握最常用的这几类：

### （1）按类名选

`.className`

比如：

`.title`

### （2）按 id 选

`#idName`

比如：

`#container`

### （3）按标签选

`div`、`p`、`a`

### （4）后代选择

`div p`

表示选中 `div` 里面所有后代 `p`。

### （5）子代选择

`div > p`

表示只选中 `div` 的直接子节点 `p`。

### （6）属性选择

`a[href]`  
`a[target="_blank"]`

### （7）伪类 / 结构选择

比如：

- `:first-child`
    
- `:last-child`
    
- `:nth-child(n)`
    
- `:not(...)`
    

这类在复杂页面中很常用。

对于爬虫而言，CSS 选择器最大的价值是：  
**让你从整页 HTML 中，精准地定位想要的内容区域。**

例如：

- 商品标题在哪个 `div`
    
- 价格在哪个 `span`
    
- 评论内容在哪个 `p`
    
- 下一页链接在哪个 `a`
    

当你学会 DOM + CSS 选择器之后，网页在你眼里就不再是一页“网页”，而是一棵“可查询的结构树”。这就是网页解析能力的开始。

---

## 3.5 XPath 和 CSS selector 的关系

书里最后提到，除了 CSS 选择器，还有一类非常常用的定位方式是 XPath，后面章节会继续讲。

你现在先建立一个认识：

- CSS selector：更像“网页前端世界”的定位方式
    
- XPath：更像“树结构路径表达式”
    

两者都能用于爬虫。  
很多库都同时支持：

- `BeautifulSoup.select()` 用 CSS selector
    
- `lxml.etree.xpath()` 用 XPath
    

实际使用里：

- 简单页面：CSS selector 往往更直观
    
- 复杂层级、相对定位：XPath 往往更强
    

---

# 四、1.3 爬虫的基本原理：终于进入“爬虫本体”

前两节铺垫完，到这里书里开始正式谈“爬虫到底是什么”。这一节的核心结论可以用一句话概括：

> **爬虫不是魔法，而是把浏览器访问网页的过程自动化。**

书里第 19～20 页把这个过程拆成：**爬取网页、提取信息、保存数据、自动化处理、动态页面分析**。这是非常标准的爬虫工作流。

## 4.1 爬虫工作的四个基本环节

### （1）爬取网页

先向目标网站发送请求，获取网页内容或接口数据。  
这里最关键的是：你要先搞清楚目标数据是通过什么请求拿到的。

### （2）提取信息

拿到 HTML 或 JSON 后，分析其中的数据结构，再把你需要的字段提取出来。

### （3）保存数据

把结果写入：

- TXT / CSV / JSON
    
- Excel
    
- MySQL / MongoDB 等数据库
    

### （4）自动化处理

把上面三步批量执行，实现从多个页面、多个链接、多个关键词中持续抓取数据。

这四步里，真正难的往往不是“写循环”，而是前面两步：

- 怎么找到真正的数据入口
    
- 怎么正确解析返回内容
    

所以书里才在前面花很大篇幅先讲 HTTP 和网页结构。

---

## 4.2 静态页面和动态页面：为什么同样是网页，难度不一样

这是初学者最容易卡住的地方。

书里在第 20 页举了一个特别重要的例子：  
有些网页初始返回的 HTML 很简单，真正的数据是后续通过 JavaScript 再请求接口、再插入 DOM 的。

这就对应两类页面：

### 静态页面（static page）

服务器返回的 HTML 里已经带有主要内容。  
你直接请求页面源码，就能在 HTML 中找到数据。

### 动态页面（dynamic page）

服务器初始返回的 HTML 只是个“壳”。  
真正的数据是后续通过 Ajax / Fetch 等方式请求接口，再由 JavaScript 渲染到页面上。

这两类页面对爬虫的影响非常大：

#### 静态页面的抓法

直接请求网页 + 解析 HTML 即可。

#### 动态页面的抓法

通常有两条路：

1. **直接分析 Ajax 接口，绕过页面渲染**
    
2. **使用 Selenium / Playwright 之类浏览器自动化工具执行 JS**
    

书里在这一节已经点到了 Ajax、Selenium、Splash、Pyppeteer、Playwright 这些方向。你要把这里当作一个非常关键的分水岭：

> 不是所有网页都应该“抓 HTML”；很多网页真正该抓的是“接口”。

---

## 4.3 为什么要学开发者工具

前面书里在 HTTP 部分已经展示了 Chrome 开发者工具的 Network 面板，这里它的意义真正体现出来了。

你分析一个网站时，最重要的问题不是“这页能不能爬”，而是：

- 页面请求了哪些接口
    
- 哪个请求返回了我要的数据
    
- 请求方法是什么
    
- 参数在哪
    
- 是否要带 cookie
    
- 返回的是 HTML 还是 JSON
    

这些都要靠开发者工具看。  
所以真正的爬虫工作流，不是“先写代码”，而是：

> **先打开开发者工具，先抓包，先分析，再写代码。**

这也是这一章想建立给你的思维方式。

---

# 五、1.4 Session 和 Cookie：为什么登录后状态能被记住

这一节是爬虫里最重要的“状态管理”知识。  
你以后只要碰到登录、个人主页、评论接口、下单记录之类需要身份状态的内容，这一节就会直接发挥作用。

书里第 21～24 页从“静态网页与动态网页”“无状态 HTTP”“Session”“Cookie”展开，逻辑很完整。

## 5.1 为什么需要 Cookie 和 Session

问题的根源是：

> **HTTP 是无状态（stateless）的。**

所谓无状态，就是服务器默认不会自动记住“前一次请求是不是你发的”。  
每一个请求，在协议层看上去都像新的请求。

但现实网站显然需要“记住你”：

- 你刚登录过
    
- 你购物车里有商品
    
- 你切换过语言
    
- 你有权限访问某页面
    

于是就必须有一套机制来维持状态，这就是 Cookie 和 Session 要解决的问题。

---

## 5.2 Cookie：保存在客户端的一小段状态信息

Cookie 可以理解成服务器发给客户端的一段小数据，客户端后续再访问同一网站时，会把它带回去。书里在第 23 页通过浏览器 Storage/Cookies 界面，列出了 Cookie 的典型字段。

常见字段包括：

- `Name`
    
- `Value`
    
- `Domain`
    
- `Path`
    
- `Max-Age` / `Expires`
    
- `Size`
    
- `HttpOnly`
    
- `Secure`
    

这些字段你至少要知道含义：

### Name / Value

Cookie 的名字和值。

### Domain

哪个域名下有效。

### Path

哪个路径下有效。

### Expires / Max-Age

什么时候过期。

### HttpOnly

若开启，JavaScript 不能通过 `document.cookie` 读取，安全性更高。

### Secure

只在 HTTPS 下传输。

对爬虫来说，Cookie 的作用非常直接：

- 携带登录状态
    
- 模拟浏览器身份
    
- 维持会话连续性
    

你以后很多请求失败，不是因为 URL 错了，而是因为**没带对 Cookie**。

---

## 5.3 Session：保存在服务器端的会话信息

Cookie 和 Session 经常一起出现，但它们不是一回事。

### Cookie

偏客户端。

### Session

偏服务端。

Session 可以理解成服务器为某个用户会话保存的一份状态记录。  
浏览器通常不会直接拿到整份 Session 内容，而是拿到一个 **Session ID**。  
这个 Session ID 通常放在 Cookie 里，后续请求再带回去，服务器就知道“你是谁”。

所以它们的关系是：

> **Cookie 常常用来存 Session ID，Session 的具体数据存在服务器端。**

这也是为什么很多网站登录后，并不是把“用户名、权限、购物车明细”全都放进 Cookie，而只是给你一个标识符。

---

## 5.4 Session Cookie 和 Persistent Cookie

书里还区分了两类 Cookie：

### Session Cookie

会话级 Cookie。  
浏览器关闭后常失效。

### Persistent Cookie

持久化 Cookie。  
有明确过期时间，可以保留更久。

这对爬虫意味着：

- 有的网站你复制一次 Cookie，可以用很久
    
- 有的网站你关掉浏览器再开，Cookie 就失效
    
- 还有的网站 Cookie 虽没过期，但 Session 在服务器端已经失效
    

所以“Cookie 还在”不等于“登录状态一定还有效”。这个理解很重要。

---

## 5.5 爬虫里怎么使用 Cookie / Session

把这一节落到实战，就是：

### 场景 1：手动复制浏览器 Cookie

适合快速测试某些需要登录的数据接口。

### 场景 2：程序登录后自动保存 Cookie

适合长期脚本。

### 场景 3：使用会话对象

例如 Python 的 `requests.Session()`，它可以自动维护 Cookie。

你要形成一个稳定思维：

> 当某个接口“明明参数都对，却返回未登录/无权限/跳转登录页”时，第一反应应该检查 Cookie 和 Session，而不是先怀疑解析代码。

---

# 六、1.5 代理的基本原理：为什么爬虫常配代理池

这一节是“身份隐藏”和“请求转发”的基础知识。  
书里第 24～26 页从代理服务器、代理作用、代理类型、代理设置讲起。

## 6.1 什么是代理

代理（proxy）可以理解成一个中间人。  
原本是：

$客户端 \rightarrow 目标服务器$

加上代理后变成：

$客户端 \rightarrow 代理服务器 \rightarrow 目标服务器$

目标网站看到的，不再是你的真实 IP，而是代理服务器的 IP。  
这就是代理在爬虫里的根本价值。

---

## 6.2 代理有什么作用

书里总结的作用大致包括：

- 提高访问速度
    
- 保护隐私、隐藏真实 IP
    
- 突破某些访问限制
    
- 分散单个 IP 的访问压力
    

对爬虫最实际的用途主要是两个：

### （1）隐藏来源
 
降低被封 IP 的风险。

### （2）分散请求

多个代理轮换访问，避免一个 IP 请求过于集中。

所以你以后听到“代理池”，本质上就是维护一批可用代理，让请求轮换出去。

---

## 6.3 代理的分类

书里给了两种分类方式。

### 按协议分

包括：

- FTP 代理
    
- HTTP 代理
    
- SSL/TLS 代理
    
- RTSP 代理
    
- Telnet 代理
    
- POP3/SMTP 代理
    
- SOCKS 代理
    

对爬虫最重要的通常是：

- HTTP 代理
    
- HTTPS 代理
    
- SOCKS5 代理
    

其中 SOCKS5 的适用范围更广，因为它不只服务于 HTTP。

### 按匿名程度分

书里也提到：

- 高匿名代理
    
- 普通匿名代理
    
- 透明代理
    

理解上可以这么记：

- **高匿名**：更像真的普通用户，不暴露你使用代理
    
- **普通匿名**：知道你用了代理，但不知道你原始 IP
    
- **透明代理**：连你原始 IP 都可能暴露出来，匿名性差
    

做爬虫当然更偏向高匿名代理。

---

## 6.4 代理不是万能药

这一点虽然书里没有用特别重的口吻强调，但从内容上你应该自然推出这个结论：

代理能解决的是“来源 IP”问题，解决不了所有问题。  
例如：

- Cookie 校验
    
- 登录态失效
    
- 请求头特征异常
    
- JavaScript 签名
    
- 验证码
    
- 行为风控
    

这些并不会因为你换了代理就自动消失。

所以正确理解是：

> **代理只是爬虫反封禁工具箱里的一个工具，不是总开关。**

---

# 七、1.6 多线程和多进程的基本原理：爬虫为什么能提速

这一节是“效率层”的知识。  
前面已经解决了“怎么抓、怎么定位、怎么维持状态”，这里开始解决“怎么更快地抓”。

书里第 26～28 页主要讲：线程和进程、并发与并行、多线程遍历场景、Python 中的多线程和多进程、GIL。

## 7.1 为什么爬虫适合并发

爬虫任务有一个很典型的特点：

> **大量时间花在等待网络响应上。**

比如你请求一个网页：

- 真正发送请求可能很快
    
- 服务器处理和返回可能要等几百毫秒甚至几秒
    

如果你一个个串行请求，那么 CPU 其实很多时候在“闲等”。  
于是就可以并发处理多个请求，把等待时间重叠起来，提高整体吞吐量。

---

## 7.2 进程和线程分别是什么

### 进程（process）

进程是资源分配的基本单位。  
每个进程一般有独立的内存空间。

### 线程（thread）

线程是调度执行的基本单位。  
一个进程里可以有多个线程，共享进程的大部分资源。

对爬虫来说，可以先粗略理解为：

- **进程** 更“重”，隔离更强
    
- **线程** 更“轻”，切换更快
    

书里也强调，线程是进程中的一个执行流。

---

## 7.3 并发（concurrency）和并行（parallelism）的区别

这个概念很多人会混。

### 并发

宏观上看，多个任务都在推进。  
哪怕底层是轮流切换执行，也可以构成并发。

### 并行

多个任务在同一时刻真正同时运行。  
通常更依赖多核 CPU。

所以：

- 单核上也能并发
    
- 真并行通常要多核
    

书里把这个区分讲得比较清楚。你学爬虫时，只要先明白：  
**多线程抓网页，哪怕不是严格并行，也能显著提速，因为网络等待可以交叠。**

---

## 7.4 Python 的 GIL：为什么多线程不总是万能

书里最后一部分专门讲了 Python 的 GIL（Global Interpreter Lock，全局解释器锁）。这是 Python 并发里绕不过去的概念。

你可以先抓住结论：

### 对 CPU 密集型任务

Python 多线程往往不能充分利用多核优势。  
因为同一时刻执行 Python 字节码会受到 GIL 限制。

### 对 I/O 密集型任务

例如网络请求、文件读写、数据库等待，  
多线程依然很有用。因为线程在等待 I/O 时，别的线程可以继续运行。

而爬虫恰恰大多数时候是 **I/O 密集型（I/O-bound）** 的。  
所以多线程在爬虫里非常常见。

---

## 7.5 多线程还是多进程，怎么选

可以这样记：

### 适合多线程的场景

- 大量网页请求
    
- 批量下载图片
    
- 等待接口返回
    
- 网络 I/O 为主
    

### 适合多进程的场景

- 大规模文本清洗
    
- 复杂数据计算
    
- CPU 密集型解析或建模任务
    

所以很多完整项目会这样组合：

- 抓取阶段：多线程 / 异步并发
    
- 计算阶段：多进程 / 向量化处理
    

这也是这一节想让你建立的工程思维：  
**爬虫不是只有“抓”，还包括效率设计。**

---

# 八、把整章串起来：你现在应该形成怎样的理解

学完这一章，你脑子里最好形成下面这条完整链路：

## 第一步：先会看请求

看到一个网页时，先想：

- 它的 URL 是什么
    
- 请求方法是 GET 还是 POST
    
- 参数在哪
    
- 返回的是 HTML 还是 JSON
    
- 状态码是什么
    

## 第二步：再会看网页结构

如果返回的是 HTML，就继续看：

- DOM 树怎么组织
    
- 我需要的数据在哪个节点
    
- 用 CSS selector 还是 XPath 定位更方便
    

## 第三步：再理解数据来源

如果页面内容不是直接写在 HTML 里，就要判断：

- 是不是 Ajax 动态加载
    
- 能不能直接抓接口
    
- 需不需要 Selenium / Playwright
    

## 第四步：处理状态

如果接口需要登录或身份校验，就要考虑：

- Cookie 带了吗
    
- Session 还有效吗
    
- 是否需要模拟登录
    

## 第五步：处理反爬和效率

如果抓取量大，就要进一步考虑：

- 代理是否需要轮换
    
- 多线程 / 多进程是否需要加入
    
- 如何限速、重试、去重、存储
    

这条链路，几乎就是一个完整爬虫项目的最小知识闭环。  
所以这章虽然叫“基础”，但其实是整个爬虫学习最不能跳过的一章。

# 九、这一章最该记住的高频考点 / 实战点

最后我帮你把这一章压缩成一组“最该带走的东西”。

## 1. URL 不是普通字符串

它里面有协议、主机、路径、参数，很多翻页和搜索逻辑都藏在 query 里。

## 2. 爬虫本质是在模拟 HTTP 请求

浏览器做的事，程序也能做。请求和响应是爬虫的底层语言。

## 3. Request 和 Response 都有固定结构

请求看方法、headers、body；响应看状态码、headers、body。

## 4. HTML 是树，不是散乱文本

网页解析本质是在 DOM 树上定位节点。

## 5. CSS selector / XPath 是定位工具

不会选节点，就无法稳定提取数据。

## 6. 不是所有数据都在 HTML 里

动态网站常把数据放在 Ajax 接口里，抓接口往往比抓页面更高效。

## 7. HTTP 无状态，所以需要 Cookie / Session

登录态、权限、用户身份保持都依赖它们。

## 8. 代理解决的是 IP 问题，不是所有反爬问题

它重要，但不是万能。

## 9. 爬虫大多是 I/O 密集型

所以多线程常有用；CPU 密集型任务更适合多进程。

这一章真正想让你建立的，不是几个零散术语，而是一种系统理解：

> **网页数据之所以能被爬到，是因为网页本质上是通过 HTTP 传输的结构化内容；爬虫就是自动化地完成“请求—接收—解析—存储”这一整套过程，并进一步处理状态、代理和并发效率问题。**
