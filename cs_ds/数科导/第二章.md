我按教材顺序，像老师上课一样给你讲，不做“提纲式总结”，而是直接把这部分内容讲透。你这份 PDF 这一段实际上在讲第 2 章“基本库的使用”，核心是：先用 `urllib` 理解 Python 发 HTTP 请求的底层方式，再用 `requests` 学更高效、更常用的写法，最后开始引入正则表达式，为后面从网页中抽取信息做准备。也就是说，这一章不是在教你“分析网页结构”本身，而是在教你爬虫最基础的机械系统：请求怎么发、响应怎么收、参数怎么带、Cookie/代理/认证/SSL 怎么处理、URL 怎么拆解、robots.txt 怎么看。

## 一、先把这一章在爬虫里的位置想清楚

你学爬虫，表面上像是在“抓网页”，但真正做事时，流程其实是这样的：

先构造请求，再把请求发给服务器，然后得到响应，接着判断响应是不是成功，最后才进入“从响应内容里提取信息”这一步。

所以这章的逻辑非常合理：

第一步先讲 `urllib`，因为它更接近 Python 标准库原生接口，能让你看清“HTTP 请求到底是什么东西”；  
第二步再讲 `requests`，因为它把很多常见操作封装得更好用；  
第三步开始讲正则表达式，因为拿到页面之后，你总得想办法从文本里把你要的内容抠出来。

你可以把这章理解成：**先学会“去网站门口敲门并说清楚你是谁、你要什么”，再学会“看对方给了你什么”，最后再学会“从拿到的材料里挑出有用信息”**。

## 二、2.1 `urllib`：它到底在干什么

教材一开始先介绍 `urllib`，并指出它是 Python 内置的 HTTP 请求库，而且不是一个单独文件，而是一组和 URL 处理、请求发送、错误处理、robots 协议等有关的模块集合。教材提到的子模块包括 `request`、`error`、`parse`、`robotparser` 等。这里你要先建立一个观念：

`urllib` 不是“一个函数”，而是一整套基础设施。

### 1. `request`：负责发请求

这是最核心的部分。教材先用 `urlopen()` 作为入口。这个函数最重要，因为它相当于“打开一个 URL 并返回服务器响应”。最简单的写法类似这样：

```python
import urllib.request
response = urllib.request.urlopen('https://www.python.org')
html = response.read().decode('utf-8')
print(html)
```

这里你一定要懂四层意思。

第一层，`urlopen()` 真正做的是一次 HTTP 请求，不是“读本地文件”。  
第二层，它返回的是一个响应对象 `response`，不是最终文本。  
第三层，真正的页面内容要靠 `response.read()` 读出来。  
第四层，读出来通常是 `bytes`，所以要 `decode('utf-8')` 转成字符串。

很多初学者的第一个误区就是：以为 `urlopen()` 返回的就是可直接处理的字符串。不是。它返回的是“服务器的响应对象”，你还得再读一层。教材在前几页正是在用这一点带你理解“请求对象”和“响应对象”的区别。

### 2. `response` 响应对象：不要只会 `read()`

教材接着讲，`response` 不只是能 `read()`，它还有很多有用属性和方法，比如：

- `status`：状态码
    
- `getheaders()`：响应头
    
- `getheader(name)`：取某个指定头
    
- `read()`：读响应体
    
- `version`、`reason` 等响应信息
    

这很重要，因为真正做爬虫时，你绝不能只看页面内容。你必须知道这次请求有没有成功。比如：

- `200` 说明通常成功；
    
- `404` 说明资源不存在；
    
- `403` 常常表示被拒绝；
    
- `500` 是服务器错误。
    

所以你以后写爬虫，脑子里要形成固定流程：

先看 `status`，再看 `headers`，最后才看正文内容。  
因为正文可能是空的、跳转页、错误页、反爬页，甚至是验证码页。

### 3. GET 和 POST：教材在这里开始让你真正理解“请求方式”

教材随后讲 `data` 参数。这个地方非常关键，因为它标志着你开始从“浏览网页”切换到“模拟客户端提交数据”。

在 `urllib.request.urlopen()` 里，如果你不给 `data`，通常就是 GET 请求；如果你传了 `data`，通常就会变成 POST 请求。教材用 `urllib.parse.urlencode()` 先把字典变成查询字符串，再把它编码成 `bytes`，然后交给 `urlopen()`。大意像这样：

```python
import urllib.parse
import urllib.request

data = bytes(urllib.parse.urlencode({'name': 'germey'}), encoding='utf-8')
response = urllib.request.urlopen('https://httpbin.org/post', data=data)
print(response.read().decode('utf-8'))
```

这里你要懂三件事。

第一，HTTP 里的数据不是随便传 Python 字典就行，底层传输要符合协议格式。  
第二，很多表单提交、本地搜索、登录、筛选，底层都是 POST。  
第三，`httpbin.org/post` 这种地址在教材里是“教学用回显网站”，它会把你发过去的内容原样展示回来，方便你观察请求结构。

这一步本质上不是在学语法，而是在学：**浏览器点“提交”按钮时，背后其实就是在发一个带数据的 HTTP 请求**。

### 4. 超时 `timeout`：为什么一定要设

教材专门讲了 `timeout`，这非常实战。因为网络请求不是你发出去就一定马上回来。服务器可能慢、可能挂、可能故意拖你。

如果你不设超时，程序可能会一直卡住。教材用 `timeout=0.1` 之类的例子，展示请求因为超过限定时间而报错。

这背后的实战意义是：

- 写爬虫不能假设网络永远正常；
    
- 每一个请求都应该有失败预案；
    
- “等不到”也是一种结果。
    

所以以后你写请求，尤其是批量抓取时，要养成习惯：

```python
response = urllib.request.urlopen(url, timeout=10)
```

这个 10 秒不是固定值，而是一个“别让程序无限卡死”的保险。

## 三、`Request` 对象：从“发一个链接”升级到“构造一个请求”

前面的 `urlopen(url)` 适合最简单场景。但教材很快告诉你：真正做爬虫，很多时候不能只传一个 URL，你还要控制请求头、请求方法、提交数据、用户代理等信息，于是就引出 `Request` 对象。

### 1. 为什么需要 `Request`

因为一个 HTTP 请求不只是“地址”。它还包含很多附加信息：

- 请求头 `headers`
    
- 请求方法 `GET / POST / PUT ...`
    
- 请求体 `data`
    
- 来源、身份、压缩方式、Cookie 等
    

教材这里其实是在告诉你：**一个完整请求 = URL + 方法 + 头部 + 可能的数据**。

所以教材给出 `Request(url, data=None, headers={}, method='POST')` 这样的构造思路。这里你最该注意的是 `headers`，尤其是 `User-Agent`。

### 2. 为什么很多网站要伪装 `User-Agent`

教材提到可以通过 `headers` 添加 `User-Agent`，模拟浏览器。这个非常常见。

因为很多网站会判断：  
“你真的是浏览器访问，还是一个脚本程序在抓我？”

如果你不加 `User-Agent`，有些网站就会认为你是异常请求，直接拒绝、返回空白页、跳到反爬页面，或者给你一个和正常浏览器不同的内容。

所以你以后看到这种代码：

```python
headers = {
    'User-Agent': 'Mozilla/5.0 ...'
}
req = urllib.request.Request(url=url, headers=headers, method='GET')
response = urllib.request.urlopen(req)
```

不要把它当成模板死记。你要知道它的真实作用是：  
**让服务器觉得“这是浏览器，不是粗糙爬虫”**。

### 3. `add_header()`：动态补请求头

教材还演示了 `req.add_header()`。这说明请求头不一定在创建对象时一次性写死，也可以后续补充。这对复杂程序很有用，比如：

- 某些头部根据运行结果动态生成；
    
- 某些接口有签名；
    
- 某些请求需要先登录后拿令牌再补头。

## 四、Handler 与 Opener：这是 `urllib` 真正“高级”的地方

教材后面开始讲 `Handler` 和 `Opener`。这往往是初学者最容易糊涂的部分，因为它一下子从“发请求”跳到了“请求处理机制”。

你可以这样理解：

- `urlopen()` 像一个简化按钮；
    
- `Opener` 像一个“可插拔的请求处理器”；
    
- `Handler` 则像给请求过程安装的各种功能模块。
    

也就是说，`urlopen()` 适合简单请求；  
而一旦你需要认证、代理、Cookie 管理等高级功能，就需要 `Handler + build_opener()` 这套机制。

### 1. 认证处理：`HTTPBasicAuthHandler`

教材给了一个需要基本认证的网站例子。这里的意思是：某些页面不是任何人都能直接访问，它会先弹一个用户名/密码验证框。这个过程对应 HTTP Basic Authentication。

教材通过下面这种逻辑来处理：

- 创建密码管理器 `HTTPPasswordMgrWithDefaultRealm`
    
- 往里添加用户名、密码和目标网址
    
- 用 `HTTPBasicAuthHandler` 生成处理器
    
- 再用 `build_opener()` 生成 opener
    
- 最后用 opener 去 `open()` 页面
    

你不要只记代码。它的本质是：

**把“遇到认证时怎么办”预先装进请求系统里。**

这样当服务器要求认证时，程序知道要自动带上身份信息。

### 2. 代理处理：`ProxyHandler`

教材接着讲代理。代理的作用是：你的请求先发给代理服务器，再由代理服务器转发给目标网站。

在爬虫里它主要有几个用途：

- 隐藏真实 IP；
    
- 降低单 IP 请求压力；
    
- 处理地域限制；
    
- 做分布式抓取。
    

教材用 `ProxyHandler` 构造 HTTP/HTTPS 代理，然后同样用 `build_opener()` 生成 opener。这里你要建立一个非常重要的意识：

**代理不是“另外一种请求方式”，而是“请求路径换了一层中转站”**。

### 3. Cookie 处理：`HTTPCookieProcessor`

这部分非常重要。教材后面用了相当篇幅讲 Cookie，包括 `CookieJar`、`MozillaCookieJar`、`LWPCookieJar`。

你要先知道 Cookie 是什么。

Cookie 是服务器让浏览器保存的一小段状态信息。最典型的用途是：

- 记住登录状态
    
- 记住用户偏好
    
- 识别同一会话
    
- 反爬追踪
    

如果你不处理 Cookie，很多网站会把你的每次请求都当成“全新的陌生访客”。这样就可能出现：

- 第一页能看，第二页要求登录；
    
- 登录接口成功了，但后续页面还是未登录；
    
- 接口返回说“权限不足”；  
    本质上往往就是因为状态没被带过去。
    

教材先讲最基础的 `CookieJar`，它能存储 Cookie；  
再讲 `MozillaCookieJar`，可以把 Cookie 保存成 Mozilla 浏览器风格文件；  
又讲 `LWPCookieJar`，可以存成另一种常见格式。

这里的关键不是文件格式本身，而是你要明白：

**Cookie 不是只能存在内存里，也可以持久化到本地，方便下次继续用。**

这在爬虫里很常见。比如你先人工登录一次，把 Cookie 存下来，后续程序复用 Cookie 去访问需要登录的页面。

## 五、异常处理：你不能假设每次请求都会成功

教材专门讲了 `urllib.error` 里的异常，尤其是 `URLError` 和 `HTTPError`。这一段非常关键，因为很多初学者写代码时只会写“成功路径”，不会写“失败路径”。

### 1. `URLError`

`URLError` 是更底层、更广义的错误。比如：

- 域名解析失败
    
- 网络连不上
    
- 超时
    
- URL 非法
    

教材用访问不存在网页、设置超时等方式演示了这些异常。你要抓住一点：

`URLError` 不是专门表示“HTTP 404”，它是更广义的“请求没有正常完成”。

### 2. `HTTPError`

`HTTPError` 是 `URLError` 的子类，但它专门对应 HTTP 层面的错误，比如：

- 404 Not Found
    
- 403 Forbidden
    
- 500 Internal Server Error
    

教材强调它除了能作为异常被捕获外，还有自己的属性：

- `code`
    
- `reason`
    
- `headers`
    

这说明什么？说明它虽然是“错误”，但服务器其实还是给了你一个正式响应。所以它往往比普通 `URLError` 更有信息量。

真正实战时，你应该像这样分层处理：

```python
from urllib import request, error

try:
    response = request.urlopen(url)
except error.HTTPError as e:
    print(e.code, e.reason)
except error.URLError as e:
    print(e.reason)
```

这不是形式主义，而是因为你得分清：

- 是服务器明确拒绝了你；
    
- 还是你压根没连上服务器。
    

### 3. 超时异常为什么要特别判断

教材还结合 `socket.timeout` 讲了超时。因为超时在爬虫里特别常见，所以你往往要单独识别它，决定是否重试。

这个细节特别实战：  
“404” 重试没意义；  
“timeout” 往往值得重试。

## 六、URL 解析：为什么爬虫一定要会拆 URL

到这里，教材转入 `urllib.parse`。这部分看起来像“字符串处理”，其实是爬虫非常基础的能力。因为你抓网页时经常遇到：

- 相对链接转绝对链接
    
- 提取查询参数
    
- 拼接下一页 URL
    
- 构造 API 请求地址
    
- 对中文关键词做编码
    

### 1. `urlparse()`：把 URL 拆开

教材讲 `urlparse()` 时，把 URL 分解成：

- `scheme`
    
- `netloc`
    
- `path`
    
- `params`
    
- `query`
    
- `fragment`
    

这六部分你必须真正懂：

`scheme` 是协议，比如 `http`、`https`；  
`netloc` 是域名和端口；  
`path` 是路径；  
`query` 是 `?` 后面的查询参数；  
`fragment` 是 `#` 后面的锚点；  
`params` 现在相对少见，但教材保留了完整结构。

这不是为了考试记名词，而是因为爬虫中你常常只想改某一部分。比如：

- 分页时只换 `page=2`
    
- 切换语言时只改路径
    
- 接口从 `http` 升成 `https`
    

### 2. `urlunparse()`：拆完了再拼回去

教材接着讲 `urlunparse()`。它和 `urlparse()` 是反向关系：  
前者负责拼 URL，后者负责拆 URL。

这很有用，因为真正工程里，直接用字符串拼 URL 很容易出错，比如漏了 `/`、多了 `?`、转义不对。  
而先拆再改、再按结构拼回去，更可靠。

### 3. `urlsplit()` 和 `urlunsplit()`

教材还讲了 `urlsplit()`。它和 `urlparse()` 很像，但更简化，通常把 `params` 并入 `path`。你可以把它理解成“更精简版本的 URL 拆分器”。

什么时候用它？  
当你根本不关心 `params` 这类历史字段时，它更直接。

### 4. `urljoin()`：这是最实战的一个函数

这一部分你一定要重视。网页里很多链接是相对链接，比如：

- `/about`
    
- `../images/a.jpg`
    
- `detail/123.html`
    

如果你直接拿去请求，会失败，因为它不完整。  
所以必须借助当前页面 URL 把相对地址变成绝对地址。教材正是用 `urljoin()` 讲这个问题。

这在爬虫里几乎是必备技能。  
比如你解析首页 HTML，提取出很多 `<a href="...">`，这些 `href` 往往不是完整网址。你就得：

```python
from urllib.parse import urljoin
full_url = urljoin(base_url, relative_url)
```

以后你采集列表页详情链接时，这个函数会非常常用。

### 5. `urlencode()`、`parse_qs()`、`parse_qsl()`

教材随后讲查询参数处理。

`urlencode()` 是把字典编码成查询字符串。比如：

```python
{'name': 'germey', 'age': 25}
```

变成：

```python
name=germey&age=25
```

这在构造 GET 请求参数时非常常见。

而 `parse_qs()` 则是反过来，把查询字符串解析成字典；  
`parse_qsl()` 则解析成键值对列表。

为什么一个给字典，一个给列表？

因为 URL 参数里同一个键可能出现多次。  
字典适合按键取值；  
列表更适合保留原始顺序和重复项。

### 6. `quote()` 和 `unquote()`

教材又讲了中文搜索词的 URL 编码。这一部分很重要。

URL 里不能随便直接放中文、空格、特殊字符，所以要做百分号编码。  
`quote()` 是编码，`unquote()` 是解码。

比如你要构造百度搜索“壁纸”之类关键词的地址，如果不编码，地址可能就不合法或行为异常。  
所以这一步本质上是在处理“URL 的合法字符集问题”。

## 七、`robots.txt` 与 `robotparser`：不是技术问题，是规则问题

教材后面开始讲 `robots.txt`。这部分很多新手只当作“附加知识”，其实它很重要，因为它对应的是爬虫与网站之间的访问规则。

### 1. `robots.txt` 是什么

它是网站根目录下通常可访问的一个文本文件，用来告诉搜索引擎或爬虫：

- 哪些路径允许抓
    
- 哪些路径不允许抓
    
- 哪类 User-Agent 适用哪些规则
    

教材给出了一些典型写法，比如：

- `User-agent`
    
- `Disallow`
    
- `Allow`
    

你要真正理解的是：  
这不是技术屏障，而是“公开声明的抓取规范”。

### 2. 为什么教材要在这里讲它

因为学爬虫不能只学“怎么抓”，还得学“该不该抓”“允不允许抓”。教材在这里实际上是在帮你建立基本规范意识。

### 3. `robotparser`

教材最后用 `urllib.robotparser.RobotFileParser` 读 `robots.txt`，然后调用 `can_fetch()` 判断某个 User-Agent 能不能访问某个页面。

这个工具的意义在于：  
你不用自己手写规则解析器，Python 已经帮你把 robots 规则转成可判断对象了。

## 八、2.2 `requests`：为什么它比 `urllib` 更常用

教材进入 2.2 后，先讲为什么要学 `requests`：虽然 `urllib` 足够底层也很完整，但使用起来相对繁琐；`requests` 把最常见的 HTTP 需求包装得更友好，所以在实际爬虫开发中更常见。

你可以把关系理解成：

- `urllib` 更像“原生工具箱”
    
- `requests` 更像“常用高效工具”
    

教材安排先学 `urllib` 再学 `requests`，本质上是先打底层认知，再学高层接口。这个顺序非常合理。

### 1. 最基础的 GET 请求

教材用 `requests.get()` 开始，非常直观：

```python
import requests
r = requests.get('https://www.baidu.com/')
```

然后教材介绍 `Response` 对象的几个关键属性：

- `status_code`
    
- `text`
    
- `content`
    
- `headers`
    
- `cookies`
    
- `url`
    
- `history`
    

这里最重要的是 `text` 和 `content` 的区别。

### 2. `text` 和 `content`：必须分清

教材用抓取网页和抓取 `favicon.ico` 说明两者区别：

- `r.text` 是字符串，适合文本内容；
    
- `r.content` 是字节，适合图片、文件、音频等二进制内容。
    

这背后的本质是：

网页“内容”不等于“字符串”。  
HTTP 响应体本质上就是一段字节流。  
只是当它恰好是 HTML/JSON/文本时，我们通常再把它解码成人类可读字符串。

所以以后你下载图片、PDF、视频片段时，用 `content`；  
而你看网页源码、接口返回 JSON 文本时，常用 `text` 或 `json()`。

### 3. `json()`：接口爬虫非常常用

教材讲到如果服务器返回的是 JSON，可以直接用 `r.json()`。这个特别重要，因为现代网页里很多数据不是写在 HTML 里，而是通过接口返回 JSON。

你以后做爬虫会经常遇到：

打开网页看着很复杂，但按 F12 抓包后发现真正数据来自一个 API。  
这时 `requests + json()` 就比解析 HTML 高效得多。

### 4. GET 参数：`params`

教材用 `params` 演示了比手动拼 URL 更规范的方式：

```python
data = {'name': 'germey', 'age': 25}
r = requests.get('https://httpbin.org/get', params=data)
```

这里它会自动帮你把参数附加到 URL 后面。

你一定要明白：

`params` 不是“额外语法糖”，而是把“查询参数”作为结构化数据交给库处理。  
这样你少犯很多拼接错误，也更适合程序动态生成请求。

### 5. POST 请求：`data`

教材随后讲 POST：

```python
data = {'name': 'germey', 'age': 25}
r = requests.post('https://httpbin.org/post', data=data)
```

`requests` 会帮你更自然地完成表单提交，不像 `urllib` 那样经常还要自己先编码成 bytes。

这也是 `requests` 更受欢迎的原因之一：  
很多常见场景它直接替你处理了。

## 九、`requests` 里的响应细节：真正决定你能不能稳定写爬虫

教材接着讲 `status_code`、`headers`、`cookies`、`url`、`history`。这一段不能只看成“属性介绍”，它们其实分别对应不同的实战问题。

### 1. `status_code`

它用来判断请求是否成功。教材还展示了 `requests.codes` 这套状态码映射。你不一定要背所有代码，但要知道常用状态码语义。

### 2. `headers`

- Set-Cookie 等
    

所以爬虫遇到奇怪问题时，不要只盯着正文，先看看头。很多线索都在这里。

### 3. `cookies`

这两个属性尤其适合处理跳转。

## 十一、`requests` 里的 Cookie、Session：为什么登录后状态能保住

这部分很重要。

教材先展示直接从响应中获取 Cookie，也演示手工设置 Cookie，以及用 `RequestsCookieJar` 去维护 Cookie。

但更关键的是 `Session`。教材用一个例子说明：

如果你第一次请求设置了 Cookie，第二次请求还想沿用前一次状态，那么普通 `requests.get()` 往往不够，因为它们彼此默认是“独立请求”；  
而 `requests.Session()` 会自动在多个请求间保持 Cookie 等会话信息。

这就是为什么登录场景几乎都要用 `Session`。  
否则你前脚刚登录，后脚访问个人主页，程序却像没登录过一样。

你可以把 `Session` 理解成：

**它不是单个请求，而是一段连续对话。**

教材用 `ssr2.scrape.center` 这类示例讲了 HTTPS 证书问题。

这里你要知道：  
HTTPS 不只是“加密连接”，还涉及证书验证。

教材给出两种处理思路：

1. `verify=False` 暂时关闭验证；

你要注意，`verify=False` 虽然方便，但只是开发或测试时的折中，不应无脑长期使用。  
因为你其实是在放弃证书校验。

## 十四、`PreparedRequest`：你已经开始接近底层机制了

教材最后还讲了 `PreparedRequest` 和 `Session.send()`。这部分属于“高级接口”，很多初学者第一次看会觉得抽象。

你可以这样理解：

平时我们写：

```python
requests.get(...)
requests.post(...)
```

这是一种“一步到位”的高级用法。  
而 `Request -> prepare -> send` 是把请求拆成更底层的三个阶段：

1. 先创建请求对象
    
2. 再准备成可发送格式
    
3. 最后由会话发送出去
    

为什么要这样？

因为有时你需要在真正发送前，对请求做更细粒度控制，比如：

- 动态改头部
    
- 检查 body
    
- 调整签名
    
- 统一发给 Session 处理
    

所以这部分其实是在告诉你：  
`requests` 虽然好用，但它也保留了接近底层的控制能力。

# 一、2.3 正则表达式（regular expression）：为什么它会出现在这里

前面你已经学会了怎么把网页请求下来，比如用 `requests.get()` 得到 HTML 源码。可是拿到源码，只是第一步。真正的爬虫不是“下载网页”，而是“从网页中提取信息”。

这时就出现了一个问题：  
网页源码本质上是一大段字符串。  
那你怎样从这段字符串里，准确找到你想要的数据？

这就是正则表达式要解决的问题。教材在这一节开头先用匹配电话号码、E-mail 地址、URL 的例子，让你看到：**有些信息虽然埋在长文本里，但它们的形式很有规律，所以可以用一种“模式”把它们提取出来。** 这就是正则表达式的核心思想。第 63–64 页的截图，就是在演示输入一段文本后，如何把其中的邮箱和 URL 单独匹配出来。

你一定要先建立一个很重要的认识：

**正则表达式处理的是“文本模式”，不是“网页结构”。**

也就是说：

- 如果你想找的是“长得像邮箱”的东西，正则很合适；
    
- 如果你想找的是“某个 HTML 标签节点里的文本”，正则未必是最稳妥的方法。
    

教材现在先讲正则，不是因为它是解析网页的唯一方法，而是因为它能最快帮助你建立“从文本中抽取信息”的能力。

---

# 二、正则表达式到底是什么：不是代码，而是“规则”

你可以把正则表达式理解成一种“字符串搜索规则”。

比如你想匹配一个手机号，它通常由连续数字组成；  
你想匹配一个邮箱，它通常形如 `用户名@域名`；  
你想匹配一个 URL，它往往以 `http://` 或 `https://` 开头。

这些东西的共同点是：它们不是固定的某一个字符串，但它们都满足某种格式规律。  
正则表达式就是用来描述这种规律的。

所以它不是“精确比较”，而是“按模式查找”。

举个最简单的对比：

```python
text == 'hello'
```

这是判断字符串是否**恰好等于** `hello`。

而正则做的是：

> 在一大段文本中，寻找“符合某种格式”的片段。

这两者完全不是一个层次的事情。

# 三、教材第 65 页那张“元字符表”是整节最核心的基础

教材在第 65 页列了一张很重要的表，里面是正则表达式的常见元字符（metacharacter）。这张表你不能死背，但你必须真正理解，因为后面所有例子都在用它。

我给你按“作用类别”讲，比逐个背更容易掌握。

## 1. 字符类：描述“什么字符可以匹配”

像这些：

- `\d`：匹配一个数字，相当于 `[0-9]`
    
- `\D`：匹配一个非数字
    
- `\w`：匹配字母、数字、下划线
    
- `\W`：匹配非字母数字下划线
    
- `\s`：匹配空白字符，比如空格、换行、制表符
    
- `\S`：匹配非空白字符
    
- `.`：匹配任意字符，但默认**不包括换行符**
    

这里最容易误解的是 `.`。  
很多初学者以为点号真的是“句号”。不是。  
在正则里，`.` 是一个通配符，表示“任意单个字符”。

例如：

```python
a.c
```

可以匹配：

- `abc`
    
- `a1c`
    
- `a-c`
    

因为中间那个 `.` 可以代表任何一个字符。

## 2. 数量词：描述“前面的东西重复多少次”

这些符号用于表示重复次数：

- `*`：前面的内容重复 0 次或多次
    
- `+`：前面的内容重复 1 次或多次
    
- `?`：前面的内容重复 0 次或 1 次
    
- `{m}`：恰好重复 m 次
    
- `{m,n}`：重复 m 到 n 次
    

例如：

```python
\d+
```

表示“一个或多个数字”。

这就比单独写 `\d` 强多了，因为 `\d` 只能匹配一个数字，而 `\d+` 可以匹配 `123456` 这样整段连续数字。

## 3. 边界符：描述“必须出现在什么位置”

- `^`：匹配字符串开头
    
- `$`：匹配字符串结尾
    

比如：

```python
^Hello
```

表示字符串必须以 `Hello` 开头。  
这就是为什么后面教材用 `re.match()` 时，很多模式默认是“从头开始匹配”的。

## 4. 集合与范围：描述“这一位可以是哪几个字符之一”

- `[abc]`：匹配 `a`、`b`、`c` 中任意一个
    
- `[^abc]`：匹配不是 `a`、`b`、`c` 的任意一个
    
- `[a-z]`：匹配一个小写字母
    
- `[0-9]`：匹配一个数字
    

这类写法在匹配某些固定字符范围时非常有用。

## 5. 分组与选择：描述“这一整段是一组”或“多种可能之一”

- `(...)`：分组
    
- `|`：或
    

分组很重要，因为你往往不只是想“匹配成功”，还想“把其中某一部分提取出来”。  
这就像你拿网去捞鱼，不只是要捞到整条，还要把鱼头、鱼尾、鱼鳞分别区分出来。

# 四、`re.match()`：教材先讲它，是为了让你理解“匹配对象”这件事

教材在第 66 页开始正式进入 Python 的 `re` 模块，先讲 `re.match()`。这是很自然的安排，因为它最能让你看清楚“正则匹配到底返回什么”。

## 1. `match()` 并不是“在全文中找”，而是“从开头试着匹配”

这点一定要记住。

```python
import re
result = re.match(pattern, content)
```

它的意思不是“去全文任何地方找 pattern”，而是：

> 从字符串开头开始，看能不能匹配上这个模式。

如果开头就不符合，哪怕后面存在符合条件的内容，`match()` 也会返回 `None`。

这就是为什么教材后面会专门拿 `match()` 和 `search()` 对比：  
`match()` 很适合判断“是不是以某种格式开头”，  
但不适合“在整段文本中到处搜索”。

## 2. `match()` 返回的不是字符串，而是 `Match` 对象

教材用第一页代码例子说明：

```python
result = re.match('Hello\s\d\d\d\s\w{10}', content)
print(result)
print(result.group())
print(result.span())
```

这里你会看到输出的不是普通文本，而是一个 `<re.Match object ...>`。

这说明正则匹配成功后，Python 返回的是一个“匹配结果对象”，里面包含：

- 匹配到的内容
    
- 匹配起止位置
    
- 分组结果
    
- 其他上下文信息
    

所以你以后不要把正则匹配理解成“返回 True 或 False”。  
更准确地说，它返回的是：

- 成功：一个 `Match` 对象
    
- 失败：`None`
    

## 3. `group()` 和 `span()`：为什么教材马上讲这两个

### `group()`

它返回匹配到的文本内容。

### `span()`

它返回匹配片段在原字符串中的位置区间，比如 `(0, 25)`。

这两个方法非常重要，因为：

- `group()` 让你拿到内容；
    
- `span()` 让你知道内容在原文本中的位置。
    

后面如果你做高亮显示、替换、切片，都可能用到位置信息。

# 五、分组（group）：这是正则从“匹配”走向“提取”的关键一步

教材接着讲分组。这个部分你必须吃透，因为爬虫里你真正需要的不是“这个正则能不能匹配”，而是“把我要的数据从中提出来”。

比如一段文本是：

```python
Hello 1234567 World_This is a Regex Demo
```

如果你写整个正则去匹配它，只能得到整段。  
但如果你在数字那部分加上括号：

```python
Hello\s(\d+)\sWorld
```

那么括号里的部分就变成了**第一个分组**。  
这时：

- `group(0)` 是整个匹配结果
    
- `group(1)` 是第一个括号内匹配到的部分，也就是数字串
    

这个机制非常关键。

你可以把括号理解成：

> “我不仅要匹配这段规则，我还要把其中这部分单独记下来。”

这就是爬虫提取字段的雏形。  
比如你 later 想从 HTML 里提取：

- 电影名
    
- 评分
    
- 上映时间
    

你通常就会给这些部分分别加分组，然后用 `group(1)`、`group(2)` 去取。

# 六、贪婪匹配（greedy）和非贪婪匹配（non-greedy）：这是初学正则最容易踩坑的地方

教材第 67–68 页专门讲了这个。这个知识点非常重要，因为很多人明明正则写得差不多，但结果总是“多吃了一大段”，原因就是没搞懂贪婪匹配。

## 1. 什么叫贪婪匹配

默认情况下，像 `*`、`+`、`{m,n}` 这类数量词都是**贪婪的**。  
也就是：**在保证整体仍然能匹配的前提下，尽可能多地匹配字符。**

比如你写：

```python
Hello.*Demo
```

如果中间有很多字符，`.*` 会尽量把它们都吞进去，直到最后还能以 `Demo` 收尾。

## 2. 为什么这在网页提取里容易出问题

因为 HTML 里同类标签经常重复出现。  
假设你想提取第一个 `<div>` 到 `</div>` 的内容，如果你写成：

```python
<div>.*</div>
```

很可能它会从第一个 `<div>` 一口气吃到最后一个 `</div>`，中间所有内容都吞了。  
这就匹配得太多了。

## 3. 非贪婪匹配怎么写

在贪婪数量词后面再加一个 `?`：

- `*?`
    
- `+?`
    
- `{m,n}?`
    

它的意思变成：**尽可能少地匹配。**

所以：

```python
Hello.*?Demo
```

会尽量短地找到从 `Hello` 到 `Demo` 的那一段。

这个在爬虫中极其常用。你以后只要是在 HTML 中提取某个局部字段，往往都优先考虑非贪婪匹配。  
因为网页里同类标签很多，默认贪婪很容易跨越多个节点，把范围吃乱。

---

# 七、`?` 不只表示非贪婪，它还有“可有可无”的意思

教材第 68 页还演示了 `?` 的另一个用法：让前面的内容变成可选项。

例如：

```python
colou?r
```

这能匹配：

- `color`
    
- `colour`
    

因为这里的 `u` 是可有可无的。

所以你要区分两种情形：

- 放在一个量词后面：变成非贪婪
    
- 直接跟在一个字符或子表达式后面：表示 0 次或 1 次
    

初学者常常把这两个意思混在一起。你一定要看它前面跟的是谁。

---

# 八、换行问题与 `re.S`：为什么你的正则在单行能用，跨行就失效

教材第 68–69 页特别强调了一个非常实战的问题：  
默认情况下，`.` 不能匹配换行符。

这意味着，如果你的 HTML 代码有很多换行，而你写的是：

```python
<div>.*?</div>
```

那么 `.` 只能在同一行里吃字符，一旦遇到换行，就断掉了。  
于是你本来以为应该匹配成功，结果却返回 `None`。

这时怎么办？  
教材给出的方法是加修饰符 `re.S`（也叫 `re.DOTALL`）。  
它会让 `.` 连换行符也能匹配。

这在处理 HTML 时非常常见，因为 HTML 源码通常不是一整行，而是多行缩进结构。  
所以很多网页正则提取都会写成这样：

```python
result = re.match(pattern, html, re.S)
```

或者：

```python
pattern = re.compile(r'...', re.S)
```

`re.S` 本质上解决的是：  
**你的模式想跨多行提取，而默认的点号做不到。**

# 九、修饰符（flags）：不是附加知识，而是“匹配环境设置”

教材第 69 页有一张修饰符表。这里你最该掌握的几个是：

## 1. `re.S`

刚讲过，使 `.` 匹配包括换行在内的所有字符。  
这是处理 HTML 最常用的修饰符之一。

## 2. `re.I`

忽略大小写（ignore case）。  
例如匹配 `hello` 时，`Hello`、`HELLO` 也能匹配。

## 3. `re.M`

多行匹配（multiline）。  
它会改变 `^` 和 `$` 的行为，使它们匹配每一行的开头和结尾，而不只是整个字符串的首尾。

## 4. `re.X`

允许你把正则写得更可读，可以在模式里加空格和注释。  
实际入门阶段用得不算特别多，但当正则变得很长时会很有帮助。

你可以把修饰符理解成：

> 同样一套正则规则，在不同“环境设置”下，解释方式会变。

所以它不是边角知识，而是正则的一部分。

# 十、转义（escape）：为什么有些字符必须加反斜杠

教材第 69 页还讲了一个关键问题：有些字符本来在正则中是“有特殊意义的”，如果你想匹配它本身，就必须转义。

最典型的是 `.`。

我们前面说过，`.` 在正则里表示“任意字符”。  
那么如果你真的想匹配一个普通的小数点、域名中的点、文件名中的点，就必须写成：

```python
\.
```

例如：

```python
www\.baidu\.com
```

这才是匹配字符串 `www.baidu.com`。  
如果你写成：

```python
www.baidu.com
```

那中间两个点会被解释成“任意字符”，匹配范围就变宽了。

这就是正则里“语法字符”和“普通字符”的区别。  
你以后看到一些模式前面总有很多反斜杠，不是写得花哨，而是为了让特殊字符回归字面含义。

# 十一、为什么教材说 `match()` 不太适合抓网页：因为它只能从头开始

教材第 69–70 页开始指出 `match()` 的局限：  
它要求从字符串开头匹配。

这在网页处理中往往不方便。因为 HTML 开头通常是：

- `<!DOCTYPE html>`
    
- `<html>`
    
- `<head>`
    

而你真正想找的数据，比如标题、链接、摘要，通常在中间甚至后半段。  
如果还死用 `match()`，你就必须在模式前面先写一大串 `.*?` 去跳过前面的内容。这样既笨，又容易出错。

所以教材自然引出 `search()`。

---

# 十二、`re.search()`：真正更适合“在全文里找目标”

`search()` 和 `match()` 最大区别就是：

- `match()`：从开头匹配
    
- `search()`：扫描整个字符串，找到第一个符合模式的位置
    

这对网页处理非常重要。  
因为网页源码本来就是一大坨文本，而你通常只是想在里面找到“第一个符合条件的位置”。  
这时 `search()` 更自然。

教材用第 70–71 页的 HTML 片段举例，演示如何在整段 HTML 中找到当前激活节点 `<li class="active"> ... </li>` 内的歌手名和歌曲名。这个例子非常典型，因为它说明：

1. 你面对的是多行 HTML；
    
2. 目标信息不是开头就出现；
    
3. 你只需要找到某个局部片段；
    
4. 所以要用 `search()` 加 `re.S`。
    

这就是为什么很多实战代码里，正则提取网页字段时更常见的是 `search()` 而不是 `match()`。

---

# 十三、`re.findall()`：当你不只想找一个，而是想把所有符合条件的都找出来

教材第 71 页继续引出 `findall()`。这一步也非常自然。

前面的 `match()`、`search()` 更像是：

> 找到一个匹配结果。

而 `findall()` 是：

> 把所有符合条件的结果都列出来。

这在网页中极其常见。  
比如一个列表页里有 10 条电影记录，你显然不是只想提第 1 条，而是要把 10 条都拿到。  
这时候 `findall()` 就很方便。

教材用歌曲列表例子说明，如果模式里带分组，那么 `findall()` 返回的往往是元组列表，每个元组对应一条记录中的多个字段。  
这已经很接近后面 2.5 的实战写法了。

也就是说：

- `search()` 更适合取一个
    
- `findall()` 更适合取一批
    

---

# 十四、`re.sub()`：它不是提取，而是替换

教材第 72 页讲 `sub()`。这个函数的作用不是“找出什么”，而是“把匹配到的内容替换掉”。

比如教材举例会把某些 HTML 标签去掉，只保留文本内容。  
这一步很有现实意义，因为网页源码里往往混着大量标签，如果你只是想留下“纯文本”，那先做替换清洗会更方便。

所以 `sub()` 常用于：

- 去掉标签
    
- 去掉空白符
    
- 去掉特殊干扰内容
    
- 统一格式
    

你可以把它理解成：  
`findall()` 偏向“抓数据”，  
`sub()` 偏向“清洗数据”。

---

# 十五、`re.compile()`：为什么教材最后要讲它

教材第 72–73 页最后讲 `compile()`，这一点非常合理。因为前面你已经看过很多模式字符串了，现在要开始考虑：**如果同一个正则要反复用很多次，能不能先“编译好”再用？**

答案就是 `re.compile()`。

```python
pattern = re.compile(r'\d+', re.S)
```

这样得到一个正则对象，后面你可以多次：

- `pattern.match(...)`
    
- `pattern.search(...)`
    
- `pattern.findall(...)`
    

它的好处有两个。

第一，更清晰。  
正则本身和使用逻辑分开了。

第二，更适合复用。  
尤其是后面你在爬虫里解析每个详情页时，同一个正则会反复使用很多次，先编译出来是更规范的做法。

这也是为什么 2.5 的实战部分里，你会看到很多 `re.compile(...)`。

# 十六、2.3 这一节你真正该学到的，不是函数名，而是“文本提取思维”

学到这里，你要把正则看成一套完整工作流：

先定义目标模式；  
再决定是从头匹配、全局搜索，还是批量提取；  
再考虑是否需要分组、是否需要非贪婪、是否需要跨行匹配；  
最后根据需要返回单个结果、多个结果，或者直接替换清洗。

也就是说，2.3 不是在教你几个零散 API，而是在教你：

> 如何把“网页中的某段文本信息”从无结构字符串里有规则地抽出来。

# 十七、2.4 `httpx`：为什么前面都学了 `requests`，这里还要再学一个库

教材第 73 页开始进入 `httpx`。这个安排不是重复，而是为了解决 `requests` 的一个现实限制。教材明确说了：前面介绍的 `urllib` 和 `requests` 都是基于 HTTP/1.1 的，而有些网站可能使用了 HTTP/2 协议，此时前两者就不一定够用了，所以引入 `httpx`。

你先不要把这理解成“又来一个新库”。  
更准确地说，这是在告诉你：

**发 HTTP 请求这件事，并不是只有一种实现。不同协议特性、不同场景，可能需要不同工具。**

## 1. 为什么会有 HTTP/2 问题

HTTP/2 是 HTTP/1.1 之后的升级版本。  
你现在不用把协议细节一次性学太深，但至少要知道：

- 它不是简单换个名字；
    
- 它在连接复用、传输效率等方面更先进；
    
- 有些网站或服务端实现会更适合甚至要求 HTTP/2。
    

教材用一个站点示例说明：用 `requests` 访问时可能报协议相关异常，而这时就可以考虑改用 `httpx`。第 74 页的截图和代码就是在展示这种场景。

# 十八、`httpx` 和 `requests` 很像，但不是简单替代品

教材第 75–76 页特意用最基本的 GET 请求展示 `httpx` 的写法。你会发现它和 `requests` 非常像：

```python
import httpx
response = httpx.get('https://httpbin.org/get')
print(response.status_code)
print(response.text)
```

返回对象也有：

- `status_code`
    
- `text`
    
- `headers`
    
- 等属性
    

这说明 `httpx` 的 API 设计是很友好的，它并没有故意跟 `requests` 反着来，而是让你能比较平滑地迁移过去。

但你不能因此觉得它只是“另一个 requests”。  
它真正的价值在于：

- 更现代的客户端设计
    
- 支持 HTTP/2
    
- 支持异步请求（async）
    

---

# 十九、`Client`：这就是 `httpx` 里很重要的“会话级客户端”

教材第 77 页开始讲 `httpx.Client()`。你可以把它理解成 `requests.Session()` 的近亲。

为什么需要 `Client`？

因为当你不是只发一次请求，而是要连续发很多次请求时，就会遇到这些需求：

- 复用连接
    
- 共享 headers
    
- 共享 cookies
    
- 统一配置超时、代理、认证等
    

如果每次都用裸的 `httpx.get()`，那是一次性调用；  
而 `Client` 更像一个长期存在的请求会话环境。

教材演示了两种写法：

```python
client = httpx.Client()
response = client.get(...)
client.close()
```

以及更推荐的上下文管理器写法：

```python
with httpx.Client() as client:
    response = client.get(...)
```

后者的好处是自动关闭资源，更安全。

你要理解，这不是写法习惯问题，而是资源管理问题。  
网络连接不是白来的，客户端对象开了以后要妥善关闭。

---

# 二十、默认请求头：为什么 `Client(headers=...)` 很有用

教材还展示了在创建 `Client` 时传入默认 `headers`。这说明：

> 你可以把一些每次请求都要带的配置，放到客户端层统一处理。

这很实战。比如你做一个爬虫，所有请求都需要带同一个 `User-Agent`、同一种认证头，那你没必要每次都手工加一遍，直接写在 `Client` 上就行。

这个设计思想其实很重要：  
**请求级参数处理单次差异，客户端级参数处理全局共性。**

---

# 二十一、开启 HTTP/2：这才是教材讲 `httpx` 的重点

教材第 77–78 页明确给出：

```python
client = httpx.Client(http2=True)
response = client.get(...)
print(response.http_version)
```

这里的关键是 `http2=True`。  
你显式告诉客户端：开启 HTTP/2 支持。

然后 `response.http_version` 能告诉你最终用的是：

- `"HTTP/1.0"`
    
- `"HTTP/1.1"`
    
- `"HTTP/2"`
    

这一点特别好，因为它让你不仅“猜测自己可能在用 HTTP/2”，而是可以实际检查。  
这在调试时很重要。

---

# 二十二、异步请求 `AsyncClient`：这是 `httpx` 比 `requests` 更现代的一面

教材第 78 页还简单讲了：

```python
async with httpx.AsyncClient(http2=True) as client:
    response = await client.get(url)
```

这说明 `httpx` 支持异步（asynchronous）编程。

你现在可以先把异步理解成：

> 当你需要同时等待很多网络请求时，不一定要一个请求完成后再发下一个，可以让等待过程更高效。

这对高并发抓取、接口采集特别有价值。  
不过因为你现在还在学爬虫基础，教材这里只是点到为止，没有深入展开。这个安排很合理。

你当前阶段只要先记住：

- `requests`：同步、经典、常用
    
- `httpx`：更现代，支持 HTTP/2，也支持异步

# 二十八、`parse_index(html)`：第一次真正用正则去提取链接

教材第 83–84 页开始写列表页解析函数。这里的目标很明确：

> 从列表页 HTML 中，提取每个电影详情页的链接。

它用正则去找 `<a>` 标签中的 `href`，并配合 `class="name"` 之类标志定位到电影标题链接。然后通过 `urljoin()` 把相对路径补成完整 URL。

这里其实把前面几节内容全部串上了：

- 用 `requests` 拿 HTML
    
- 用正则 `findall()` 提取目标片段
    
- 用非贪婪匹配避免吃多
    
- 用 `urljoin()` 把相对链接转绝对链接
    

这就是为什么教材前面要分开讲那么多基础知识。  
到了这里你会发现，它们不是散的，而是拼起来刚好能干活。

# 二十九、为什么要先拿详情页 URL，而不是直接在列表页提全部信息

因为列表页通常只展示摘要信息。  
完整字段，比如剧情简介、完整分类、上映时间、评分细节等，往往只在详情页里。教材第 79 页和第 80 页的截图已经展示了列表页与详情页信息量差异。

所以这里体现的是一个非常常见的爬虫策略：

1. 列表页采链接；
    
2. 详情页采字段。
    

你以后做电商、招聘、论坛、新闻、点评类网站时，经常都会遇到这个模式。

# 三十、`scrape_detail(url)`：详情页抓取本质上和列表页抓取是同一件事

教材第 85 页定义 `scrape_detail(url)`，你会发现它几乎只是再次调用 `scrape_page(url)`。

这说明一个很重要的思想：

- 列表页和详情页在“请求”层面没有本质区别；
    
- 它们真正的区别在“解析规则”不同。
    

所以程序应该复用请求逻辑，而不是为不同页面类型重复写一遍请求代码。

# 三十一、`parse_detail(html)`：这才是整节实战最核心的解析函数

教材第 85–86 页对详情页 HTML 做分析，指出各字段在源码中的位置，比如：

- 封面图在某个 `img` 的 `src`
    
- 名称在 `h2`
    
- 类别在若干 `button` 或标签中
    
- 上映时间在带 `published-at` 类名的节点里
    
- 剧情简介在 `drama` 类节点里
    
- 评分在 `score` 节点里
    

然后教材用多个正则分别提取这些字段。这里你要真正看懂的不是某一条正则本身，而是**正则如何围绕 HTML 结构设计**。

## 1. 封面 `cover`

它本质上是从 `img` 标签的 `src` 属性里提取图片地址。  
所以正则思路是：先定位到有 `cover` 标识的那段，再把 `src="..."` 里的内容抓出来。

## 2. 名称 `name`

通常在标题标签中，比如 `h2`。  
所以思路是：定位到该标题标签，再把标签内部文本作为分组取出。

## 3. 类别 `categories`

一个电影可能有多个类别，所以教材这里用的是 `findall()` 而不是 `search()`。  
这非常合理，因为分类是“多个值”。

## 4. 上映时间 `published_at`

这里往往混着额外说明，所以会用一个更细的分组只取日期主体。

## 5. 剧情简介 `drama`

剧情简介通常是较长文本，而且可能跨行，所以经常需要：

- 非贪婪匹配
    
- `re.S`
    
- 再用 `strip()` 清理首尾空白
    

## 6. 评分 `score`

评分看似简单，但网页中可能有空格或换行，所以常会用更宽松的匹配模式后再转成数值或字符串。

# 三十二、为什么教材在详情页解析里大量使用 `search()`、`findall()`、`strip()`

因为不同字段类型不一样。

- 单值字段，如封面、名称、上映时间、评分：通常用 `search()`
    
- 多值字段，如类别：通常用 `findall()`
    
- 文本字段，如剧情：提取后再 `strip()` 去空白
    

这其实就是从“正则 API 学习”走向“字段工程处理”的一步。  
你不再是为了练习某个函数而用它，而是根据字段形态选工具。

# 三十三、主流程 `main()`：一页页抓，再一条条进详情

教材第 86–87 页把前面函数串起来，形成主流程：

1. 遍历页码
    
2. 抓列表页 HTML
    
3. 解析出详情页 URL 列表
    
4. 遍历每个详情 URL
    
5. 抓详情页 HTML
    
6. 解析字段
    
7. 保存结果
    

这就是一个最基础、最标准的爬虫骨架。  
你以后写很多项目，本质上都是这个结构，只不过页面更复杂、字段更多、反爬更强。

# 三十四、`save_data(data)`：为什么数据保存被单独写成函数

教材第 87–88 页把保存逻辑单独封装，使用：

- `os`
    
- `json`
    
- `makedirs`
    
- `json.dump(..., ensure_ascii=False, indent=2)`
    

这里非常值得你学。

## 1. 为什么保存逻辑要独立封装

因为“抓取”和“存储”是两个不同环节。  
把它们拆开，代码更清楚，也更方便以后改成：

- 存 CSV
    
- 存数据库
    
- 存 Excel
    
- 发到 API
    

## 2. `ensure_ascii=False` 为什么重要

因为如果不设这个参数，中文会被写成 Unicode 转义，看起来很难读。  
设成 `False` 后，JSON 文件里会保留正常中文显示。

## 3. `indent=2` 为什么也很重要

它让 JSON 文件更易读。  
这不是功能必须，但在调试和教学阶段非常有价值。

# 三十五、多进程 `multiprocessing.Pool`：为什么教材最后又加速了一版

教材第 88–89 页在程序已经能跑通后，又引入多进程池，把每个分页任务并行化。

这一点很关键，因为它体现了工程思维：

**先让程序正确，再让程序更快。**

多进程的基本想法是：

- 每一页列表页几乎是相互独立的任务
    
- 所以可以让多个进程同时处理不同页码
    
- 最后加快整体抓取速度
    

教材把原来大循环封装成 `main(page)`，然后用：

```python
pool.map(main, pages)
```

让多个页码并行处理。

这是一种非常经典的并发优化方式。  
因为分页天然就是可切分任务。

当然你也要知道，多进程不是永远都能乱用。  
如果网站限流严格、反爬强，进程开太多反而更容易触发封禁。  
教材这里主要是展示“如何把串行任务并行化”的思路。

# 三十七、但你也要看懂它的“教学性质”：为什么这里用正则解析 HTML

这点我专门提醒你一下。

教材这里用正则表达式解析 HTML，是为了让你把“请求 + 正则 + 保存 + 多进程”这一整套基础链条先跑通。  
从教学角度看，这样安排很合理。

但在真实项目里，如果 HTML 结构复杂、标签嵌套很多、格式不稳定，那么单纯靠正则去解析 HTML 会比较脆。  
这时更常见的是：

- `lxml + XPath`
    
- `BeautifulSoup`
    
- `parsel`
    
- CSS Selector
    

不过你现在先别急着跳过去。  
因为如果你连这套基础链条都还没吃透，直接上更复杂的解析工具，很容易只会“套模板”，不理解背后逻辑。

# 三十八、你现在最该把这一部分消化成什么能力

学完这部分后，你应该至少形成下面三种能力。

## 第一种：能看懂一个正则在匹配什么

不是只会背 `\d+`、`.*?`，而是能判断：

- 这个模式是从哪开始找
    
- 哪一部分会被分组提取
    
- 会不会贪婪吃多
    
- 需不需要跨行匹配
    

## 第二种：能把网页采集过程拆成多个函数

比如：

- 请求函数
    
- 列表页解析函数
    
- 详情页解析函数
    
- 保存函数
    

## 第三种：能从页面现象反推出程序规则

比如：

- 页码怎么变化
    
- 详情页链接在哪里
    
- 字段在 HTML 哪个节点附近
    
- 哪些字段是单值，哪些是多值
    

这三种能力，比单独记住某几个 API 更重要。
